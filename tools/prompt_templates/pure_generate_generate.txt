Authoring rules (the deterministic gates your bundle must pass ‚Äî the host runs them on your source AFTER you return, and you get no second look at them, so satisfy them on the first emission).
(1) Style lint (`Generate.lint`, fortitude): keep every source line UNDER 100 columns ‚Äî 99 is the longest that passes, since S001 fires at exactly 100 (`line length of 100, exceeds maximum 100`); split with a trailing `&` continuation; import with an explicit `use <module>, only: <names>` list ‚Äî never a bare `use` (C121); import from an intrinsic module with the `use, intrinsic :: iso_fortran_env, only: ...` form (C122); PORT011 rejects a number-literal kind on EITHER a real or an integer, so always declare with a named kind from `iso_fortran_env` ‚Äî `real(dp)` after `use, intrinsic :: iso_fortran_env, only: dp => real64` (never `real(8)`), `integer(int64)` (never `integer(8)` or `integer(4)`) ‚Äî and note the named kind comes FROM `iso_fortran_env`, so adding that import is itself subject to the C122 rule above: "fixing" PORT011 with a bare `use iso_fortran_env` is the classic cascade that earns a fresh C122 on the next attempt, so add the `intrinsic` modifier in the same edit; give every module a default accessibility statement (C131) ‚Äî a bare `private` in the specification part ‚Äî and then a `public :: <spec_id>__<op>` line for EVERY name the module must expose, because a bare `private` alone publishes nothing: it silences C131 and then breaks the next gate, where a consumer staged alongside you (your own checks module, a depending node) fails `Generate.syntax` with `Symbol '<name>' referenced at (1) not found in module '<spec_id>_model'`. State both halves, never just one; give every `select case` a `case default` arm (C011), even one that only guards an unreachable value.
(2) The `implicit none` trap ‚Äî this exact pair of mistakes cost a prior run its whole retry budget, so read it twice. The lint wants `implicit none` to carry the F2018 spec-list `implicit none (type, external)`, but the build runs `-std=f2008`, where that spec-list is a COMPILE ERROR. Do NOT "fix" one by introducing the other; do NOT oscillate between them. The ONE correct form is a plain `implicit none` with a suppression directive on its OWN LINE immediately above it: a line containing only `! allow(C003)`, then the next line `implicit none`. Putting the directive on the SAME line as `implicit none` is wrong twice over (the C003 finding survives AND you earn a FORT002 finding). Writing `implicit none (external)` or `implicit none (type, external)` is wrong (compile error under `-std=f2008`). Emit the two-line form everywhere and move on.
(3) Syntax gate (`Generate.syntax` ‚Äî a real `gfortran -fsyntax-only -std=f2008` front-end run over your source staged with its dependency closure, so a `! allow(...)` directive CANNOT suppress anything here: fortitude has no unused-symbol rule and an unknown-rule allow self-rejects under FORT001/FORT002): identifiers must be 63 characters or shorter, and if `<spec_id>_model` or `<spec_id>__<op>` would itself exceed the limit, do NOT abbreviate it ‚Äî the literal `spec_id` prefix is load-bearing (a depending node is required to `use <spec_id>_model`), so an overlong one is a spec-level problem to report, never a name for you to shorten; `stop` and `error stop` accept only a CONSTANT expression as their code/message ‚Äî to report a runtime value, `print` it first and then `error stop` with a literal; the gate promotes `-Werror=unused-dummy-argument` and `-Werror=unused-variable`, which has two consequences ‚Äî (a) a dummy an interface FIXES but your algorithm never consumes (an inert input, an ABI-fixed `name` / `case_id`) stays a live `intent(in)` dummy and is bound immediately after the declarations with the EMPTY block `associate (unused_<name> => <name>); end associate` ‚Äî an arithmetic no-op such as `0*<name>` is forbidden as the binding, and it computes nothing on purpose, so put no work inside it; an unused LOCAL is deleted rather than bound ‚Äî and (b) an `intent(out)` dummy your body never sets is this gate's error too (`declared INTENT(OUT) but was not set`), so assign every `intent(out)` you declare; Fortran identifiers are case-INSENSITIVE, so `Flux` and `flux` collide ‚Äî when a local or dummy would collide with an IR-derived name, keep the IR name verbatim and rename YOUR side.
(4) Dependency-dataflow gate (`Generate.static` ‚Äî it inspects `problem` nodes with dependencies, and every subroutine there that declares at least one `intent(out)` dummy): declare the incoming state `intent(in)` and the updated state (the IR's `io_contract.outputs` / `algorithm.required_update_paths` variables) `intent(out)`. What the gate checks is NOT merely that each `intent(out)` gets assigned ‚Äî it is that the values a load-bearing `call <dep>__<op>(...)` RETURNS reach an `intent(out)` dummy through an assignment chain (likewise the IR's `io_contract.semantic_dependency.required_sources`). Assigning every `intent(out)` from your own local computation does not satisfy it: a dependency result consumed only by an `intent(inout)` in-place update reaches no `intent(out)` and FAILS with `does not propagate dependency operation outputs to intent(out) dataflow`.
(5) Dependency calls: for each operation your node depends on, check the IR for a declared SINK for its result ‚Äî a variable in any `algorithm.steps[]` entry (`inputs` or `outputs`), a `derived_field_rules` / `invariants` quantity, an `io_contract` output, or a `semantic_dependency.required_sources` variable. IR variable names carry no provenance marker, so resolve the correspondence from the step semantics ‚Äî and when the correspondence is UNCERTAIN, treat the call as load-bearing; that is the default, and inert is the narrow exception. Do NOT invent a purpose for an inert call ‚Äî no probes, no thresholds, no validity flags, no branching on its result.
