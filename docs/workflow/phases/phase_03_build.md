# Phase 3: Build

## Overview
The phase that **deterministically** compiles the source generated by `Generate` and produces a binary. It requires no LLM inference, and only makes build-command calls via the MCP server. Build is a single `step` that has no standard `substep`.

## I/O contract
- execution input: `source/<source_id>/src/`, the `impl_defaults` of `spec.ir.yaml`
- verification input: `spec.ir.yaml`, `source_meta.json`
- output: `workspace/pipelines/<node_key_safe>/<pipeline_id>/binary/<binary_id>/bin/`, `binary_meta.json`, the `command_id` and `command_log_ref` of `compile_project`
  - The in-source Make of `build_system=make` runs with out-of-source overrides: the execution binary is output to `BINDIR=<pipeline>/binary/<binary_id>/bin`, and the object/`.mod` to `OBJDIR=workspace/tmp/<agent_run_id>/build` (per-run tmp, auto-clean), and nothing other than the cross-phase MCP audit log is written to `src/`. **Only `OBJDIR` and `BINDIR` are overridden — `BIN` must not be passed.** The binary basename is owned by the Makefile's `BIN` default (`<spec_id>_runner`); renaming it here (e.g. `BIN=<slug>`) diverges from what `Validate.execute`'s `make test` resolves via the same default and causes its non-relinking guard to fail (or, with a relinking guard, an `unauthorized_write_violation`). `binary_meta.json#binary_artifact_ref` points to `binary/<binary_id>/bin/<exe>`, where `<exe>` is exactly that Makefile-default `BIN`.

## `binary_id` format
- Format: `bin_<YYYYMMDD>_<seq3>`, e.g. `bin_20260511_001`

## Required requirements
- `Build` runs via the `MCP` server. An artifact must not be generated by LLM inference.
- `Build` uses `compile_project`, and for `fortran` / `c` / `cpp` / `mixed` families uses a standard build tool that can handle dependencies (default `make`).
- The `Build` input with `spec.ir.yaml.impl_defaults.toolchain.build_system=make` must be a dependency-complete version in which `src/Makefile` makes explicit the language-dependent compile-order dependencies as prerequisites.
- The `Build` with `spec.ir.yaml.impl_defaults.toolchain.build_system=make` requires a dependency description whose success/failure does not change with `make -j`.
- The `Build` with `spec.ir.yaml.impl_defaults.toolchain.build_system=make` passes `OBJDIR=<abs>/workspace/tmp/<agent_run_id>/build` and `BINDIR=<abs>/<pipeline>/binary/<binary_id>/bin` to `extra_args` of `compile_project` (these two overrides **only**; **`BIN` must not be overridden** — the binary basename is the Makefile's `BIN` default `<spec_id>_runner`, and a rename here desyncs from `Validate.execute`'s `make test`), and must output build artifacts outside `src/` (object to per-run tmp, exe to `binary/<binary_id>/bin/`). Enumerate the execution binary path in `allowed_output_paths` in file form. Writing `.o`/`.mod`/exe under `src/` is outside the Build capability write_root (only `binary/`) and invites an `unauthorized_write_violation` → `fail_closed`.
- Save the actual-command record of `compile_project` in `JSONL` format, with the default destination `project_dir/command_log.jsonl`.
- The `Build` trial metadata (`binary_meta.json`) records the `command_id` and `command_log_ref` (or `command_log_path`) traceably.
- For a `node` that has dependencies, `Build` requires verifying that the dependency-`operation` resolution target matches `spec.ir.yaml.dependency`. On mismatch, it is a `Build fail`.
- `Build` runs individually per `node`, and must not mix in the artifacts of other `node`.
- `Build` must `fail` the case in which a `node`-specific `module`, `subroutine`, or `runner` implementation of a dependency `node` is mixed into the depending `src/`, as a `dependency implementation encapsulation` violation.
- Before `Build` completes, run `python3 tools/validate_pipeline_semantics.py --stage post_build --pipeline-root workspace/pipelines/<node_key_safe>/<pipeline_id>/`, and add `--source-id <source_id>` as needed. `exit code 0` is required, and on `fail` it is a `Build fail`.

## `binary_meta.json` required keys
- `attempt_count`, `verification_status`, `last_fail_reason`
- `source_source_id`: the `source_id` that issued the input source
- `binary_artifact_ref`: the canonical path of the execution binary `binary/<binary_id>/bin/<exe>` (the out-of-source `BINDIR` output; it must not point under `src/`). Required for the `run_program` input resolution of `Validate.execute`.
- `command_id`, `command_log_ref` (or `command_log_path`)
- `validation_stage`: `post_build` or `full`
- `failure_category`, `failure_source_refs[]`, `failure_excerpt`: required when `verification_status=fail`. Used as a deterministic trigger for Generate retry (for details see the "On-failure behavior" section).

## On-failure behavior
- A `Build` failure **always feeds a retry back to `Generate`** (because Build is a deterministic process, there is no room for fixing other than the code).
- `Build` must not internally retry itself (a rebuild of the same source does not change the result).

### retry trigger (no LLM involvement)
The return to Generate on a `Build` failure is **decided deterministically without LLM inference**. Record the following fields in `binary_meta.json` as required, and the `orchestration agent` reads them and forwards them to Generate:

| field | range | extraction source |
|---|---|---|
| `failure_category` | `compile_error` / `link_error` / `make_error` / `dependency_violation` / `validate_post_build_violation` | mechanically classified by the `command_log`'s exit code and stderr patterns |
| `failure_source_refs[]` | a path list to `src/...` | the group of source paths extracted from the error output in the `command_log` |
| `failure_excerpt` | text (last 50 lines of stderr) | a direct excerpt of the `command_log` |

Classification convention for `failure_category`:
- `compile_error`: a non-zero exit of the compiler + stderr containing `error:`.
- `link_error`: detect the linker's `undefined reference` / `unresolved external` etc.
- `make_error`: a dependency-resolution failure of `make` itself (`No rule to make target` etc.).
- `dependency_violation`: detection of a `dependency implementation encapsulation` violation (mixing-in of a dependency `node`'s implementation).
- `validate_post_build_violation`: a structural fail of `validate_pipeline_semantics --stage post_build`.

### Determination of `repair_strategy` (no LLM involvement)
The `orchestration agent` chooses `repair_strategy` deterministically from `failure_category`:

| `failure_category` | `repair_strategy` | basis |
|---|---|---|
| `compile_error` | `reuse` | assumed to converge with a local syntax / type fix |
| `link_error` | `reuse` | a local fix of function-signature consistency |
| `make_error` | `restart` | the whole Makefile / dependency description needs reconstruction |
| `dependency_violation` | `restart` | an encapsulation violation requires structural regeneration |
| `validate_post_build_violation` | `restart` | a structural invariant violation does not converge with a local fix |

For the `Generate` re-submission's `launches/<new_agent_run_id>.request.json#repair_reason`, quote the `failure_category` / `failure_source_refs[]` / `failure_excerpt` of `binary_meta.json`. The Generate side defaults to fixing only the source pointed to by `failure_source_refs[]` (except on `restart`).

## Design trade-offs
- Because a failure always returns to `Generate`, "whether Build should be integrated into Generate's verify" is a point of discussion, but Build is a primary producer that produces an **observable primary artifact (binary)**, and keeping the phase independent has the advantages of:
  - being able to make explicit the difference in nature between LLM-driven synthesis (`Generate`) and deterministic compilation (`Build`)
  - the binary remaining as a first-class `workspace/` artifact, reusable from external tools (debugger / profiler / re-execution)
  - the observation point "build success ≠ execution success" being visible from the orchestration

  so these advantages prevail, and it is made an independent phase.
