# Phase 2: Generate

## Overview
The phase that, with `spec.ir.yaml` as the canonical input, generates the source code of the target `node`'s `model` (physics computation) and `runner` (input/output / execution coordination). `Generate.generate` reads only `spec.ir.yaml` (§2-1); `Generate.verify` also reads `controlled_spec.md` (§2-2).

## Generate-executor (`legacy` / `pure`)
The `Generate.generate` / `Generate.verify` leaves execute under one of two `generate-executor` selections (`docs/GLOSSARY.md`), chosen per run and recorded in `orchestration_meta.json#invocation.generate_executor`. The authoring rules of §2-1 and the verification items of §2-2 are properties of the emitted source and apply identically under both executors; only the transport and the write path differ.
- `legacy`: the agentic leaves run with tools enabled and the forced document reads described below. `Generate.generate` writes `source/<source_id>/src/` directly; `Generate.verify` re-authors `source_meta.json`.
- `pure` (`Z2`, a `claude` backend `M3c` node only): the `Generate.generate` leaf is a `pure-function leaf` — it reads no `SKILL` and no injected must-read (the host inlines the target IR, `<dependency_facts>`, exemplar, target profile, and harness capability ABI into the launch prompt) and returns exactly one `CodegenBundle`. After the leaf terminates, the host validates the bundle (`validate_bundle`, the post-generate gate), writes the declared files under `source/<source_id>/src/`, assembles the runner/Makefile build graph, and authors `source_meta.json`. `Generate.verify` is likewise a `pure-function leaf` reviewing the bundle and assembled source in an independent conversation; it returns a verify verdict the host projects onto `source_meta.json`. A pure leaf holds no write authority (`write_roots` empty); its terminal record carries `output_refs == []` and the deliverable is vouched by the host-written artifact's on-disk existence (`docs/ORCHESTRATION.md`). Under `pure` the source directory additionally holds `codegen_bundle.json` (the accepted bundle) and `bundle_meta.json` / `verdict_meta.json` (per-attempt provenance). A non-`M3c` or `codex` node stays `legacy` even under `--generate-executor pure`. Launch-prompt shape: `docs/workflow/LAUNCH_PROMPT_REFERENCE.md`.

## I/O contract
- execution input: `spec.ir.yaml`
- verification input: `spec.ir.yaml`, `controlled_spec.md` (verify-only), the generated `source/<source_id>/src/`
- output: `workspace/pipelines/<node_key_safe>/<pipeline_id>/source/<source_id>/src/`, `source_meta.json`, `lint_meta.json` / `syntax_meta.json` (the conductor-authored `Generate.lint` / `Generate.syntax` deliverables; see substep structure below). Under the `pure` `generate-executor` the source directory additionally holds `codegen_bundle.json`, `bundle_meta.json`, and `verdict_meta.json` (see Generate-executor above).

## substep structure
- `Generate.generate`: the LLM substep that generates the full source set. Under `legacy` it reads `spec.ir.yaml` and writes `src/` directly; under `pure` it is a `pure-function leaf` returning one `CodegenBundle` the host validates and writes (see Generate-executor above).
- `Generate.lint`: a **deterministic** substep the conductor runs **in-process** (no leaf) between `Generate.generate` and `Generate.syntax` — it runs the MCP `run_linter` `preset` for `impl_defaults.toolchain.language` over `source/<source_id>/src/`, writes `lint_meta.json` and the host-authored lint evidence (`<pipeline_root>/lint_evidence/<source_id>.json`). A lint finding re-runs `Generate.generate` via a warm-resume reopen (the same leaf fixes its source with context intact); see `tools/workflow_conductor.py:_lint_inproc`.
- `Generate.syntax`: a **deterministic** substep the conductor runs **in-process** (no leaf) between `Generate.lint` and `Generate.static` — for `impl_defaults.toolchain.language=fortran` it runs the MCP `run_syntax_check` compiler front-end gate (`gfortran -fsyntax-only -std=<toolchain.standard> -Werror=unused-dummy-argument -Werror=unused-variable`, plus `-fopenmp` when `target.backend=openmp`) over the node's `src/*.f90` staged together with the certified dependency-closure `<dep>_model.f90` (per-compiler throwaway dir under `workspace/tmp/<agent_run_id>/syntax/<compiler>/`, isolated `.mods` — never Build's `$(OBJDIR)`). This surfaces the whole class of syntax / standard-conformance `compile_error`s here instead of at Build. Two warning classes are promoted to errors over the whole staged set: an unreferenced dummy argument or an unused local variable is a gate failure, so a dummy the interface fixes but the algorithm defines as inert stays a live dummy bound by the `associate (unused_<name> => <name>); end associate` idiom (canonical: [CHECKS_MODULE_CONTRACT.md](../CHECKS_MODULE_CONTRACT.md) §5). It writes `syntax_meta.json` and the host-authored syntax evidence (`<pipeline_root>/syntax_evidence/<source_id>.json`). A compiler finding in the node's own source re-runs `Generate.generate` via a warm-resume reopen (same as a lint finding). A failing stage is first attributed by re-compiling two isolated source sets with the same argv: a canary valid under every standard (a failure there means the invocation is unviable — typically an `impl_defaults.toolchain.standard` the compiler driver does not accept; it is passed verbatim as `-std=<value>`), then the staged dependency closure alone (a failure there means the closure is at fault — either a dependency's certified source is defective, since a dependency certified before a gate rule existed is not clean by induction, or this node's declared `standard` is narrower than the sound closure needs; the compiler's diagnostics say which). Both are transport `fail_closed`: the leaf authors neither the IR nor a dependency's source, so no retry of this node could clear them. A missing mandatory `gfortran` is likewise a transport `fail_closed` (environment problem). Optional **target-compiler stages** come from the `METDSL_SYNTAX_COMPILERS` env var (comma-separated adapter ids, default `gfortran`; e.g. `gfortran,frt` once a Fujitsu adapter exists) — a stage whose compiler binary is not installed is recorded as `skipped` and does not gate, so one configuration runs on machines with and without the target compiler. Non-fortran languages pass through (no adapter; Build stays their backstop). See `tools/workflow_conductor.py:_syntax_inproc` / `mcp_servers/build_runtime_server.py:tool_run_syntax_check`.
- `Generate.static`: a **deterministic** substep the conductor runs **in-process** (no leaf) between `Generate.syntax` and `Generate.verify` — it runs `validate_workspace_root.py` then `validate_pipeline_semantics --stage post_generate` over the generated source and writes `static_meta.json`. These are the purely-static gates the `Generate.verify` leaf used to own; running them here means the LLM `Generate.verify` pass is reached only on a deterministically-clean source. A violation re-runs `Generate.generate` via a warm-resume reopen (same as a lint finding); see `tools/workflow_conductor.py:_static_inproc`.
- `Generate.verify`: an independent LLM substep that verifies the source is consistent with the `case` / `algorithm` / `io_contract` / `dependency` / `impl_defaults` sections of `spec.ir.yaml`. It launches **no** validator gate (those are `Generate.static`'s, above); it performs only the LLM semantic checks. Under `pure` it is a `pure-function leaf` reviewing the bundle and assembled source and returns a verify verdict the host projects onto `source_meta.json`.

## `source_id` format
- Format: `src_<YYYYMMDD>_<seq3>`, e.g. `src_20260511_001`

## `source_meta.json` required keys
- Common: `attempt_count`, `verification_status`, `last_fail_reason`, `debug_mode`, `context_isolated`
- When `context_isolated=false`, `constraint_reason` is required.
- `lint_command_ref` is no longer recorded by the leaf: lint is the deterministic conductor-run `Generate.lint` substep, and `post_generate` certifies it against the host-authored `<pipeline_root>/lint_evidence/<source_id>.json` (which the leaf cannot write).

## substep details

### 2-1. Generate.generate substep
- `Generate` runs per `node` and generates source dedicated to the target `node_key`.
- The `generate substep` generates `source/<source_id>/src/` and `source_meta.json`.
- `Generate` must not take `controlled_spec.md` as direct input. The required operation composition is interpreted from the `algorithm` section of `spec.ir.yaml`.
- **M3c host-render scope (R1/M3c-β → M3d).** On an *M3c node* (a make+fortran physics node with exactly one `infrastructure`/harness `direct_dep`) the `runner` is **host-rendered by the conductor**: the leaf authors `<spec_id>_model.f90` + `<spec_id>_checks.f90` (fixed ABI, [docs/workflow/CHECKS_MODULE_CONTRACT.md](../CHECKS_MODULE_CONTRACT.md)) and `<spec_id>_runner.f90` / `src/Makefile` are NOT in its write set. The runner-output and Makefile bullets below then describe the **host-rendered** output (still the contract `Validate.judge` reviews), not a leaf write; they apply as **authoring** rules only to a runner-authoring leaf — an `infrastructure` self-test runner or a legacy no-harness node — for which `RUNNER_OUTPUT_CONTRACT.md` remains the injected must-read. It is NOT injected for an M3c physics generate leaf (node-aware; see `AGENT_SKILLS.md` / `leaf_contract_doc_refs`).
- Regardless of language, generate `model` (physics computation) and `runner` (input/output / execution coordination) separately.
- The `runner` calls the `model` via `call` / `use` / `import`, and must not duplicate the physics-update logic.
- **Runner program output** (`diagnostics.json` / `perf.json` / `raw/` composition, per-case snapshot naming, forbidden judge-artifact filenames, and the Fortran JSON numeric/boolean descriptor rules) is canonical in [docs/workflow/RUNNER_OUTPUT_CONTRACT.md](../RUNNER_OUTPUT_CONTRACT.md) — the must-read for a **runner-authoring** `Generate.generate` leaf (per the M3c scope above, that is a non-M3c node; an M3c physics leaf authors no runner and does not read it). The bullets below restate it for this phase's context; on any divergence, `RUNNER_OUTPUT_CONTRACT.md` governs.
- The `runner` source must not contain, as a substring of its **full text including comment lines**, any of the strings `verdict.json`, `aggregate_verdict.json`, `summary.json`, or `trial_meta.json`.
- The tokens the `runner` writes to JSON output (`diagnostics.json` / `perf.json` / `raw/metrics_basis.json` / `raw/state_snapshots/*.json`) must be restorable by a standard JSON parser. With `impl_defaults.toolchain.language=fortran`, the `F0` / `F0.d` format, which can omit the leading zero for a number, and the `L`-family edit descriptors that produce `T`/`F` for a logical value (`L1` etc.) must not be used directly for a JSON token. Output booleans as the literals `true` / `false`. For the canonical serialization rules and the safe `jnum()` / `jbool()` idiom, refer to [docs/PERFORMANCE_DIAGNOSTICS.md](../../PERFORMANCE_DIAGNOSTICS.md) §6. The post_generate static analysis detects this violation **by descriptor presence** (it does not inspect the runtime output, so a manual leading-zero fixup does not satisfy the gate) and fails it.
- The `perf.json` the `runner` outputs must have, without omission, the required fields of [docs/PERFORMANCE_DIAGNOSTICS.md](../../PERFORMANCE_DIAGNOSTICS.md) §2: `case_id` / `target` / `walltime_sec` / `steps` / `cells_updated` / `throughput_cells_per_sec` / `parallelism` (object: `mpi_ranks` / `threads_per_rank` / `gpu_devices` / `parallel_degree_total`). These are not derived from the IR but are a runner output contract that the canonical doc uniformly requires, and the `post_execute` gate of `Validate.execute` mandatorily verifies `walltime_sec` / `throughput_cells_per_sec` / `parallelism`. A custom minimal schema (e.g. `{case_count, wall_seconds}`) fails.
- The `raw/metrics_basis.json` the `runner` outputs must have a **`per_test` list** carrying one entry per (`test_id`, target `case_id`) pair — every `test_id` of `io_contract.test_evidence_requirements`, once for each case named in that test's `io_contract.test_predicates[].target_cases`. Each entry must hold, without omission, the `test_id`, the `case_id`, and that test's `required_raw_variables` valued from that case (needed for the per-test recomputation of `Validate.judge`). A custom structure without a per-test index (e.g. a single `evidence[]`) is rejected by the `post_execute` gate (`must contain per_test list or tests object`); the legacy `tests` object form is keyed by `test_id` and so cannot express a multi-target test's several entries — it is deprecated.
- When `state_snapshots` is required, the `runner` writes **one snapshot per case** to `raw/state_snapshots/<case_id>.json`, with `<case_id>` taken from its argv (`--cases <spec.ir.yaml> <case_id>...`, one positional per `case.test_case_set[]`). Build the path from that `case_id` (`trim(case_id)//'.json'`), never a fixed/sequential literal (`snapshot_0001.json`) or a single combined file: a wrong name is flagged by `post_generate` static analysis (string-literal names) or fails `Validate.execute`'s per-`<case_id>.json` deliverable gate (runtime-built names). Canonical filename rule: `phase_04_validate.md` §43.
- The `diagnostics.json` the `runner` outputs must satisfy `spec.ir.yaml.io_contract.diagnostics_contract`: it must hold a `checks.<id>` entry for **every** `io_contract.diagnostics_contract.checks[].id`, and — when `diagnostics_contract.verdict.required=true` — a top-level `verdict` object carrying the `diagnostics_contract.verdict.fields` keys (e.g. `verdict.overall` / `verdict.failed_checks`). This is how the runner conveys the `tests.md §3` diagnostics contract that `Generate` cannot read directly (Generate consumes only the IR). A custom per-case array (e.g. only `guard_pass`/`a_x`/`a_y`) that omits the contracted `checks.*`/`verdict` is a `fail`. Emitting a `verdict` **key inside `diagnostics.json`** is permitted and does not conflict with the rule above forbidding the literal filename string `verdict.json` in the runner source (that rule targets the judge artifact *filenames*, not a diagnostics field name).
- When `spec.ir.yaml.impl_defaults.toolchain.language` is a `fortran` / `c` / `cpp` / `mixed` family, the `runner` must not launch an external interpreter such as `python` / `bash` / `sh` / `node`.
- The `model` must perform a numerical state update or a judgment-target operation. A fixed-value-return-only, fixed-`JSON`-output-only, or `no-op`-only implementation is forbidden.
- The `model` of a `node` that has dependencies requires calls to the published `operation` of the dependency `node` resolved by `spec.ir.yaml.dependency.direct_deps`.
- A function equivalent to a dependency `operation` must not be re-implemented in the depending `node`'s `model` / `runner`. When detected, it is a `Generate fail`.
- When the dependency is a `profile` with no published `operation`, the depending `problem` requires recording an implementation trace that references the `profile`'s selection result and constraints.
- `Generate` must not copy, relocate, or redefine the implementation body equivalent to a dependency `node`'s `source/<source_id>/src/` into the depending `node`'s `src/`. When detected, it is a `Generate fail`. (Carve-out: the deterministic dependency build — Model B, `docs/design/deterministic_followups.md` — has the conductor stage a dependency's `<dep>_model.f90` into the per-run `$(OBJDIR)`, never the canonical `src/`, so it is not a copy/violation.)
- `Generate` must confirm, as input consistency, the `ir_ref`, `pipeline_ref`, and `aggregate_verdict` of the immediate dependency `node`. When it detects that the dependency `node`'s workflow is incomplete, do not substitute-generate the dependency's code; stop with `blocked` or `fail`.
- The `model` / `runner` must not introduce, into a judgment metric, any arbitrary constant scaling, constant offset, or case-dependent correction without physical basis. A correction other than the evaluation expression explicitly defined in `controlled_spec.md` or `tests.md` is forbidden.
- With `impl_defaults.toolchain.language=fortran`, the node's own `module` name must be the **literal** `<spec_id>_model` (the `spec_id` is prefixed verbatim; an abbreviation or derived short form is **not** permitted), and the published `subroutine` names must use the literal prefix `<spec_id>__<op>`. The literal `spec_id` is load-bearing, not merely a collision-avoidance device: a depending node is required to `use <spec_id>_model` / `call <spec_id>__*`, so an abbreviated module name breaks dependency resolution. The deterministic `post_generate` validator accepts only this exact name and rejects any other (it reports the required `<spec_id>_model.f90`, and when a model source was emitted under a different name it names the offending file and instructs a rename). If a `spec_id` is long enough that `<spec_id>_model` or `<spec_id>__<op>` would exceed the f2008 63-char identifier limit, do **not** abbreviate at `Generate`'s discretion — treat it as a `spec`-level problem (the `spec_id` is too long) and stop, rather than emitting a name the validator will reject.
- **The `<spec_id>__` prefix marks the node's published surface.** For a `spec_kind: infrastructure` node that surface is exact: the `operation_id` entries of `public_api.published_operations[]` plus the derived types of `public_api.published_types` (both prefixed). A physics node declares no such set (its interface is derived post-hoc — [phase_01_compile.md](phase_01_compile.md)), so its published operations are the entry points realizing `io_contract.outputs`, which a depending node resolves as `call <spec_id>__<op>`; the intermediate `algorithm.steps[].operation_ref` names are **not** published (when the IR does carry `algorithm.published_operations`, that list is the set). An internal helper `subroutine` / `function` takes a **bare** name; a bare name may still be `module`-public when the `<spec_id>_checks` module `use`s it (the `module` scope keeps a bare name collision-free — canonical: [docs/workflow/CHECKS_MODULE_CONTRACT.md](../CHECKS_MODULE_CONTRACT.md) §1). A public `procedure` carrying the `<spec_id>__` prefix that is outside the published surface is a `Generate.verify` `fail` (G4: an `operation` the node does not publish).
- With `impl_defaults.toolchain.language=fortran`, the source file name requires `<module_name>.f90` matching the defined `module` name (i.e. `<spec_id>_model.f90`).
- For an `infrastructure` node (`spec_kind: infrastructure`), the `model` must publish every `public_api` symbol with its IR `public_api.signatures[].interface` **verbatim** — same argument names, order, types, ranks, `intent`s, `result` name, and (for a published derived type) component layout. Because `Generate.generate` cannot read `controlled_spec.md` (§2-1), the IR's `public_api.signatures` (which Compile transcribed from the controlled_spec §5.1 canonical interface block and the `--stage compile` gate pinned == §5.1) is the only carrier of the signatures. The deterministic `Generate.static` gate (`validate_pipeline_semantics --stage post_generate`, `_validate_infrastructure_generated_signatures`) normalizes (comments stripped, `&` continuations joined, case-folded, whitespace-insensitive) and requires every §5.1 interface line to be present in `<spec_id>_model.f90`; an argument-name / type / rank / `intent` / `result` / component drift warm-resumes `Generate.generate`. This moves signature-exactness off the `Generate.verify` leaf (and the Build link error) to a cheap deterministic check.
- With `impl_defaults.toolchain.language=fortran`, the `model` of a `node` that has a dependency `component` requires `use <spec_id>_model` and `call <spec_id>__*` per dependency `spec_id`, and forbids redefining `subroutine <spec_id>__*`. Each `call <dep>__<op>(...)` must pass arguments in the **exact positional order** of the dependency's published interface — never guessed. The conductor injects that order into the launch prompt's `<dependency_facts>` ("Published dependency operations"), resolved from the dependency's certified source (the exact source Build links); use it verbatim. Under each op's header the block also lists **each dummy argument's declared type, intent, and rank/shape**; the actual argument you pass must **match the dummy's rank and shape**, not only the order. When a dummy is lower-rank than your full state array (e.g. a rank-2 `(:,:)` field dummy vs. your rank-3 `(ncomp,:,:)` state), **loop over the extra component/dimension and pass lower-rank slices** (`call <dep>__<op>(u(c,:,:), ...)`) — never pass the whole higher-rank array. Fortran is positional and rank-checked: a wrong order **or** a rank/shape mismatch fails Build. See the `generate` `SKILL`.
- **Inert dependency call.** Each dependency `operation` is judged individually by whether `spec.ir.yaml` declares a sink for its results. An operation whose result corresponds to a variable in any `algorithm.steps[]` entry (`inputs` OR `outputs` — a result can flow straight from a step's `outputs` to an `io_contract` output with no intervening `steps[].inputs` reference), to a `derived_field_rules` / `invariants` quantity, an `io_contract` output, or a `semantic_dependency.required_sources` variable is **load-bearing**: its results are bound to those declared variables and flow exactly as declared (IR variable names carry no provenance marker, so the correspondence is resolved from the step semantics; an uncertain correspondence is treated as load-bearing). Only when no such sink exists — every declared output is computable from the case/request facts and the node's own step outputs (typical for a `profile` node whose `algorithm` resolves request facts only) — is the call authored **inert** (`docs/GLOSSARY.md`): every actual argument is a local variable assigned before the call, and no call result reaches an `io_contract` output, a `diagnostics_contract` check input, or an `algorithm.invariants` quantity. An invented purpose for an inert call — a probe configuration, an acceptance threshold or tolerance, a validity/binding flag folded into an output, a conditional branch on the call's results — is an execution path absent from `spec.ir.yaml.algorithm` and a `Generate.verify` `fail`. The inert form does not collide with the deterministic gates: an actual assigned before the call is not counted as a dependency result by the `Generate.static` dependency-dataflow gate (which also scopes to `problem/` nodes), and a variable passed as an actual argument is a use for the `Generate.syntax` unused-variable gate.
- **`src/Makefile` authorship:** for any `build_system=make` ∧ `language=fortran` node — **leaf or with dependencies** — the conductor authors `src/Makefile` host-side (runtime-owned, like `lineage.json`; not in the generate leaf's `allowed_output_paths`). For a dependency node it adds the closure object rules (Model B), each `<dep>_model.f90` staged into `$(OBJDIR)` at Build. The Makefile rules below document what that authored Makefile satisfies and what a non-fortran make family (LLM-authored) must produce.
- With `impl_defaults.toolchain.build_system=make` and `impl_defaults.toolchain.language=fortran`, the generated `src/Makefile` makes explicit the object dependencies corresponding to the `use` dependencies, and requires writing the dependency `.o` into each target's prerequisites.
- With `impl_defaults.toolchain.build_system=make` and `impl_defaults.toolchain.language=fortran`, write the object-rule targets and prerequisites **in a form resolvable to a literal `.o`/`.mod` basename**. **For an out-of-source build (`$(OBJDIR)` override), when a build-rule target is prefixed `$(OBJDIR)/` (e.g. `$(OBJDIR)/foo.o:`), the prerequisite consuming it must carry the same `$(OBJDIR)/` prefix (or a variable expanding to it). A bare `foo.o`/`foo.mod` becomes, under override, a target with no rule, and `make -j` stops with `No rule to make target`** (post_generate detects the prefix mismatch). For in-source-only rules, keep both targets and prerequisites bare. A `$(VAR)` prerequisite is allowed when the variable is defined **before the rule** (a forward reference expands empty → missing dependency); see `docs/RUNBOOK.md` §1-2-1.
- With `impl_defaults.toolchain.build_system=make` and `impl_defaults.toolchain.language=fortran`, the `src/Makefile` must not cause a failure due to a missing dependency in a parallel build (e.g. `make -j 4`).
- **Out-of-source output contract (`build_system=make`):** the `src/Makefile` parameterizes `OBJDIR ?= .` / `BINDIR ?= .` / `RUNDIR ?= .` **and `BIN ?= <spec_id>_runner`** with `?=` defaults (`BIN` must be `?=` so Build/`Validate.execute` impose `<spec_id>_runner` — see [phase_03_build.md](phase_03_build.md)). Object/module go to `$(OBJDIR)` (`gfortran` uses `-J$(OBJDIR)` / `-I$(OBJDIR)`), the execution binary to `$(BINDIR)/$(BIN)`, and `test`/`check` output under `$(RUNDIR)`. The `test`/`check` target also declares `SPEC ?=` and `CASES ?=` (`?=` overridable) and invokes the runner as `$(BINDIR)/$(BIN) --cases $(SPEC) $(CASES)` so `Validate.execute`'s make-test re-run matches `run_program`'s argv (`Validate.execute` imposes the authoritative `SPEC`/`CASES` via the make-test env; the `?=` defaults — `SPEC ?= spec.ir.yaml` and `CASES ?= <all case_ids>` — keep a local `make all test` runnable standalone). The generated `src/` stays pristine (only the MCP audit log); `Build` overrides put the object in a per-run tmp and the exe in `binary/<binary_id>/bin/`. The `?=` default `.` keeps a local plain `make` in-source.
- Within the same `pipeline`, the same `src` must not be copied to a different `node_key`. Make commonization explicit as a common library.
- With `impl_defaults.target.class=cpu` and no explicit specification of the loop parallelization method, add `OpenMP` to parallelizable loops.
- When the physics update cannot be implemented, it is a `Generate fail`, and a fixed string or fixed `JSON` must not be output as a substitute.
- `static lint` is NOT run by the `Generate.generate` leaf. It is the deterministic `Generate.lint` substep the conductor runs in-process over `source/<source_id>/src/` as `project_dir`, with the `run_linter` `preset` corresponding to `impl_defaults.toolchain.language` (e.g. `fortitude` for `fortran` / `cuda_fortran`, `cppcheck` for `c` / `cpp` / `cuda_c`, two pieces of evidence of `fortitude` and `cppcheck` for `mixed`, `ruff` for `python`). It is not run via the `Makefile`'s `lint` target or via `compile_project`. The registration of the `build-runtime` MCP server including `run_linter`, and the tool-permission grant to the child `Agent` session, are verified by `preflight` as a workflow-start premise (on the Claude backend, both checks `claude_mcp_build_runtime_registered` ∧ `claude_mcp_build_runtime_permission_granted`; for details see [docs/RUNBOOK.md](../../RUNBOOK.md) §0-2).
- **fortitude default-rule idioms (write Fortran source that passes `static lint` on the first attempt — this is the canonical checklist; a finding warm-resumes `Generate.generate` and costs a regenerate cycle):** the conductor's `Generate.lint` substep runs `fortitude check` with its default ruleset, so generate source that already satisfies these rules up front:
  - **S001 (line length ≤ 100 columns):** split long `subroutine` signatures and `write` statements at a `&` continuation (break between arguments or `//`-concatenated pieces, never inside a string literal).
  - **C121 (`use` must have `only:`):** every `use` names the imported entities, e.g. `use <dep>_model, only: dp, <dep>__<op>`.
  - **C122 (intrinsic module `use` needs the `intrinsic` modifier):** import from an intrinsic module as `use, intrinsic :: iso_fortran_env, only: error_unit, int64` (the `, intrinsic ::` form), not a bare `use iso_fortran_env, only: ...`.
  - **PORT011 (no number-literal integer kind):** declare kind-qualified integers with a named kind (it fires for any literal kind, e.g. `integer(4)` too), `integer(int64) :: clock0` — never `integer(8)`. `int64` comes from `iso_fortran_env`, so importing it pairs with **C122**: write `use, intrinsic :: iso_fortran_env, only: int64`. (Fixing PORT011 by adding a bare `use iso_fortran_env` is the common cascade that introduces a fresh C122 finding on the next attempt — add the `intrinsic` modifier in the same edit.)
  - **C011 (`select case` needs a default):** every `select case` block ends with a `case default` arm (even if it only guards an unreachable value).
  - **C003 ↔ `-std=f2008` conflict (required workaround):** rule **C003** wants the F2018 spec-list `implicit none (type, external)`, but under `impl_defaults.toolchain.standard=f2008` (Makefile `FFLAGS=-std=f2008`) that form is a `compile_error` (`Error: Fortran 2018: IMPLICIT NONE with spec list`) — a structural conflict. Use plain `implicit none` (no spec list) and disable only C003 via the **inline `! allow(C003)` directive** placed on the line **immediately before** `implicit none`. The `[check] ignore=["C003"]` route in `src/<source_id>/src/fortitude.toml` is rejected by the output-manifest source-extension restriction, so use the inline directive, not a config file. Concretely — ✅ correct: `! allow(C003)` on its own line, then plain `implicit none` on the next line. ❌ `implicit none  ! allow(C003)` (directive on the **same** line) — C003 still fires and FORT002 additionally flags the comment as an unused allow. ❌ `implicit none (external)` (the spec-list form fortitude's C003 "help" suggests) — a build-breaking `compile_error` under `-std=f2008`; the deterministic `Generate.syntax` gate (real `gfortran -fsyntax-only -std=f2008`) rejects it, warm-resuming `Generate.generate`. Do not oscillate between these two wrong forms — only the ✅ form passes both lint and the syntax gate.
- **Write standard-conforming f2008 — the `Generate.syntax` gate compiles your source.** The conductor's deterministic `Generate.syntax` substep runs the REAL compiler front-end (`gfortran -fsyntax-only -std=f2008`, with the dependency closure staged) over `source/<source_id>/src/*.f90` before `Generate.static`, so **every** syntax / standard-conformance `compile_error` — not a curated subset — is caught there and warm-resumes `Generate.generate` (this replaced the former `post_generate` text heuristics). Known instances that used to slip through to Build and are now caught at the gate: an identifier over the 63-char f2008 name limit (keep generated names within the limit; if `<spec_id>_model` / `<spec_id>__<op>` itself would exceed it, that is a spec-level problem — do not abbreviate); a non-constant `STOP` / `ERROR STOP` code (❌ `error stop 'unknown case_id: '//cid` — under `-std=f2008` the code must be a scalar CHARACTER/INTEGER **constant** expression; ✅ use a fixed literal, or `print`/`write` the run-time value first and then `error stop <integer>`); the `implicit none (...)` spec-list form above. The gate additionally promotes `-Werror=unused-dummy-argument -Werror=unused-variable` over the whole staged set, so a dummy argument the interface fixes but the body never consumes (an inert input the algorithm defines as unused; an ABI-fixed `name` / `case_id`) is a `compile_error` — ✅ keep it a live `intent(in)` dummy and bind it immediately after the declarations with `associate (unused_<name> => <name>)` / `end associate`; ❌ an arithmetic no-op (`0*<name>`); ❌ `! allow(...)` (fortitude has no unused-symbol rule and an unknown-rule allow self-rejects under FORT001 / FORT002). An unused local variable is deleted, not bound. Canonical: [CHECKS_MODULE_CONTRACT.md](../CHECKS_MODULE_CONTRACT.md) §5. Getting these right up front avoids a regenerate cycle, exactly as with the fortitude idioms.
  - **Fortran identifiers are case-insensitive.** A pair of names differing only in case (a dummy `g` and an output `G`; a dummy `U_L` and a local `u_l`) is one identifier in the same scope, so it is a `compile_error` — a duplicate declaration, or an `intent` conflict on the dummy. When an IR-derived name collides this way, rename the **local / dummy** side; the IR-derived name stays verbatim.
- **The `Generate.static` gate traces the dependency dataflow into `intent(out)`.** On a `problem` `node` with dependencies, declare the incoming state `intent(in)` and the updated state (the `io_contract.outputs` / `algorithm.required_update_paths` variables) **`intent(out)`**, and reach each `intent(out)` dummy by an assignment chain from the values the `call <dep>__<op>(...)` results carry (likewise for `io_contract.semantic_dependency.required_sources`). The gate inspects each `subroutine` that declares at least one `intent(out)` dummy; a dependency result consumed only by an `intent(inout)` in-place update reaches no `intent(out)` variable and is a `Generate fail` (`does not propagate dependency operation outputs to intent(out) dataflow`) that warm-resumes `Generate.generate`. A `subroutine` with no `intent(out)` dummy is skipped by the gate, but it exposes no `intent(out)` expression tree for the G5 review below either. This is the deterministic counterpart of G5; canonical authoring rule: `skills/workflow-generate-generate/SKILL.md`.
- `Generate` must include in its official output a preset-compatible quality path with which `Validate.execute` can run using only the `preset` of `run_quality_checks`. On shortage, it is a `Generate fail`.
- With `impl_defaults.toolchain.build_system=make` and `impl_defaults.toolchain.language=fortran` / `c` / `cpp` / `mixed` families, a `test` or `check` target must be defined in `source/<source_id>/src/Makefile`. On absence, it is a `Generate fail`. The relevant target references the existing binary in `$(BINDIR)` with a **non-relinking** guard that fails closed when the binary is absent — `test -x $(BINDIR)/$(BIN) || { echo "error: $(BINDIR)/$(BIN) not built; run 'make all' first" >&2; exit 1; }` — and **must not** invoke `$(MAKE)` in the guard. Relinking inside `Validate.execute` writes into the read-only-bound `binary/` (outside the execute `runs/` write_root) and escalates a mere binary-name/availability mismatch into an `unauthorized_write_violation` → `fail_closed`; the fail-closed guard instead yields a clean retryable `make test` failure. For local use, run `make all test` (the `?=` in-source defaults are unchanged). The target generates the output destination with `mkdir -p $(RUNDIR)` (and the `raw/` required subdirectories the runner writes) **before** `cd $(RUNDIR)`, then runs the runner as `$(BINDIR)/$(BIN) --cases $(SPEC) $(CASES)` and confines the run output under `$(RUNDIR)`. The `--cases $(SPEC) $(CASES)` argv is **required**: the runner consumes its case set from `--cases` (and aborts without it), and `Validate.execute` imposes the same `SPEC`/`CASES` `run_program` uses so the make-test re-run's diagnostics match — a bare `$(BINDIR)/$(BIN)` makes the make-test candidate emit no `diagnostics.json` and fails `quality_check` (`post_generate` flags a `test`/`check` recipe that runs the binary without `--cases`). Because `$(RUNDIR)` can point to a not-yet-created run node dir, omitting `mkdir -p` makes `cd` fail.

### 2-2. Generate.verify substep
The `post_generate` static gate is **not** run by `Generate.verify`. The conductor's deterministic `Generate.syntax` substep (the compiler front-end gate) and then the `Generate.static` substep — `validate_workspace_root.py` then `python3 tools/validate_pipeline_semantics.py --stage post_generate --pipeline-root workspace/pipelines/<node_key_safe>/<pipeline_id>/ --source-id <source_id>` (`exit code 0` required) — both run **before** `Generate.verify`; a violation warm-resumes `Generate.generate` and `Generate.verify` never starts. So verify is reached only on a deterministically-clean source and performs only the LLM semantic checks below.

`Generate.verify` must not run the write-family tools of the `build-runtime` MCP including `run_linter` (`run_linter` / `run_syntax_check` / `compile_project` / `run_program` / `run_quality_checks`) (`static lint` is the deterministic `Generate.lint` substep and the compiler syntax gate is the deterministic `Generate.syntax` substep, both conductor-run). `Generate.verify` launches **no** external validator gate: `validate_workspace_root.py` and `validate_pipeline_semantics --stage post_generate` both run in the conductor's `Generate.static` substep (the `post_generate` run there certifies the conductor lint AND syntax gate by *inspecting* the host-authored `<pipeline_root>/lint_evidence/<source_id>.json` / `<pipeline_root>/syntax_evidence/<source_id>.json` and the referenced `command_log.jsonl` records). Because verify's `allowed_output_paths` does not authorize the side output of `run_linter` (`command_log.jsonl`), running `run_linter` from verify makes that write outside the manifest and invites an `unauthorized_write_violation` → `fail_closed`.

`Generate.verify` required verification items:

#### G1. case coverage
- All `case_id` of `spec.ir.yaml.case.test_case_set` and all expanded `case` are reachable from the implementation path of the `runner` or `model`.
- When an unimplemented `case`, an unreachable branch, or a fixed-`case_id`-limited implementation is detected, it is a `fail`.
- When an implementation in which the `model` composes the judgment metric only with `case_id` branching and fixed-numeric assignment is detected, it is a `fail`.

#### G2. case input acceptance
- The `runner` and `model` accept each runtime input of `spec.ir.yaml.case.test_case_set[].inputs`.
- When the input-propagation or recording path corresponding to at least `case_id`, the grid conditions, the time conditions, the initial-condition identifier, the boundary-condition identifier, the `profile`- or `component`-selection result, and the `test_profile_id` and `test_profile_version` derived from `tests.md` cannot be confirmed, it is a `fail`.

#### G3. implementation of case-dependent differences
- The per-allowed-selection-value difference implementation in `case.test_case_set[].inputs` is not collapsed into a fixed constant or a single default value.
- When an implementation that ignores case-dependent inputs such as `boundary`, `initial_profile`, `topography_profile`, `dt_rule`, `refinement`, or the `sweep` expansion result is detected, it is a `fail`.

#### G4. algorithm reflection
- Based on `spec.ir.yaml.algorithm.steps`, `ordering`, `control_condition`, and `iteration_contract`, the `test case set` coverage, operation composition, dependency `operation`, and the data dependency of the output metrics are reflected in the generated code.
- `algorithm.update_semantics`, `temporaries`, `derived_field_rules`, `invariants`, and `splitting_policy` are reflected in the generated code. When a missing state-update target, an unimplemented derived-quantity computation, an update order that breaks an invariant that should be conserved, or a `splitting_policy` mismatch is detected, it is a `fail`.
- The generated code does not introduce an additional operation, additional iteration, additional conditional branch, or additional dependency `operation` call not described in `algorithm`. When a guessed re-introduction of `controlled_spec.md`-derived information is detected, it is a `fail`. The `associate (unused_<name> => <name>)` / `end associate` binding of an intentionally-unused dummy argument ([CHECKS_MODULE_CONTRACT.md](../CHECKS_MODULE_CONTRACT.md) §5) is **not** an additional operation: it computes nothing, it is required by the deterministic `Generate.syntax` gate, and it needs no `algorithm` step to license it. Failing a source for carrying it would oscillate against that gate (remove the binding → the gate rejects the dead dummy → restore it), so its **absence** on an inert dummy, not its presence, is the defect. An **inert dependency call** (the requirement above; `docs/GLOSSARY.md`) is likewise **not** an additional dependency `operation` call: `spec.ir.yaml.dependency.direct_deps` licenses it without an `algorithm` step, and failing a source for carrying it would oscillate against the dependency-call mandate — its results feeding a declared output, not the call itself, is the defect.

#### G5. io_contract consistency
- When a constant output unrelated to the `model` output, a fixed-`JSON` output, or `diagnostics` generation by direct assignment of an analytic expression is detected, it is a `fail`.
- Verify that the final expression tree of an `intent(out)` variable reaches the group of output variables declared in `spec.ir.yaml.io_contract.semantic_dependency.required_sources` and `io_contract.outputs`.
- Reconcile `spec.ir.yaml.io_contract.raw_requirements.required_evidence` and `test_evidence_requirements` with the `runner`'s raw-evidence output design, and verify that the raw evidence needed for `Validate.judge` recomputation is held for every `test_id`. When a configuration in which the `runner` writes out only a whole-suite summary, a configuration that overwrites and aggregates the primary evidence of multiple `test` into one, or a missing `required_raw_variables` is detected, it is a `fail`.
- Verify that the `runner` emits, in `diagnostics.json`, a `checks.<id>` entry for every `io_contract.diagnostics_contract.checks[].id`, and — when `io_contract.diagnostics_contract.verdict.required=true` — a top-level `verdict` object holding the `verdict.fields` keys. When a `checks.*` key required by the contract is missing, or the required `verdict` object is absent, it is a `fail`.

#### G6. impl_defaults reflection
- `spec.ir.yaml.impl_defaults.target.class`, `target.backend`, `target.architecture`, `toolchain.language`, `toolchain.standard`, `toolchain.build_system`, and `selected.backend_key` are reflected in the generated source composition and the `build` artifacts.
- Verify that the execution-algorithm choices specified in `impl_defaults.abstract` and `backend_overrides` — parallelization, layout, fusion, tiling, vectorization, asynchrony, etc. — are reflected in the generated code or `build` settings to the extent expressible in the target language and target. When ignoring of a specified knob, the mixing-in of an optimization for a forbidden target, or a violation of the default `OpenMP` rule for `target.class=cpu` is detected, it is a `fail`.

#### G7. dependency consistency
- Verify that the generated code does not introduce a reference to a dependency `node` not present in `spec.ir.yaml.dependency` or to an undeclared `operation`. When a call outside `direct_deps`, an unresolved `component` reference, or an implementation choice contradicting a `profile` constraint is detected, it is a `fail`.
- Verify that each emitted `call <dep>__<op>(...)` passes its positional arguments in the order **and rank/shape** of the conductor-resolved published-operation interface in `<dependency_facts>` (each dummy's type/intent/rank is listed under its header): confirm the actual argument's rank matches the dummy's rank — in particular that a lower-rank dummy (e.g. rank-2 `(:,:)`) is fed a matching slice inside a component loop, not the whole higher-rank state array. A deterministic order/rank gate is infeasible (Fortran is positional and the consumer uses its own local names), so this is an LLM check; Build's compiler is the deterministic backstop (an order or rank/shape mismatch fails Build and routes back to `Generate`). See the `verify` `SKILL`.

## On-failure behavior
- A `Generate fail` retry defaults to an in-phase retry.
- When `Generate.verify` judges that "the IR itself has an error", record "ir_inconsistency" in `source_meta.json.last_fail_reason`, and the `orchestration agent` decides whether to go back to `Compile`. When `Compile` is already checkpointed `pass`, the return to `Compile` first runs `reopen-phase --from-phase compile --node-key <node_key> --trigger-agent-run-id <generate_verify_fail_substep_agent_run_id> --reason <reason_code>` to invalidate the stale `Compile` checkpoint / `step_result` and reset its `phase_state`, then re-runs `Compile` → `Generate`. Canonical: `docs/CLI_REFERENCE_RARE.md#reopen-phase`, `docs/RUNBOOK.md` §3-1.
