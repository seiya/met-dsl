# Phase 2: Generate

## Overview
The phase that, with `spec.ir.yaml` as the canonical input, generates the source code of the target `node`'s `model` (physics computation) and `runner` (input/output / execution coordination). `Generate.generate` reads only `spec.ir.yaml` (§2-1); `Generate.verify` also reads `controlled_spec.md` (§2-2).

## I/O contract
- execution input: `spec.ir.yaml`
- verification input: `spec.ir.yaml`, `controlled_spec.md` (verify-only), the generated `source/<source_id>/src/`
- output: `workspace/pipelines/<node_key_safe>/<pipeline_id>/source/<source_id>/src/`, `source_meta.json`

## substep structure
- `Generate.generate`: the LLM substep that reads `spec.ir.yaml` and generates the full source set.
- `Generate.verify`: an independent LLM substep that verifies the source is consistent with the `case` / `algorithm` / `io_contract` / `dependency` / `impl_defaults` sections of `spec.ir.yaml`.

## `source_id` format
- Format: `src_<YYYYMMDD>_<seq3>`, e.g. `src_20260511_001`

## `source_meta.json` required keys
- Common: `attempt_count`, `verification_status`, `last_fail_reason`, `debug_mode`, `context_isolated`
- When `context_isolated=false`, `constraint_reason` is required.
- When `verification_status=pass`, `lint_command_ref.run_linter[]` (`command_id`, `command_log_ref`, `preset`) is required.

## substep details

### 2-1. Generate.generate substep
- `Generate` runs per `node` and generates source dedicated to the target `node_key`.
- The `generate substep` generates `source/<source_id>/src/` and `source_meta.json`.
- `Generate` must not take `controlled_spec.md` as direct input. The required operation composition is interpreted from the `algorithm` section of `spec.ir.yaml`.
- Regardless of language, generate `model` (physics computation) and `runner` (input/output / execution coordination) separately.
- The `runner` calls the `model` via `call` / `use` / `import`, and must not duplicate the physics-update logic.
- The `runner` source must not contain, as a substring of its **full text including comment lines**, any of the strings `verdict.json`, `aggregate_verdict.json`, `summary.json`, or `trial_meta.json`.
- The tokens the `runner` writes to JSON output (`diagnostics.json` / `perf.json` / `raw/metrics_basis.json` / `raw/state_snapshots/*.json`) must be restorable by a standard JSON parser. With `impl_defaults.toolchain.language=fortran`, the `F0` / `F0.d` format, which can omit the leading zero for a number, and the `L`-family edit descriptors that produce `T`/`F` for a logical value (`L1` etc.) must not be used directly for a JSON token. Output booleans as the literals `true` / `false`. For the canonical serialization rules and the safe `jnum()` / `jbool()` idiom, refer to [docs/PERFORMANCE_DIAGNOSTICS.md](../../PERFORMANCE_DIAGNOSTICS.md) §6. The post_generate static analysis detects this violation **by descriptor presence** (it does not inspect the runtime output, so a manual leading-zero fixup does not satisfy the gate) and fails it.
- The `perf.json` the `runner` outputs must have, without omission, the required fields of [docs/PERFORMANCE_DIAGNOSTICS.md](../../PERFORMANCE_DIAGNOSTICS.md) §2: `case_id` / `target` / `walltime_sec` / `steps` / `cells_updated` / `throughput_cells_per_sec` / `parallelism` (object: `mpi_ranks` / `threads_per_rank` / `gpu_devices` / `parallel_degree_total`). These are not derived from the IR but are a runner output contract that the canonical doc uniformly requires, and the `post_execute` gate of `Validate.execute` mandatorily verifies `walltime_sec` / `throughput_cells_per_sec` / `parallelism`. A custom minimal schema (e.g. `{case_count, wall_seconds}`) fails.
- The `raw/metrics_basis.json` the `runner` outputs must have a **`per_test` list (or `tests` object)** that targets all `test_id` of `io_contract.test_evidence_requirements`, and each entry must hold, without omission, the `test_id` and its `required_raw_variables` (needed for the per-test recomputation of `Validate.judge`). A custom structure without a per-test index (e.g. a single `evidence[]`) is rejected by the `post_execute` gate (`must contain per_test list or tests object`).
- The `diagnostics.json` the `runner` outputs must satisfy `spec.ir.yaml.io_contract.diagnostics_contract`: it must hold a `checks.<id>` entry for **every** `io_contract.diagnostics_contract.checks[].id`, and — when `diagnostics_contract.verdict.required=true` — a top-level `verdict` object carrying the `diagnostics_contract.verdict.fields` keys (e.g. `verdict.overall` / `verdict.failed_checks`). This is how the runner conveys the `tests.md §3` diagnostics contract that `Generate` cannot read directly (Generate consumes only the IR). A custom per-case array (e.g. only `guard_pass`/`a_x`/`a_y`) that omits the contracted `checks.*`/`verdict` is a `fail`. Emitting a `verdict` **key inside `diagnostics.json`** is permitted and does not conflict with the rule above forbidding the literal filename string `verdict.json` in the runner source (that rule targets the judge artifact *filenames*, not a diagnostics field name).
- When `spec.ir.yaml.impl_defaults.toolchain.language` is a `fortran` / `c` / `cpp` / `mixed` family, the `runner` must not launch an external interpreter such as `python` / `bash` / `sh` / `node`.
- The `model` must perform a numerical state update or a judgment-target operation. A fixed-value-return-only, fixed-`JSON`-output-only, or `no-op`-only implementation is forbidden.
- The `model` of a `node` that has dependencies requires calls to the published `operation` of the dependency `node` resolved by `spec.ir.yaml.dependency.direct_deps`.
- A function equivalent to a dependency `operation` must not be re-implemented in the depending `node`'s `model` / `runner`. When detected, it is a `Generate fail`.
- When the dependency is a `profile` with no published `operation`, the depending `problem` requires recording an implementation trace that references the `profile`'s selection result and constraints.
- `Generate` must not copy, relocate, or redefine the implementation body equivalent to a dependency `node`'s `source/<source_id>/src/` into the depending `node`'s `src/`. When detected, it is a `Generate fail`.
- `Generate` must confirm, as input consistency, the `ir_ref`, `pipeline_ref`, and `aggregate_verdict` of the immediate dependency `node`. When it detects that the dependency `node`'s workflow is incomplete, do not substitute-generate the dependency's code; stop with `blocked` or `fail`.
- The `model` / `runner` must not introduce, into a judgment metric, any arbitrary constant scaling, constant offset, or case-dependent correction without physical basis. A correction other than the evaluation expression explicitly defined in `controlled_spec.md` or `tests.md` is forbidden.
- With `impl_defaults.toolchain.language=fortran`, the node's own `module` name must be the **literal** `<spec_id>_model` (the `spec_id` is prefixed verbatim; an abbreviation or derived short form is **not** permitted), and the published `subroutine` names must use the literal prefix `<spec_id>__<op>`. The literal `spec_id` is load-bearing, not merely a collision-avoidance device: a depending node is required to `use <spec_id>_model` / `call <spec_id>__*`, so an abbreviated module name breaks dependency resolution. The deterministic `post_generate` validator accepts only this exact name and rejects any other (it reports the required `<spec_id>_model.f90`, and when a model source was emitted under a different name it names the offending file and instructs a rename). If a `spec_id` is long enough that `<spec_id>_model` or `<spec_id>__<op>` would exceed the f2008 63-char identifier limit, do **not** abbreviate at `Generate`'s discretion — treat it as a `spec`-level problem (the `spec_id` is too long) and stop, rather than emitting a name the validator will reject.
- With `impl_defaults.toolchain.language=fortran`, the source file name requires `<module_name>.f90` matching the defined `module` name (i.e. `<spec_id>_model.f90`).
- With `impl_defaults.toolchain.language=fortran`, the `model` of a `node` that has a dependency `component` requires `use <spec_id>_model` and `call <spec_id>__*` per dependency `spec_id`, and forbids redefining `subroutine <spec_id>__*`.
- With `impl_defaults.toolchain.build_system=make` and `impl_defaults.toolchain.language=fortran`, the generated `src/Makefile` makes explicit the object dependencies corresponding to the `use` dependencies, and requires writing the dependency `.o` into each target's prerequisites.
- With `impl_defaults.toolchain.build_system=make` and `impl_defaults.toolchain.language=fortran`, write the object-rule targets and prerequisites **in a form resolvable to a literal `.o`/`.mod` basename**. **For the validity of an out-of-source build (`$(OBJDIR)` override), if the build-rule target of a dependency object is prefixed with `$(OBJDIR)/` like `$(OBJDIR)/foo.o:`, then the prerequisite that consumes it must also be written with the same `$(OBJDIR)/` prefix (`$(OBJDIR)/foo.o` / `$(OBJDIR)/foo.mod`, or a variable that expands to `$(OBJDIR)/`). A bare basename (`foo.o` / `foo.mod` without a prefix) becomes, on `OBJDIR` override, a separate target with no build rule, and `make -j` stops with `No rule to make target`, so it fails** (the `$(OBJDIR)/` prefix is not cosmetic, and the post_generate static analysis detects a prefix mismatch). When writing all rules bare for in-source only, align both targets and prerequisites as bare (e.g. `foo.o:`). When writing prerequisites/targets with a variable (e.g. `$(MODEL_OBJ)` referencing `MODEL_OBJ = $(OBJDIR)/foo_model.o`), it is also allowed because the static analysis expands a simple variable definition (`=` / `:=` / `?=` / `+=`) within the same Makefile and resolves it including the `$(OBJDIR)/` prefix. However, define the variable **before the relevant rule** (because make immediately expands prerequisites at read time, a forward reference to a later definition expands empty and becomes a missing dependency). Using an undefined variable with no definition in the same Makefile (which cannot be expanded and leaves no literal basename) is judged as a missing dependency.
- With `impl_defaults.toolchain.build_system=make` and `impl_defaults.toolchain.language=fortran`, the `src/Makefile` must not cause a failure due to a missing dependency in a parallel build (e.g. `make -j 4`).
- **Out-of-source output contract (`build_system=make`):** the `src/Makefile` parameterizes `OBJDIR ?= .` / `BINDIR ?= .` / `RUNDIR ?= .` **and `BIN ?= <spec_id>_runner`** with `?=` defaults (`BIN` must be `?=` so Build/`Validate.execute` impose `<spec_id>_runner` — see [phase_03_build.md](phase_03_build.md)). Object/module go to `$(OBJDIR)` (`gfortran` uses `-J$(OBJDIR)` / `-I$(OBJDIR)`), the execution binary to `$(BINDIR)/$(BIN)`, and `test`/`check` output under `$(RUNDIR)`. The generated `src/` stays pristine (only the MCP audit log); `Build` overrides put the object in a per-run tmp and the exe in `binary/<binary_id>/bin/`. The `?=` default `.` keeps a local plain `make` in-source.
- Within the same `pipeline`, the same `src` must not be copied to a different `node_key`. Make commonization explicit as a common library.
- With `impl_defaults.target.class=cpu` and no explicit specification of the loop parallelization method, add `OpenMP` to parallelizable loops.
- When the physics update cannot be implemented, it is a `Generate fail`, and a fixed string or fixed `JSON` must not be output as a substitute.
- `Generate.generate` treats `source/<source_id>/src/` as the `project_dir` and runs `static lint` with the `run_linter` `preset` corresponding to `impl_defaults.toolchain.language` (e.g. `fortitude` for `fortran` / `cuda_fortran`, `cppcheck` for `c` / `cpp` / `cuda_c`, two pieces of evidence of `fortitude` and `cppcheck` for `mixed`, `ruff` for `python`). It must not require execution via the `Makefile`'s `lint` target or via `compile_project`. The registration of the `build-runtime` MCP server including `run_linter`, and the tool-permission grant to the child `Agent` session, are verified by `preflight` as a workflow-start premise (on the Claude backend, both checks `claude_mcp_build_runtime_registered` ∧ `claude_mcp_build_runtime_permission_granted`; for details see [docs/RUNBOOK.md](../../RUNBOOK.md) §0-2).
- **fortitude C003 ↔ `-std=f2008` conflict (required workaround when generating Fortran source):** the fortitude lint rule **C003** requires the F2018 spec-list-bearing `implicit none (type, external)`, but under `impl_defaults.toolchain.standard=f2008` (Makefile `FFLAGS=-std=f2008`) this form becomes a `compile_error` with `Error: Fortran 2018: IMPLICIT NONE with spec list`. The two are a structural conflict. The workaround is to use plain `implicit none` (without spec list) and disable only C003. **The canonical means of disabling is the inline `! allow(C003)` directive in the generated source** (placed on the line immediately before `implicit none`). The method of placing `[check] ignore=["C003"]` in `src/<source_id>/src/fortitude.toml` is rejected by the source-extension restriction of the output manifest, so do not use it.
- **Fortran identifier ≤ 63 characters (f2008 name limit):** every identifier (subroutine / function / module / type / variable name) in the generated Fortran source must be **at most 63 characters**. Under the f2008 standard series an over-63-char name is a `compile_error` that only surfaces at the Build step, forcing an expensive regenerate → rebuild retry. `post_generate` (`validate_pipeline_semantics`) now flags any over-limit identifier in `source/<source_id>/src/*.f90` as a `Generate.verify fail`, so keep generated names within the limit from the start (shorten descriptive composite names rather than relying on the build to catch them).
- `Generate` must include in its official output a preset-compatible quality path with which `Validate.execute` can run using only the `preset` of `run_quality_checks`. On shortage, it is a `Generate fail`.
- With `impl_defaults.toolchain.build_system=make` and `impl_defaults.toolchain.language=fortran` / `c` / `cpp` / `mixed` families, a `test` or `check` target must be defined in `source/<source_id>/src/Makefile`. On absence, it is a `Generate fail`. The relevant target references the existing binary in `$(BINDIR)` with a **non-relinking** guard that fails closed when the binary is absent — `test -x $(BINDIR)/$(BIN) || { echo "error: $(BINDIR)/$(BIN) not built; run 'make all' first" >&2; exit 1; }` — and **must not** invoke `$(MAKE)` in the guard. Relinking inside `Validate.execute` writes into the read-only-bound `binary/` (outside the execute `runs/` write_root) and escalates a mere binary-name/availability mismatch into an `unauthorized_write_violation` → `fail_closed`; the fail-closed guard instead yields a clean retryable `make test` failure. For local use, run `make all test` (the `?=` in-source defaults are unchanged). The target generates the output destination with `mkdir -p $(RUNDIR)` (and the `raw/` required subdirectories the runner writes) **before** `cd $(RUNDIR)`, then runs and confines the run output under `$(RUNDIR)`. Because `$(RUNDIR)` can point to a not-yet-created run node dir, omitting `mkdir -p` makes `cd` fail.

### 2-2. Generate.verify substep
Before `Generate.verify` completes, run `python3 tools/validate_pipeline_semantics.py --stage post_generate --pipeline-root workspace/pipelines/<node_key_safe>/<pipeline_id>/`, and when fixing the `source_id` to verify, add `--source-id <source_id>`. `exit code 0` is required.

`Generate.verify` must not run the write-family tools of the `build-runtime` MCP including `run_linter` (`run_linter` / `compile_project` / `run_program` / `run_quality_checks`) (`static lint` is the responsibility of `Generate.generate` in 2-1). The external gates verify may launch are limited to the read-only `validate_workspace_root.py` and `validate_pipeline_semantics --stage post_generate`, and `lint_command_ref.run_linter` is limited to being the target of an *inspection* of an existing record. Because verify's `allowed_output_paths` does not authorize the side output of `run_linter` (`command_log.jsonl`), running `run_linter` from verify makes that write outside the manifest and invites an `unauthorized_write_violation` → `fail_closed`.

`Generate.verify` required verification items:

#### G1. case coverage
- All `case_id` of `spec.ir.yaml.case.test_case_set` and all expanded `case` are reachable from the implementation path of the `runner` or `model`.
- When an unimplemented `case`, an unreachable branch, or a fixed-`case_id`-limited implementation is detected, it is a `fail`.
- When an implementation in which the `model` composes the judgment metric only with `case_id` branching and fixed-numeric assignment is detected, it is a `fail`.

#### G2. case input acceptance
- The `runner` and `model` accept each runtime input of `spec.ir.yaml.case.test_case_set[].inputs`.
- When the input-propagation or recording path corresponding to at least `case_id`, the grid conditions, the time conditions, the initial-condition identifier, the boundary-condition identifier, the `profile`- or `component`-selection result, and the `test_profile_id` and `test_profile_version` derived from `tests.md` cannot be confirmed, it is a `fail`.

#### G3. implementation of case-dependent differences
- The per-allowed-selection-value difference implementation in `case.test_case_set[].inputs` is not collapsed into a fixed constant or a single default value.
- When an implementation that ignores case-dependent inputs such as `boundary`, `initial_profile`, `topography_profile`, `dt_rule`, `refinement`, or the `sweep` expansion result is detected, it is a `fail`.

#### G4. algorithm reflection
- Based on `spec.ir.yaml.algorithm.steps`, `ordering`, `control_condition`, and `iteration_contract`, the `test case set` coverage, operation composition, dependency `operation`, and the data dependency of the output metrics are reflected in the generated code.
- `algorithm.update_semantics`, `temporaries`, `derived_field_rules`, `invariants`, and `splitting_policy` are reflected in the generated code. When a missing state-update target, an unimplemented derived-quantity computation, an update order that breaks an invariant that should be conserved, or a `splitting_policy` mismatch is detected, it is a `fail`.
- The generated code does not introduce an additional operation, additional iteration, additional conditional branch, or additional dependency `operation` call not described in `algorithm`. When a guessed re-introduction of `controlled_spec.md`-derived information is detected, it is a `fail`.

#### G5. io_contract consistency
- When a constant output unrelated to the `model` output, a fixed-`JSON` output, or `diagnostics` generation by direct assignment of an analytic expression is detected, it is a `fail`.
- Verify that the final expression tree of an `intent(out)` variable reaches the group of output variables declared in `spec.ir.yaml.io_contract.semantic_dependency.required_sources` and `io_contract.outputs`.
- Reconcile `spec.ir.yaml.io_contract.raw_requirements.required_evidence` and `test_evidence_requirements` with the `runner`'s raw-evidence output design, and verify that the raw evidence needed for `Validate.judge` recomputation is held for every `test_id`. When a configuration in which the `runner` writes out only a whole-suite summary, a configuration that overwrites and aggregates the primary evidence of multiple `test` into one, or a missing `required_raw_variables` is detected, it is a `fail`.
- Verify that the `runner` emits, in `diagnostics.json`, a `checks.<id>` entry for every `io_contract.diagnostics_contract.checks[].id`, and — when `io_contract.diagnostics_contract.verdict.required=true` — a top-level `verdict` object holding the `verdict.fields` keys. When a `checks.*` key required by the contract is missing, or the required `verdict` object is absent, it is a `fail`.

#### G6. impl_defaults reflection
- `spec.ir.yaml.impl_defaults.target.class`, `target.backend`, `target.architecture`, `toolchain.language`, `toolchain.standard`, `toolchain.build_system`, and `selected.backend_key` are reflected in the generated source composition and the `build` artifacts.
- Verify that the execution-algorithm choices specified in `impl_defaults.abstract` and `backend_overrides` — parallelization, layout, fusion, tiling, vectorization, asynchrony, etc. — are reflected in the generated code or `build` settings to the extent expressible in the target language and target. When ignoring of a specified knob, the mixing-in of an optimization for a forbidden target, or a violation of the default `OpenMP` rule for `target.class=cpu` is detected, it is a `fail`.

#### G7. dependency consistency
- Verify that the generated code does not introduce a reference to a dependency `node` not present in `spec.ir.yaml.dependency` or to an undeclared `operation`. When a call outside `direct_deps`, an unresolved `component` reference, or an implementation choice contradicting a `profile` constraint is detected, it is a `fail`.

## On-failure behavior
- A `Generate fail` retry defaults to an in-phase retry.
- When `Generate.verify` judges that "the IR itself has an error", record "ir_inconsistency" in `source_meta.json.last_fail_reason`, and the `orchestration agent` decides whether to go back to `Compile`. When `Compile` is already checkpointed `pass`, the return to `Compile` first runs `reopen-phase --from-phase compile --node-key <node_key> --trigger-agent-run-id <generate_verify_fail_substep_agent_run_id> --reason <reason_code>` to invalidate the stale `Compile` checkpoint / `step_result` and reset its `phase_state`, then re-runs `Compile` → `Generate`. Canonical: `docs/CLI_REFERENCE_RARE.md#reopen-phase`, `docs/RUNBOOK.md` §3-1.
