# Workflow Orchestration

This document defines the orchestration contract — the **conductor** (`tools/workflow_conductor.py`) that drives the whole `workflow` — and the independent-agent execution conventions for phases / substeps. It presumes the 5-phase structure `Spec -> Compile -> Generate -> Build -> Validate`.

## Related documents

- CLI reference of all subcommands: [`docs/CLI_REFERENCE.md`](CLI_REFERENCE.md)
- workspace artifact placement: [`docs/WORKSPACE_LAYOUT.md`](WORKSPACE_LAYOUT.md)
- workflow operation / startup (operator procedure): [`docs/RUNBOOK.md`](RUNBOOK.md)
- launch request templates (rendered by the conductor's `record-launch`): [`tools/prompt_templates/`](../tools/prompt_templates/)

## Purpose
- Hierarchize the workflow execution and separate phase responsibilities from audit responsibilities.
- Execute each `step` / each `substep` as an independent agent, and make the execution path traceable.

## Scope
- `Compile` / `Generate` / `Build` / `Validate`
- Per-`node workflow` phase execution, and the execution of in-phase `substep` (`generate` / `lint` / `syntax` / `static` / `verify` / `pre_judge` / `execute` / `judge` / `post_judge`)

## term rules
- `phase` refers to the workflow's logical unit defined in the contract documents under `docs/workflow/WORKFLOW_CORE.md` and `docs/workflow/phases/`.
- `step` refers to the orchestration-level execution unit corresponding to one phase.
- `substep` refers to a lower execution unit decomposed from a `step`.
- `stage` is used only as existing field names such as `generated_by_stage`.
- The **conductor** (`tools/workflow_conductor.py`) is the orchestration actor: it drives the deterministic phase/substep loop, launches each `step` / `substep` as a leaf agent, and routes failures deterministically (escalating to a one-shot LLM diagnostician only on an unclassifiable failure). It is recorded in `agent_runs.jsonl` under an `orchestration_agent_run_id` with `agent_role=orchestration`.
- **Deterministic in-process execution.** The non-LLM substeps `Compile.static`, `Generate.lint`, `Generate.syntax`, `Generate.static`, `Build`, `Validate.pre_judge`, `Validate.execute` and `Validate.post_judge` are **always** run by the conductor **in-process** (no `claude -p` leaf, no `SKILL.md`) — it calls the build-runtime MCP handlers (`run_linter` / `run_syntax_check` / `compile_project` / `run_program` / `run_quality_checks`) or the static validators directly. `Compile.static` runs `validate_workspace_root.py` + `check_artifact_syntax.py` (on `spec.ir.yaml` + `ir_meta.json`) + `validate_pipeline_semantics --stage compile` (writing `compile_static_meta.json` under the IR dir) — the purely-static IR gates the verify leaf used to own — between `Compile.generate` and `Compile.verify`, and on a violation warm-resumes `Compile.generate` (a same-phase reopen), so the LLM `Compile.verify` is reached only on a deterministically-clean IR and launches no validator gate (it is a pure spec-cross-reference semantic pass). `Generate.lint` runs `run_linter` over `source/<source_id>/src/` between `Generate.generate` and `Generate.syntax`, writes a host-authored leaf-non-writable `lint_evidence/<source_id>.json` (certified by `post_generate`), and on a lint finding warm-resumes `Generate.generate` (a same-phase reopen). `Generate.syntax` then runs the `run_syntax_check` compiler front-end gate (mandatory gfortran `-fsyntax-only`, plus optional `METDSL_SYNTAX_COMPILERS` target-compiler stages) over the node + staged dependency-closure sources between `Generate.lint` and `Generate.static`, writes `syntax_meta.json` and a host-authored leaf-non-writable `syntax_evidence/<source_id>.json` (certified by `post_generate`), and on a compiler finding in the node's own source likewise warm-resumes `Generate.generate`. A failing stage is first attributed by re-running the same compiler over two isolated source sets, never by reading the diagnostics text (a clean file still prints default-on warnings naming itself): a **canary** valid under every standard (if it fails, the invocation itself is unviable — typically an `impl_defaults.toolchain.standard` the compiler driver does not accept, which the leaf does not author), then the **staged dependency closure alone** (if it fails, the closure is at fault — either a dependency's certified source is defective, or this node's declared standard is narrower than the sound closure needs; the message names both causes and attaches the diagnostics, which say which). Each attribution terminalizes `fail_closed`, because the leaf authors neither the IR nor a dependency's certified source and no retry of this node could clear either. Every other compiler finding is the node's own and warm-resumes `Generate.generate`. `Generate.static` then runs `validate_workspace_root.py` + `validate_pipeline_semantics --stage post_generate` (writing `static_meta.json`) — the purely-static gates the verify leaf used to own — between `Generate.syntax` and `Generate.verify`, and on a violation likewise warm-resumes `Generate.generate`, so the LLM `Generate.verify` is reached only on a deterministically-clean source and launches no validator gate. The in-process bodies implement the in-source **Make** layout only (`build_system=make`; OBJDIR/BINDIR overrides, `make_test`); a non-Make `build_system` (cmake/meson/ninja) is rejected with a clear error rather than misplaced. A content failure (lint findings, compile/link error, post_build/post_execute violation, quality-check mismatch, runtime error) returns cleanly and routes via the Lint/Build/Validate decision tables (→ Generate etc.), exactly like the leaf era; a genuine conductor/MCP infra failure fail-closes as a transport error, as do the two `Generate.syntax` failures its attribution runs single out (an unviable compiler invocation; a defective staged dependency closure), neither of which any retry of this node could clear. They are the only substeps that are not leaf agents. The conductor still issues the `record-launch` → `record-child-return` → `finalize-child` bookkeeping so the executor is a normal `step` / `substep` `agent_run_id` with a capability and phase_state, but their launch prompt is a **minimal deterministic stub** (sentinel `Conductor-executed deterministic step` + core ids; no skill section), and the validator applies a reduced launch-prompt-marker set to it. `Validate.judge` is always a leaf (its independent LLM semantic check is essential), but the purely-structural `--stage pre_judge` gate it used to run is **hoisted to the conductor as two deterministic substeps wrapping the judge** (G4, mirroring `Compile.static` / `Generate.static`). The `Validate.pre_judge` substep (index 0) runs a **dependency-DAG readiness** check before execute/judge (a `dependency_graph.json` sidecar `all_nodes` closure node not built+validated in its own pipeline authors `pre_judge_meta.json` `status=fail` and terminalizes `fail_closed`, reusing `validate_pipeline_semantics._closure_node_validated_in_own_pipeline`). The `Validate.post_judge` substep (index 3) runs the `--stage pre_judge` validator after the judge returns (scoped `--pipeline-root`/`--run-id`, with BOTH the post_judge substep's own arid AND the judge arid declared `--in-flight-agent-run-id`) and records the verdict + a severity `disposition` in `post_judge_meta.json` under the run-node dir. (Naming caution: the `post_judge` substep runs the validator stage literally named `pre_judge`.) `determine_substep_status` gates the post_judge substep on that verdict; a **recoverable** violation (a judge-authored deliverable) warm-resumes the judge in place to re-author it; an orchestration-record/DAG **integrity** violation (or any `pre_judge` `fail`) is a **non-physics integrity blocker** the conductor terminalizes `fail_closed` (writing no routeable `fail` `step_result`, which the judge `pre_phase_complete` hook forbids atop a `pass` `semantic_review`); an **unknown** violation routes to the unified escalate LLM in prod, and terminalizes `fail_closed` in dev (G5). The judge leaf itself launches no validator gate (`ALLOWED_VALIDATE_PIPELINE_STAGES` is empty for all three validate substeps). See [`docs/workflow/phases/phase_03_build.md`](workflow/phases/phase_03_build.md) and [`phase_04_validate.md`](workflow/phases/phase_04_validate.md).

## phase / substep types

| phase | step type | substep |
|-------|-----------|---------|
| Compile | has substeps | `generate` / `static` / `verify` |
| Generate | has substeps | `generate` / `lint` (conductor-run deterministic) / `syntax` (conductor-run deterministic) / `static` (conductor-run deterministic) / `verify` |
| Build | single step | - |
| Validate | has substeps | `pre_judge` / `execute` / `judge` / `post_judge` |

## Requirements

### preflight and launch control
- Workflow execution runs exactly one conductor per `node` (started by `tools/run_workflow.py`).
- Before the workflow starts, the preflight of an execution platform that can launch the `step agent` and `substep agent` independently must be run. The preflight includes the `multi_agent` feature and the launchability of a child `agent` in its verification scope, and when it is not `pass` the workflow must not start.
- The preflight of `backend=codex` must simultaneously satisfy `feature_states.hooks=true`, `checks.hooks_enabled.pass=true`, and `checks.codex_home_writable.pass=true`.
- A pre-existing `preflight.json` that contains `feature_states.codex_hooks=true` and `checks.codex_hooks_enabled.pass=true` is accepted only as a read-time compatibility input. A newly generated `preflight.json` must use `feature_states.hooks` and `checks.hooks_enabled`.
- The preflight must include `sandbox_runtime=bwrap` and `sandbox_enforced=true` as required conditions. When at least 1 of `checks.sandbox_bwrap_available.pass=true`, `checks.sandbox_bwrap_userns.pass=true`, or `checks.sandbox_bwrap_exec.pass=true` is not satisfied, the workflow must not start.
- The `hooks` feature decision is probed **host-side by the conductor** (`Conductor._ensure_codex_feature_cache`) once per `orchestration_id` and written to `workspace/orchestrations/<orchestration_id>/codex_feature_check.json` (the orchestration-dir ROOT, which is read-only inside the bwrap sandbox — NOT the leaf-writable `hooks/` dir, so a confined leaf cannot forge it). The in-sandbox native hook reads this cache read-only and fail-closes when it is missing/invalid.
- Making `preflight.json` `pass` by manual editing is forbidden.
- Just before launching a child `agent`, `multi_agent` and the launchability of the child `agent` must be re-checked by a live probe of the execution platform. On `fail`, `record-launch` and the child-`agent` launch are forbidden, and the workflow transitions to `fail`.
- Before starting each phase, run `workflow-launch-check`, and simultaneously check the required child-`agent`-type decision, execution-platform allowability, session-policy allowability, and dependency readiness.

### phase type and agent type
- Before starting each phase, explicitly judge by phase type whether the target phase requires a `step agent` or a `substep agent`. `Compile` / `Generate` / `Validate` require a `substep agent`, and `Build` requires a `step agent`.
- Phase-artifact generation, MCP execution, and verification-purpose provisional implementation are performed only by a launched child `agent`, never by the conductor before the child leaf launch completes.
- The phase-artifact roots of `workspace/ir/` and `workspace/pipelines/` can be materialized only by a child `agent` that satisfies the 3 conditions of `record-launch`, a capability token, and `phase_state=child_running`. Direct generation by the conductor is forbidden.
- When a root-path reservation is needed before launching a child `agent`, generate only the reservation artifact `workspace/orchestrations/<orchestration_id>/reservations/<node_key_safe>/<step>.json`, and the actual directories of `workspace/ir/` and `workspace/pipelines/` must not be created.
- The conductor is responsible only for the progress control of the whole workflow, and must not directly generate the phase-body artifacts (e.g. `spec.ir.yaml`, `diagnostics.json`).
- As a substitute for workflow execution, a script that batch-automates the progress of multiple phases and artifact generation must not be newly generated or executed.
- The `Build` step is a deterministic process that calls the MCP `compile_project` and requires no LLM inference. The `step agent` limits its responsibility to the MCP call and recording the result.
- The conductor directly launches each substep of `Compile` / `Generate` / `Validate` as a leaf (`record-launch` + a `claude -p` / `codex exec` subprocess).

### capability / write_root
- The changes to phase artifacts permitted to a child `agent` must be limited to under the `write_root` permitted by the capability token.
  - `Compile.generate` / `Compile.static`: `workspace/ir/<node_key_safe>/<ir_id>/` (`Compile.static` writes `compile_static_meta.json`)
  - `Compile.verify`: **narrowed to the single file pin** `workspace/ir/<node_key_safe>/<ir_id>/ir_meta.json` — the verifier authors only `verification_status`, so the certified `spec.ir.yaml` (and the rest of the IR dir) is not RW-bound to it and cannot be mutated post-gate.
  - `Generate.generate`: `workspace/pipelines/<node_key_safe>/<pipeline_id>/source/<source_id>/` — **agentic (residual `codex` / non-`M3c`) generate leaves only**. On a pure-leaf node this substep writes nothing: its `write_roots` is `[]` and the host authors the deliverables after the leaf terminates (§pure-function leaf (`Z2` generate-executor)).
  - `Generate.verify`: **narrowed to the single file pin** `workspace/pipelines/<node_key_safe>/<pipeline_id>/source/<source_id>/source_meta.json` (agentic leaves only; a pure-leaf node writes `[]`) — the verifier re-authors only that meta and never rewrites the producer sources, which the deterministic `lint` / `syntax` / `static` gates already certified.
  - `Build`: `workspace/pipelines/<node_key_safe>/<pipeline_id>/binary/<binary_id>/`
  - `Validate.pre_judge` / `Validate.execute` / `Validate.post_judge`: `workspace/pipelines/<node_key_safe>/<pipeline_id>/runs/<run_id>/<node_key_safe>/` (the two gate substeps author `pre_judge_meta.json` / `post_judge_meta.json` there)
  - `Validate.judge`: **narrowed to the single file pin** `workspace/pipelines/<node_key_safe>/<pipeline_id>/runs/<run_id>/<node_key_safe>/semantic_review.json` — the judge is a pure semantic pass authoring only that file, so the same-dir host-authored `verdict.json` (the R2 verdict) / `diagnostics.json` are not RW-bound to it.
  - The three narrowed file pins above (`compile.verify` / `generate.verify` / `validate.judge`) are pre-created by `build_bwrap_profile` and `--bind`-mounted at file granularity, so `bwrap` and the terminal FS-diff enforce the same per-substep scope as `allowed_output_paths` — a structural second layer under the pattern-based file-tool hook. A missing/malformed `source_id` / `run_id` collapses the pin to `[]`, which the empty-`write_roots` invariant rejects at launch.
- Changes under `ir_ref` / `pipeline_ref` are limited to a direct `Edit` / `Write` to a path enumerated in `allowed_file_tool_paths` of `output_manifests/<agent_run_id>.json` — uniform across extensions (managed `.json` / `.txt` included; under `bwrap` confinement the write is authorized by `write_roots` containment, see "file write paths" below).
- `record-launch` must generate `workspace/orchestrations/<orchestration_id>/output_manifests/<agent_run_id>.json` per child `agent_run_id`, and finalize `allowed_output_paths`, `allowed_file_tool_paths`, and `allowed_tmp_root` (`workspace/tmp/<agent_run_id>`). **Exception — a pure launch (§pure-function leaf (`Z2` generate-executor)) writes no output manifest at all**: a no-write leaf has nothing to manifest, and the manifest's *absence* is itself a validated tripwire (its presence on a pure arid is a sweep violation, since it would mean the pure launch took the write-bearing branch).
- **Required file-pin auto-inject for Make builds**: when the step is `Generate` and `spec.ir.yaml.impl_defaults.toolchain.build_system=make` **and the Makefile is not conductor-authored** — i.e. a non-fortran make family (c/cpp/mixed) — `record-launch` auto-injects the in-source `Makefile` (`<pipeline_ref>/source/<source_id>/src/Makefile`) into `allowed_output_paths` and flows it into `allowed_file_tool_paths`. The conductor lists each source file explicitly (so it auto-derives into `allowed_file_tool_paths` and is written with `Edit` / `Write`), but the extension-less `Makefile` is intentionally excluded from the source-extension set of the directory allowlist (`tools/hooks/common.py`), so without the explicit pin it cannot be written via any path and the child fail-stops mid-run to avoid guessing. The conductor usually just omits `allowed_file_tool_paths` (auto-derive). For a `make` ∧ `fortran` node — **leaf OR with dependencies** — the conductor authors `src/Makefile` host-side (`_resolved_makefile_host_authored` mirrors `Conductor._conductor_authors_makefile`); the pin is then **suppressed** and the file dropped from the child's `allowed_output_paths` (the child must not write it). For a dependency node the authored Makefile also carries the closure object rules, with each `<dep>_model.f90` staged into `$(OBJDIR)` at Build (Model B; see `docs/design/deterministic_followups.md`).
- **launch-time provisioning verification**: when, in a Make-build Generate launch, the required `Makefile` pin is missing from the finalized `allowed_file_tool_paths` (e.g. the caller passed an explicit `allowed_file_tool_paths` and missed the pin), `record-launch` **fail-fasts with a `ValueError` before launching the child**. This converts an artifact-contaminating mid-run fail-stop into a cheap, recoverable launch-time error.
- `record-launch` must generate `workspace/orchestrations/<orchestration_id>/read_manifests/<agent_run_id>.json` per child `agent_run_id`, and finalize `allowed_read_roots` and `denied_read_roots`.
- `record-launch` must generate `workspace/orchestrations/<orchestration_id>/sandbox_profiles/<agent_run_id>.json` per child `agent_run_id`, and finalize the `read_roots`, `write_roots`, and runtime bind composition needed for `bwrap` execution. The child launch allows only `bwrap` execution using that profile.

### file write paths
- When a `step agent` / `substep agent` changes a phase artifact, the canonical invocation — uniform across extensions (managed `.json` / `.txt`, `.yaml` / `.yml` / `.md`, source code) — is a direct `Edit` / `Write` to a path enumerated in `output_manifests/<agent_run_id>.json.allowed_file_tool_paths`. Under mandatory `bwrap` confinement the write is authorized by `write_roots`-containment of the FS-diff at terminalization, so no gate step is involved (`guarded-apply-patch` and the `apply_patch_writes` gate have been removed — see §"Patch application contract (removed in P2-7)"). The MCP-owned `command_log.jsonl` is excluded from `allowed_file_tool_paths` and is written only by the MCP server.
- `spec.ir.yaml` is in `.yaml` format, so write it with `Edit` / `Write`.
- The direct execution of normal `apply_patch`, shell redirection, `tee`, `sed -i`, `perl -0pi`, and file writes via `python` / `sh` / `bash` are forbidden.
- A file write via shell, regardless of whether the target path is a phase artifact, must be forbidden unless it is included in the canonical invocation explicitly stated in the child-`agent` launch request.

### LLM context
- The `step agent` and `substep agent` must not share the same `LLM` context. Each `agent_run_id` has a unique `context_id` and requires recording `context_isolated=true`.
- `Compile.generate` and `Compile.verify` are launched in independent contexts. The same applies to `Generate.generate` and `Generate.verify`, and `Validate.execute` and `Validate.judge`.

### agent_run recording
- For a phase that has `substep`, the conductor must launch the required `substep` group, make the completion judgment, and then finalize `step_result.json`.
- The conductor must judge the launchability of a `step agent` or `substep agent` based on the dependencies reconstructed from `deps.yaml` and `spec_catalog.yaml` and the `dependency` section of `spec.ir.yaml`.
- Every `agent` execution has an `agent_run_id` and must record the input references, output references, and parent-child relationship.
- Each line of `agent_runs.jsonl` requires recording `started_at` and `status`, and when `status` is a terminal status (`pass` / `fail` / `blocked` / `timeout` / `cancel`), requires recording `finished_at`.
- `fail_closed` is used only as a terminal status of `orchestration_meta.status`.
- The `agent_runs.jsonl` of the `step` / `substep` roles requires recording `parent_agent_run_id`, `agent_backend`, `agent_model`, `context_id`, `context_isolated`, `agent_session_id`, `launch_request_ref`, `launch_response_ref`, `launch_prompt_ref`, `launch_reply_ref`, `agent_result_ref`, and `agent_summary_ref`.
- The `parent_agent_run_id` of a `substep agent` points to the `orchestration agent_run_id` that launched that `substep`.
- The child leaf's `agent_run_id` is recorded as its `agent_session_id`.
- **`agent_session_id` is a bookkeeping id; the claude backend ALSO pins a real session to it.** The conductor reuses the child's `agent_run_id` (UUID) as its `agent_session_id` / `context_id`. For orchestration bookkeeping these are identifiers only, and the **canonical** record of a child's work is `workspace/orchestrations/<orchestration_id>/agents/<agent_run_id>/dialogs/` (the launch prompt/request, and the final reply/summary). However, for the **claude backend** the leaf is launched with `--session-id <agent_run_id>`, so Claude Code **does** write a real transcript at `~/.claude/projects/.../<agent_run_id>.jsonl`. That host transcript is what `repair_strategy=reuse` warm-`--resume`s (see the reuse-repair rule below); it is NOT mirrored into the orchestration artifacts, so a warm resume leaves no trace in `agent_runs.jsonl` (which still records `context_isolated: true`) — to tell a warm resume from a cold start, inspect the host `~/.claude` transcript, not the orchestration dir.
- `record-launch` must update `workspace/orchestrations/<orchestration_id>/session_run_index.json` immediately after launching the child, and record `agent_run_id`, `agent_session_id`, `context_id`, `agent_role`, and `status`.
- The `edge` of `agent_graph.json` uses `orchestration -> step` or `orchestration -> substep` as the canonical source.
- `agent_runs.jsonl` and `agent_graph.json` must be generated by sequentially appending in-flight events. Post-generation is forbidden.
- The `agent_runs.jsonl` row of the conductor itself is appended **only once, immediately after launch** with `agent_role=orchestration`, `status=running`. The conductor itself has no path to update this row via `record-agent-run` (a double invoke is rejected with `ValueError: duplicate agent_run_id`). At termination, `set-status` rewrites this row in-place to a terminal status with runtime privilege and assigns `finished_at` (it is a rewrite, not an append, so it does not hit the duplicate guard), and on resume (terminal reset) it conversely returns it to `running`. This is to prevent the agent_runs-based audit / `validate_workspace_root` from mistaking the orchestration row as permanently `running`, and the canonical terminal state of the whole orchestration continues to be expressed on the `orchestration_meta.json` side via `set-status`. For details, [docs/CLI_REFERENCE.md#record-agent-run](CLI_REFERENCE.md#record-agent-run) and [docs/CLI_REFERENCE.md#set-status](CLI_REFERENCE.md#set-status) are the canonical source.
- `record-launch` saves the launch request/response and issues the capability/manifests; it runs as part of the leaf launch.
- `record-launch` must record `sandbox_runtime=bwrap`, `sandbox_enforced=true`, and `sandbox_profile_ref` in the launch response.

### child agent launch request
- When launching a child `agent`, the conductor must, using `docs/workflow/WORKFLOW_CORE.md` and the `docs/workflow/phases/phase_*.md` corresponding to the target `step` as the canonical source, make explicit the `execution input`, `verification input`, and `expected output` of the target `step` or `substep`.
- The conductor must make explicit, in the child-`agent` launch request, that the canonical source for the requirement definition and judgment rules is `docs/`, `spec/`, and the relevant trial's artifacts. An instruction or implication to read the implementation under `tools/` and extract rules is forbidden.
- The prohibition on reading the implementation under `tools/` applies to workflow execution and child-`agent` rule derivation. Repository improvement, maintenance, testing, and refactoring may inspect `tools/*.py` directly.
- The validator invocation by the child `agent` defaults to `run-gate`, and the canonical invocation is `python3 tools/orchestration_runtime.py run-gate --gate <gate_name> --agent-run-id <agent_run_id> --capability-token <capability_token> --args-json '<json>'`.
- The direct execution of a validator script is permitted only as an exceptional operation. The permitted targets are limited to `validate_workspace_root.py` and `check_artifact_syntax.py`.
- `guarded-apply-patch` and the `apply_patch_writes` gate have been **removed** (P2-7); a child writes every artifact directly with `Edit` / `Write`, authorized by `write_roots`-containment of the FS-diff at terminalization.
- `record-agent-run` authorizes a step/substep child's writes by `write_roots`-containment of the actually-changed paths (the diff against the baseline): a change that landed inside the capability token's `write_roots` is the child's own confined output and is authorized; a change outside `write_roots` is rejected as an `unauthorized write`. (Under mandatory `bwrap` a write outside `write_roots` is impossible by construction.) The canonical MCP audit logs are handled by their own placement check. The conductor's host writes of the `generate.lint` certificate `<pipeline_root>/lint_evidence/<source_id>.json` and the `generate.syntax` certificate `<pipeline_root>/syntax_evidence/<source_id>.json` — made in-process during their substep windows and deliberately at the pipeline root (non-forgeable, outside the substep's `source/` write_root, like `lineage.json`) — are exempted, scoped to step==`generate` ∧ substep==`lint`/`syntax` respectively so the sandboxed `Generate.generate` leaf is never exempted. Each exemption is bound to that EXACT certificate filename (the `<source_id>` from the host-authored launch request), not the whole evidence directory, so a stray/unexpected sibling there is still flagged. No `apply_patch_writes` gate record is consulted for step/substep.
- **runtime placeholder restoration: removed (P2-7).** The `created_file_pin_stubs` 0-byte placeholder restoration `record-agent-run` previously performed is gone, together with the file-pin *stub* system. It remains unnecessary even though the `compile.verify` / `generate.verify` / `validate.judge` substeps now declare a single file-pin `write_root` (`ir_meta.json` / `source_meta.json` / `semantic_review.json`): each pin is a managed artifact the owning leaf writes **in place** — `ir_meta.json` / `source_meta.json` are producer-authored and already exist; `semantic_review.json` is pre-touched (0-byte) by `build_bwrap_profile` and every consumer fail-closes on empty/invalid JSON — and a bind-mounted file cannot be deleted or renamed-over from inside the sandbox (EBUSY), so there is no runtime-owned placeholder a leaf could collaterally delete. `lineage.json` stays host-authored at the non-writable pipeline root as before.
- The child-`agent` launch prompt must be rendered from the corresponding template in `tools/prompt_templates/`; a free-form prompt that does not use the template is forbidden. The conductor supplies the launch-request parameters; `record-launch` renders the prompt (the canonical render path) and returns the rendered `launch_prompt_text`, which the conductor passes to the leaf subprocess.
- `ir_ref`, `pipeline_ref`, and `dependency_ref` must be finalized to canonical paths before launching the child `agent`. A placeholder must not be recorded in the launch request.

### repair / retry
- The conductor must evaluate the child `agent`'s returned result and judge `issue_severity` (`minor` / `major` / `critical`). A verify finding is never tolerated.
- The conductor routes a verify finding by `issue_severity` (`classify_verify_severity`): `minor` → `repair_strategy=reuse` **same-phase reopen** of the phase's producer substep (warm-resume + slim finding injection); `major` / `critical` → `dev`: `fail_closed`, `prod`: `escalate` to the diagnostician (which decides how far back to recover — `reuse`/`restart`/`reopen`/`fail_closed`, INCLUDING a `restart` of **this phase's own producer**, e.g. regenerate the IR from scratch). Any decision that targets the current phase with a `repair_strategy` (`reuse`→warm / `restart`→cold) re-runs that phase's producer (rotating its id); a same-phase decision with no `repair_strategy` terminalizes. For non-verify failures it chooses `repair_strategy` per the relevant decision table.
- The conductor must not directly perform the repair of a phase artifact itself. When a repair is needed, it must re-delegate to the child `agent` of the target `step` or `substep`.
- `repair_strategy=reuse` may be chosen only when it can converge with a local fix without changing the input contract and expected output of the target `step` or `substep`.
- `repair_strategy=restart` is chosen when any of contract reinterpretation, design reconstruction, or wide-scope regeneration is needed.
- On re-submission, regardless of `repair_strategy`, a new `agent_run_id` and a new `context_id` are issued.
- With `repair_strategy=reuse`, the `agent_session_id` may be reused. With `repair_strategy=restart`, a new `agent_session_id` must be issued.
- **Warm-resume reuse (claude backend).** For the claude backend, a `reuse` repair whose target session transcript still exists is **always** warm-`--resume`d (forked) so the repair leaf inherits the producer's context instead of cold-starting, falling back to a cold launch if the transcript was GC'd. `restart` repairs stay cold (anchoring avoidance) — so the warm/cold choice is driven by the failure classification's `repair_strategy` (deterministic-gate findings route `reuse`→warm; LLM-verify-attributed `restart`→cold). When a warm resume fires AND a findings excerpt is present (the `Generate.lint`/`Generate.syntax`/`Generate.static`/`Compile.static` reopens), the conductor sends a **slim findings-only repair turn** (the resumed leaf already holds the SKILL/contract docs); a warm reuse without findings re-sends the full prompt. Both behaviours are always-on (no env gate). Canonical: [docs/design/deterministic_followups.md](design/deterministic_followups.md) "G1-slim".
- The `launches/<agent_run_id>.request.json` on re-submission requires recording `issue_severity`, `repair_strategy`, `repair_target_agent_run_id`, and `repair_reason`.
- With `repair_strategy=reuse`, a step/substep `pass` no longer requires `apply_patch_writes` gate evidence (`_validate_apply_patch_gate_coverage` early-returns for step/substep), so there is nothing to inherit: a reuse retry that writes nothing terminates `pass` on its own, and a reuse retry that re-writes a path does so directly with `Edit` / `Write`. (Same-identity `(node_key, step, substep)` is still verified runtime-side.) With `repair_strategy=restart`, the contract is reinterpreted and output is regenerated from scratch.
- **verify-family meta re-author (still required for a different reason):** a `compile.verify` / `generate.verify` substep must produce its declared meta output (`ir_meta.json` / `source_meta.json`) to terminate `pass` — an inspect-only verify that writes nothing cannot `pass` (it has no `output_refs`). When the inspection finds nothing to change, re-author the meta anyway (refresh an idempotent field such as `verify_attempts`) with `Edit` / `Write`. The verify launch prompt MUST instruct the meta re-author as mandatory (not optional) — see the verify-family line of the `substep agent` template in [tools/prompt_templates/substep_agent.txt](../tools/prompt_templates/substep_agent.txt). **This bullet is the agentic (residual) generate-leaf contract; a `pure` verify leaf inverts it — see §pure-function leaf (`Z2` generate-executor).**

### pure-function leaf (`Z2` generate-executor)
- Under the `pure` `generate-executor` — the only executor since `M-F` (`docs/GLOSSARY.md`) — the `(generate, generate)` and `(generate, verify)` substeps run as pure-function leaves on a `claude` `M3c` node: a `claude -p` turn with tools disabled that returns exactly one JSON document and writes nothing. (A residual `codex` / non-`M3c` node runs the shared agentic leaves instead.) This is a distinct executor class from both a full agentic leaf and a deterministic in-process substep (the "Deterministic in-process execution" rule of §term rules above): the executor is a normal `substep` `agent_run_id` with a capability and phase_state, but its capability is `write_roots == []` (`mode: "pure_readonly"`) and its launch prompt reads no `SKILL` and no `skill_must_read_refs`.
- **Deliverable vouch by on-disk existence.** A pure leaf holds no write authority, so it cannot itself produce an `output_refs` deliverable. Its terminal `agent_run` row therefore carries `output_refs == []` — enforced **in both directions**: a pure terminal `pass` row with a non-empty `output_refs` is a violation (a no-write leaf must not claim a write). The phase's declared required outputs (`phase_required_outputs`: `<source_id>/src/<spec_id>_model.f90`, `<spec_id>_checks.f90` on an `M3c` node, and `<source_id>/source_meta.json`) are written by the **host** after the leaf terminates and are vouched by their **on-disk existence** at pass-evaluation time (a declared-but-unwritten deliverable still fails — the anti-mock-green invariant holds). `codegen_bundle.json` is host-written but is **not** a required output: it is certified separately by the `post_generate` gate (`_validate_post_generate_bundle`), which re-runs the full bundle acceptance contract against it. This inverts the agentic verify-family meta re-author bullet above: an agentic verify leaf re-authors its own meta, the `pure` verify leaf returns a verdict the host projects onto `source_meta.json`.
- **Verdict-category failure routing → cold generate restart.** A pure leaf's bounded in-conversation repair (schema-malformed bundle / verdict) is capped; on budget exhaustion the failure category, excerpt, and per-attempt model/usage are persisted (`bundle_meta.json` for the producer, `verdict_meta.json` for the reviewer) and no `source_meta.json` / verdict projection is written (proof-of-work: no schema-valid verdict, no meta). A pure `generate.verify` verdict category does **not** route `reuse` to the verify session: a cross-phase reopen structurally returns to the `generate.generate` producer, and the exhausted reviewer session has no reuse value, so it routes `restart` → **cold `generate.generate`**. (A `minor` verify finding on a schema-valid verdict still routes `reuse` → same-phase producer reopen per the `### repair / retry` table; only the schema-failure categories force the cold restart.)
- **Persona separation.** A pure `generate.verify` leaf never resumes the producer's session (the separate-persona hard rule). The guarantee is **structural, not a check**: `_run_pure_verify_substep` accepts no external resume seed at all — the only session it ever resumes is its own prior attempt's `agent_run_id`, assigned inside its repair loop — so a producer arid is unreachable rather than rejected. (The pure producer, by contrast, does accept an external reopen seed, which is why the two are not symmetric.) The producer and reviewer are always launched under fresh, distinct `--session-id`s.

### leaf transient retry
- A leaf that exits nonzero because of an LLM-infrastructure fault is classified from its captured output into an infra tag (`llm_usage_limit` / `llm_overloaded` / `llm_rate_limit` / `llm_client_error` / `llm_permission_probe_unavailable` / `llm_transport_flake`). `stderr` is authoritative and the most severe tag in it wins; `stdout` — which carries the leaf's own prose — decides only when `stderr` names nothing. The exception is the two **non-retryable** tags `llm_usage_limit` and `llm_client_error`: found on `stdout` (where the CLI reports both, as its result text) they override a `stderr` tag, which can only ever remove a re-launch, never add one. The conductor retries a **transient** tag in place instead of terminalizing the run.
- Retryable tags: `llm_transport_flake`, `llm_overloaded`, `llm_rate_limit`. Budget: at most 2 retries (3 launches of the same `substep`). Backoff before each re-launch is per tag (`llm_transport_flake`: 2s, 10s; `llm_overloaded`: 15s, 60s; `llm_rate_limit`: 30s, 90s).
- Non-retryable, terminal on the first occurrence: `llm_usage_limit` (a hard stop of hours; an immediate re-launch consumes budget and delays the operator's `--resume`), `llm_client_error` (a 4xx: the request itself is rejected, so every re-launch reproduces it byte-for-byte), `llm_permission_probe_unavailable`, and an **unclassifiable** nonzero exit (a crash, an out-of-memory kill, or a hook denial is deterministic; re-running it reproduces the same failure).
- **Opt-in `--wait-usage-reset` (default OFF)**: `llm_usage_limit` is still terminal by default, but with the flag set AND a **machine-form reset epoch on STDERR** (a trailing `|<unix-epoch>` on the dead leaf's usage-limit line; a human-worded reset carries none and is never guessed at), the conductor instead **waits it out in place** and re-launches the same `substep` — a same-run, substep-granular resume instead of a next-day manual `--resume`. The epoch is read from **stderr only**: stdout is the leaf's own output surface (the model's result text / the pure JSON envelope), and although the classifier may *promote* an `llm_usage_limit` tag out of stdout, that promotion was documented-safe only because the tag was terminal (it could only ever REMOVE a re-launch — see `_CROSS_STREAM_PROMOTING_TAGS`). The wait ADDS a multi-hour sleep + relaunch, so arming it from untrusted stdout would let a leaf that crashed for an unrelated reason but whose prose contains `usage limit reached|<epoch>` park a run for hours; a usage limit seen only on stdout therefore still `fail_closed`s (labelled `llm_usage_limit`) but declines the wait. It emits a `leaf_usage_limit_wait` event (`reset_epoch` / `wait_seconds` / `wait_attempt` / `dead_agent_run_id`), tombstones the dead attempt under the reason prefix `leaf_usage_limit_wait_orphan:` (same finalize-then-tombstone-outside-the-window ordering as a transient retry), then sleeps to the reset + a 120s margin. The wait budget is **one per `substep`, independent of the transient-retry budget** (a usage limit is never a transient tag, so the two cannot compound), and a reset further out than 6h is treated as a weekly limit / misparse and falls back to `fail_closed`. The re-launch's `record-launch` re-runs the TTL-driven preflight live-probe, so a stale preflight self-heals; the residual is that waking before the reset actually lands makes that probe fail and the phase `fail_closed` (the 120s margin exists to avoid it). The flag is **not** auto-recovered on `--resume` (a per-invocation preference — re-pass it); it is recorded on `orchestration_meta.json#invocation` for provenance and **refreshed on resume** to the effective re-passed value so the record never reports the opposite of the resumed run's behavior. Wired into all three leaf loops (`run_substep`, the pure producer, the pure reviewer); a `substep`-terminal usage limit after the wait is spent reports the honest launch count in `[attempts=<n>]` exactly as a transient exhaustion does.
- The retry loop is closed inside the `substep` execution: the phase's outcome list stays one entry per `substep` of `SUBSTEPS[phase]`, and a retried `substep` contributes exactly one outcome (its surviving attempt).
- Each attempt is a new `agent_run_id` with its own `record-launch`, its own `launches/<agent_run_id>.request.json`, its own `agents/<agent_run_id>/dialogs/` (the dead attempt's leaf output is retained as failure evidence), and its own launch instant (the freshness bound of `determine_substep_status`, so an artifact left half-written by a dead attempt cannot satisfy the retry).
- The warm-resume target is invariant across attempts: a retry re-forks the same producer session and re-sends the same slim findings-only repair turn. A retry must not fall back to a cold launch (the slim prompt, hence the findings excerpt, is conditional on warm resume).
- A dead attempt is `finalize-child`ed and then tombstoned with `add-superseded-runs` (reason prefix `leaf_transient_retry_orphan:`), both **before** the next `record-launch`: it is never vouched by a `step_result.json` (`substep_agent_run_ids` records only the surviving attempt), and an un-vouched terminal `agent_run_id` is rejected by `_validate_orchestration_completion_for_pass`. The order is load-bearing in both directions. `finalize-child` must come **first** because it closes the child's write window: `record-agent-run` diffs the live workspace against the baseline `record-launch` took, and `<orch_root>/reopen/` — where the tombstone writes — is not runtime-ignored, so tombstoning inside the window would charge the conductor's own writes to the dying leaf as an unauthorized write and the retry would never launch. It must also **precede the next `record-launch`** (the runtime rejects a launch while a child of the same parent is active). The residual is a host crash between the two writes, which leaves a terminal, un-vouched, un-tombstoned `agent_run_id` that a `--resume` cannot repair on its own (recovery: a manual `add-superseded-runs`); this is the same window the other tombstone call sites carry.
- A transient retry is an **infra** retry, not a content retry: `retry_decisions` stays `null`, no `issue_severity` / `repair_strategy` is assigned, and a recovered `phase` writes its `step_result.json` exactly as an unretried one does.
- Deterministic in-process `substep`s (`Build`, `Validate.{pre_judge,execute,post_judge}`, `Generate.{lint,syntax,static}`, `Compile.static`) are out of scope: they spawn no leaf, so they have no transport to fail.
- On budget exhaustion the phase terminalizes exactly as before: `fail_closed` with reason `leaf_transport_error: leaf_exit=<rc> (tag: <tag>; <evidence>) [attempts=<n>]`, and the final attempt's `agent_run_id` is tombstoned by the phase-level transport branch. The `tag` is the cause of the **last** attempt and `attempts` is the launch count — a run may mix tags (a retried transport flake that then hits a usage limit terminalizes as `llm_usage_limit` with `attempts=2`), so recovery is decided by the tag, not by the presence of the suffix. Every `agent_run_id` the loop issues is therefore tombstoned (the dead retries by the retry loop, the last attempt by the transport branch).

### baseline diff contract on re-submission
- The child baseline diff in a retry after a `record-agent-run` reject walks the live workspace each time and compares it with the baseline (`_compute_changed_paths_against_baseline`). This keeps any filesystem modification after deactivate within the scope of terminal write validation.
- To prevent retry brick-cascade, only `<orch_root>/agent_runs_invalid.jsonl` (and the lock sidecar) is narrowly excluded from the live diff as a runtime-owned single file. This is the file that causes the previous `record-agent-run` failure to contaminate the next retry's diff, and `record-agent-run` itself is confirmed to be the writer. Other control-plane paths such as `<orch_root>/audit/`, `<orch_root>/violations/`, and `<orch_root>/failure_analysis.json` are **not** blanket-exempt — a hook-bypass write by the child remains as a backstop for terminal validation (`tools/orchestration_runtime.py::_should_ignore_runtime_snapshot_path` is the canonical source).
- Per-arid runtime-managed prefixes such as `<orch_root>/launches/...`, `<orch_root>/violations/...`, and `<orch_root>/agents/...` are also excluded from the diff (originally runtime-only directories). The parent scratch under `workspace/tmp/<parent_arid>/` is excluded by the `parent_tmp_root` exclusion of `_validate_actual_write_paths`.
- `deactivate-child` saves the child-authored path set in `<orch_root>/agents/<agent_run_id>/deactivate_snapshot.json` for audit. This snapshot is for manual audit and future debug use only, and is not referenced in the `record-agent-run` validation path (the live diff is always run).

- The feedback direction from a failed phase is fixed per phase:
  - `Compile` failure → only an in-Compile retry (no automatic retry because upstream is the manual Spec).
  - `Generate` failure → an in-Generate retry. When a verify failure of `source_meta.json` is judged `attribution=ir`, go back to `Compile`.
  - `Build` failure → go back to Generate. Because Build itself is a deterministic process, it does not involve an LLM, and forwards to Generate, as `repair_reason`, one of `compile_error` / `link_error` / `make_error` recorded in `build_log` (for details, the retry-trigger section of `docs/workflow/phases/phase_03_build.md`).
  - `Validate` failure → deterministically decide whether to go back to Generate / Compile / Spec by the combination of the `judge`'s `semantic_review.json#findings[*].attribution` and `verdict.json#failure_class` (the canonical decision table is the "Decision criteria for retry on failure" section of `docs/workflow/phases/phase_04_validate.md`).

## Design Policy
- Single responsibility: one `agent` has only one responsibility.
- Hierarchical delegation: control via the 2 systems `conductor -> step agent` and `conductor -> substep agent`.
- Contract-driven: when launching a child `agent`, fix the input contract and output contract, and forbid reads/writes outside the contract.
- Traceability: save all launch / termination events chronologically, and make the same judgment reproducible on re-execution.

## Orchestration instruction contract
### Common required items
- The conductor must record, in the launch request to a child `agent`, `orchestration_id`, `agent_run_id`, `parent_agent_run_id`, `node_key`, `step`, `substep` (when it exists), `ir_ref`, `pipeline_ref`, and `dependency_ref` as required.
- The launch prompt to a child `agent` is based on the corresponding template in `tools/prompt_templates/`, with the in-template placeholders substituted by the actual values of the target `agent_run`. `record-launch` performs this rendering; the conductor supplies the request parameters rather than the rendered prompt.
- The launch request to a child `agent` records `execution input`, `verification input`, `expected output`, `write_root`, and `read_roots` as required.
- `execution input` is limited to the input that the relevant `agent` may directly reference to generate the artifact.
- `verification input` is made explicit as input that the relevant `agent` may use only for pass/fail judgment, consistency confirmation, and dependency confirmation.
- `expected output` is made explicit including the file name, storage location, and update responsibility.
- The parent `agent` must not instruct guessed completion on an input shortage. When there is an input shortage, it instructs a `fail-fast` stop.
- The launch request to a child `agent` records `skill_name`, `skill_ref`, and `skill_must_read_refs` as required.

### Conventions for `ir_ref` / `pipeline_ref` / `dependency_ref`
- `ir_ref` is only `workspace/ir/<node_key_safe>/<ir_id>`, and an additional path segment must not be appended. `<ir_id>` is in the canonical `<slug>_<YYYYMMDD>_<seq3>` form (regex `^[a-z0-9]+(?:-[a-z0-9]+)*_[0-9]{8}_[0-9]{3}$`, canonical source: `docs/workflow/WORKFLOW_CORE.md` and `_SLUG_DATE_SEQ3_PATTERN` of `tools/orchestration_runtime.py`). Place `<node_key_safe>` as the parent directory of `<ir_id>`, and do not prepend it as a prefix to `<ir_id>` itself (because `node_key_safe` contains `__` / `_`, it violates the slug regex).
- `pipeline_ref` is only `workspace/pipelines/<node_key_safe>/<pipeline_id>`, and an additional path segment (including `source/` or `source_meta.json`) must not be appended.
- `dependency_ref` fixes the canonical path per phase. `Compile` records `spec/.../deps.yaml`, and from `Generate` onward records the phase root of `workspace/...` (`ir_ref` or `pipeline_ref`), forbidding a direct `spec` reference.
- In the launch request of `Generate verify`, `source_id` must be recorded as required.
- When a `step agent` / `substep agent` ends with `pass`, each path of `output_refs` must be included under the `ir_ref` or `pipeline_ref` directory recorded in the corresponding launch request.

### `Compile` launch request
- The `skill_must_read_refs` of `Compile.generate` includes `controlled_spec.md`, `tests.md`, `deps.yaml`, and `spec/registry/spec_catalog.yaml`.
- The `skill_must_read_refs` of `Compile.verify` includes the `spec.ir.yaml` generated by `Compile.generate`, `controlled_spec.md`, `tests.md`, and `deps.yaml`.
- `Compile.verify` has as a required responsibility the structural-invariant verification of `spec.ir.yaml` (all cases covered by algorithm.steps / closure consistency of dependency resolution / consistency of the output contract and the algorithm output). In addition, when `impl_defaults.toolchain.language` / `impl_defaults.toolchain.standard` / `impl_defaults.toolchain.build_system` / `impl_defaults.target.architecture` are undefined, it is a `fail` (canonical source: `docs/IMPL_PLAN_SPEC.md` "Required items").

### `Generate` launch request
- The `skill_must_read_refs` of `Generate.generate` includes `spec.ir.yaml`. It must not read `controlled_spec.md` directly.
- The `skill_must_read_refs` of `Generate.verify` includes `spec.ir.yaml`, `controlled_spec.md` (verify-only requirement-fidelity cross-check; `Generate.generate` does not read it), and `tests.md`, plus — as relative paths based on `pipeline_ref` — `lineage.json` and `source/<source_id>/source_meta.json`.

### `Validate` launch request
- The `skill_must_read_refs` of `Validate.execute` includes `spec.ir.yaml` and `pipeline_ref/binary/<binary_id>/binary_meta.json`.
- The `skill_must_read_refs` of `Validate.judge` includes `spec.ir.yaml`, `tests.md`, the `raw/` / `diagnostics.json` / `perf.json` / `quality_check.json` / `trial_meta.json` under the same `run_id`, and `pipeline_ref/source/<source_id>/`. The judge's launch request does not require `source_id` / `source_binary_id` (the runtime enforces only `run_id`); instead the judge reads `trial_meta.json` under the same `run_id` and resolves `trial_meta.json.source_source_id` as `<source_id>` (because trial_meta is written by Validate.execute and the runtime has verified its match with `binary_meta.json.source_source_id`, there is no path for the judge to read the wrong source even in a pipeline where multiple sources coexist due to retries).
- `Validate.judge` must recompute the judgment metrics from `raw/` via an independent path, and confirm consistency with `diagnostics.json`. It must execute an `LLM` semantic check.

## Operations Rules
1. At workflow start, issue an `orchestration_id` and create `workspace/orchestrations/<orchestration_id>/orchestration_meta.json`. Alongside `orchestration_id` / `status` / `started_at` (and, when supplied, `spec_ref` / `source_dependency_ref`), record `repo_revision` — the git version of the repository resolved at run time so a run's artifacts can be traced back to the exact source that produced them. `repo_revision` is `{ "commit": "<HEAD sha>", "dirty": <bool> }` (`dirty` reflects `git status --porcelain`; `workspace/` is gitignored so orchestration artifacts do not flip it). The SHA is resolved at run time, not pinned to a constant, mirroring the model-alias convention, and is captured **once at the initial start** and preserved across re-init / `--resume` (so it stays the source that produced the run rather than being overwritten by a later `HEAD`). When the run directory is not a git checkout (or git is unavailable), `repo_revision` is omitted so an absent revision stays distinguishable from a clean/dirty one; if `HEAD` resolves but the dirty state cannot be determined (or the repository has no commits yet), `repo_revision` is either omitted or carries `dirty: null`.
2. Before the workflow starts, record the preflight result in `workspace/orchestrations/<orchestration_id>/preflight.json`, and when `can_launch_step_agents=true`, `can_launch_substep_agents=true`, and `sandbox_enforced=true` are not simultaneously satisfied, stop with `fail`.
3. Before starting each phase, confirm the phase type, and finalize the launch target as a `substep agent` for `Compile` / `Generate` / `Validate` and a `step agent` for `Build`.
4. The conductor saves `launches/<agent_run_id>.request.json`, `launches/<agent_run_id>.response.json`, `launches/<agent_run_id>.prompt.txt`, and `launches/<agent_run_id>.reply.txt` per launch request of a `step agent` or `substep agent`.
5. The `response.json` and `child.response.json` saved by `record-launch` are a complete save of the leaf launch response, and must not drop the child-`agent` identifier.
6. On completion of each `step agent` and each `substep agent`, save `agents/<agent_run_id>/dialogs/agent.result.json` and `agents/<agent_run_id>/dialogs/agent.summary.txt`.
7. `agent.summary.txt` includes at least the final `status` and the failure cause or main-artifact reference.
8. `launches/<agent_run_id>.prompt.txt` is the body that concretizes the corresponding template in `tools/prompt_templates/`.
9. The conductor reconciles `deps.yaml`, `spec_catalog.yaml`, and the `dependency` section of `spec.ir.yaml`, and finalizes the execution queue based on the `spec` dependencies. **Scope clarification:** a single orchestration targets exactly **one `node`** (one `spec_ref`); items 9/28 describe the dependency-order reconciliation and the phase/substep queue **within that node**, not the automatic execution of dependency nodes' own workflows. By default a missing dependency is fail-closed (`workflow-launch-check` → `dependency_not_ready`; the operator runs each dependency node first). The entrypoint `tools/run_workflow.py --with-deps` opts into running the target's transitive dependency closure bottom-up (one orchestration per node) before the target; without it the default single-node, fail-closed behavior is unchanged.
10. The conductor issues a `step agent` or `substep agent` per launch target, and passes `node_key`, `step`, `ir_ref`, `pipeline_ref`, and `dependency_ref` as input.
11. Before launching the `Compile` of an upper `node`, the conductor reconciles the `ir_ref` and `ir_meta.json.verification_status` per immediate dependency `node`, and must not launch when `direct dependency ir readiness` is not satisfied.
12. Before launching `Generate` onward of an upper `node`, the conductor reconciles the `ir_ref`, `pipeline_ref`, and latest `aggregate_verdict` per immediate dependency `node`, and must not launch when `direct dependency execution readiness` is not satisfied.
13. When `direct dependency ir readiness` or `direct dependency execution readiness` is not satisfied, the conductor records the relevant `node` as `blocked` or `fail`.
13a. **Dependency-freshness readiness (R6-lite).** Both readiness levels additionally require each dependency's *certified* dependency resolution to still be the one the registry derives today. A certified node records the closure it was built against in its `dependency_graph.json` sidecar (each `all_nodes[]` entry a `kind/spec_id@version` node_key); readiness re-derives that closure from the current `deps.yaml` + `spec_catalog.yaml` with the same pure builder (`tools/dependency_graph.py`) and compares. A mismatch means **stale**, not unbuilt: the node is not ready. This is the mechanism behind "when a dependency `spec` is updated, its dependents are regenerated" — bumping a shared node's catalog version makes every dependent stale, and `run_workflow.py --with-deps` re-certifies the closure bottom-up in one run; no content-free version bump of the dependents is required. On a single-node run the launch check names the drifted node and points at `--with-deps`. A node whose derived closure is only itself (a leaf) can never go stale. The scope is **version granularity**: a content change within one `spec_version` is not detected, which the respec discipline (content change ⇒ `spec_version` bump) makes sufficient; content-hash chaining is R6 proper.
14. The conductor makes explicit the `execution input`, `verification input`, and `expected output` of the target `step`.
15. For a phase that has `substep`, the conductor launches each `substep agent` sequentially.
16. The `substep agent` generates its own artifact and the corresponding phase's metadata, and returns `agent_output_ref` to the conductor.
17. The conductor evaluates the child `agent`'s returned result, and finalizes `issue_severity` and whether re-submission is needed.
18. When re-submission is needed and `repair_strategy=reuse`, the conductor may permit continuation repair of the same `agent_session_id`. It issues a new `agent_run_id` and adds a `record-launch` record with `relation_type` of `reuse`.
19. When re-submission is needed and `repair_strategy=restart`, the conductor relaunches a `substep agent` with a new `agent_session_id` and adds a `record-launch` record with `relation_type` of `restart`.
20. For a phase that has `substep`, the conductor verifies the required artifacts of all `substep`, and outputs `step_result.json` to `workspace/orchestrations/<orchestration_id>/steps/<node_key_safe>/<step>/<agent_run_id>/step_result.json`. This `agent_run_id` is the `orchestration agent_run_id` (for the no-substep `Build` phase it is the `step agent_run_id`). `write-step-result` enforces this fail-fast: a wrong-role `--agent-run-id` whose role is resolvable in `agent_runs.jsonl` is rejected at write time (before the file is written and before the phase transitions, so the step stays `child_finished` for retry), rather than only at the Validate `pre_judge` gate. The `substep_agent_run_ids` of `step_result.json` enumerates, without omission, the `agent_run_id` of **all** `substep` launched in that `step` and recorded in `agent_runs.jsonl`.
21. `step_result.json` holds a `retry_decisions` array when re-submission was performed, and records `issue_severity`, `repair_strategy`, `repair_target_agent_run_id`, `new_agent_run_id`, and `repair_reason` in each element. In the `retry_decisions` recorded in a `write-step-result` with `status=pass`, each `new_agent_run_id` is limited to a `pass` run finally adopted into the `effective pass substep` set.
22. A re-submission triggered by a `noncanonical_phase_write_attempt` requires `repair_strategy=restart`.
23. The `status=pass` judgment in the `step_result.json` of a phase that has `substep` is made over the `effective pass substep` set.
24. In a `step_result` with `status=pass`, each run included in the `effective pass substep` set must have terminated with `pass`.
25. The `required_outputs` coverage judgment in a `step_result` with `status=pass` is made over only the `output_refs` of the `effective pass substep` set.
26. The `step agent` verifies its own artifact in a phase that has no standard `substep` (`Build`), and outputs `step_result.json`.
27. The conductor receives `step_result.json` and judges the launchability of the next `step`.
28. `node` execution proceeds sequentially in the dependency order reconstructed from `deps.yaml`, `spec_catalog.yaml`, and `spec.ir.yaml.dependency`. It does not perform parallel execution unless explicitly instructed. When the closure spans multiple `node` (only under `tools/run_workflow.py --with-deps`), the nodes are run bottom-up (dependencies before dependents), one orchestration per node, also sequentially; on the first dependency-node failure the run stops before the dependent/target node.
29. When a `step agent` or `substep agent` is `fail` / `timeout` / `cancel`, the relevant `step` of the relevant `node` is `fail`, and downstream `step` launch is forbidden.
30. The conductor appends each `agent` execution event to `workspace/orchestrations/<orchestration_id>/agent_runs.jsonl`.
31. The conductor saves the parent-child relationship in `workspace/orchestrations/<orchestration_id>/agent_graph.json`, and requires recording `parent_agent_run_id`, `child_agent_run_id`, and `relation_type`.
32. All `agent` of the core workflow must not write outside of `workspace/`.
33. When the actual processing of a `step` / `substep` is proxied by a script during workflow execution, it is a `fail`, and the relevant trial is discarded.
34. On re-submission, issue a new `agent_run_id`, and do not overwrite existing `launch` evidence or `agent_runs` rows.
35. `status` and `can_launch_*` must not be changed by manual editing or post-editing of `preflight.json`.
36. When the live probe just before launching a child `agent` is `fail`, `record-launch` must not be run.
37. The live probe that `record-launch` runs is skipped when a probe that succeeded within the TTL set by `METDSL_PREFLIGHT_TTL_SECONDS` (default 30 minutes) exists. It is disabled when `METDSL_ORCHESTRATION_ENFORCE_LIVE_PREFLIGHT=1` is explicitly set.
38. The execution result of a native hook is appended to `workspace/orchestrations/<orchestration_id>/hooks/native_hook_events.jsonl`.
39. `tools/run_workflow.py` sets `METDSL_MISSING_ORCHESTRATION_ID_POLICY=strict` at workflow start, and forbids hook execution without an orchestration_id.
40. When about to deviate into a contract-violating shortcut in a child-`agent`-required phase, the conductor makes explicit that the relevant phase requires a child-`agent` launch, and returns to the legitimate launch procedure.
41. After `write-step-result` completes with `status=pass`, `orchestration_checkpoint.json` is auto-updated by `tools/orchestration_runtime.py`.
42. In an orchestration with `resume_enabled=true`, the conductor runs `check-step-completed` before launching each `step`, and permits skipping the relevant `step` only when `completed=true` and `integrity=ok`.
43. A `step` skipped by checkpoint is recorded in `agent_runs.jsonl` as `agent_role=skipped_by_checkpoint`.
44. In an orchestration with `resume_enabled=false`, a `step` must not be skipped by trusting `orchestration_checkpoint.json`.
45. When the `status` of `write-step-result` is terminal (`pass` / `fail` / `blocked` / `timeout` / `cancel`), `validation_stage` is required in the `step_result.json` of `Compile` / `Generate` / `Build` / `Validate`. The allowed values are `Compile: compile|full`, `Generate: post_generate|full`, `Build: post_build|full`, `Validate: post_execute|pre_judge|full` (runtime canonical: `STEP_REQUIRED_VALIDATION_STAGES` of `tools/orchestration_runtime.py`). The runtime rejects with `ValueError` for anything other than the per-step allowed values, or on omission.
46. `codex_feature_check.json` is authored host-side by the conductor (orchestration-dir root, leaf-unwritable). The probe runs the leaf's configured command prefix (`--llm-command` wrapper, else the bare backend), so it certifies the executable the leaf actually runs. When the hooks feature is NOT certified (disabled / probe error) and the requirement is on (`METDSL_REQUIRE_CODEX_HOOKS_FEATURE` ≠ `0`/`false`/`no`), the conductor **fails closed before launching the codex leaf** (`SandboxEnforcementError` → `sandbox_enforcement_violation`) — a leaf whose hooks would not fire must not run, since the in-sandbox fail-closed read only happens if the hook runs. The in-sandbox hook itself reads the cache read-only and never re-probes; a missing/invalid/disabled cache fail-closes the hook gate too (defence in depth). The host re-probes on the next orchestration run / `--resume`.
47. `workspace/tmp/<agent_run_id>/` can be used as each agent's temporary working area. The agent directly specifies that literal path (`output_manifest_write_guard` judges only the write-target path and does not reference the `$TMPDIR` env). `record-agent-run` auto-deletes `workspace/tmp/<agent_run_id>/` after recording that `agent_run`. `tools/run_workflow.py` sets the environment variable `TMPDIR` to `workspace/tmp/<orchestration_agent_run_id>/` after `init` succeeds, but this is an insurance for subprocess inherit, and `export TMPDIR=...` on the agent side is unnecessary and forbidden (the Claude Code session sandbox's approval request would stop the workflow).
48. Because `Build` is a step-only phase (one child == one `step_result.json`), the conductor must write the `step agent`'s `step_result.json` — including a **`fail` / `timeout` / `cancel`** outcome — before launching the next `Build` child. `record-launch` enforces this: launching a `Build` child while a **prior terminal `Build` step agent for the node still lacks its `step_result.json`** is rejected fail-closed (the check is against the actual `step_result` files, not the `child_finished` phase-state proxy, so a crash that interrupts the phase transition after the result was already written does not wedge recovery). This runs before any durable launch/session/graph write, so a blocked relaunch leaves no dangling `agent_graph` edge. It prevents the original gap where a failed `Build` was folded into another build's `retry_decisions` and left without its own `step_result`, which later deadlocked `_validate_orchestration_completion_for_pass`.
49. Recovery-only: a `step agent` left **stranded without a `step_result`** (e.g. its `child_finished` authority lost when a checkpoint resume reset the `Build` phase) is remediated with `write-step-result --backfill`, which writes the missing `step_result` directly without consuming a fresh `child_finished` and without advancing the phase state. It is gap-fill only (refuses to overwrite), requires the agent to be a terminal `step` agent in `agent_runs.jsonl` for the same `node_key` / `step`, and requires the `--result-json` `status` to match the recorded run status (the anti-fabrication guard — a recorded `pass` is backfillable, since otherwise a build that recorded `pass` but lost its `child_finished` before the result was written would have no recovery path). Canonical: `docs/CLI_REFERENCE.md#write-step-result`.
50. Cross-phase retry of an already-passed phase (`reopen-phase`): when the decision table (`docs/workflow/phases/phase_04_validate.md`, `docs/workflow/phases/phase_02_generate.md`) routes a `Validate` / `Generate` failure back to a `Compile` that is already checkpointed `pass`, the conductor must run `reopen-phase --from-phase compile --node-key <node_key> --trigger-agent-run-id <failed_downstream_arid> --reason <reason_code>` before re-launching `Compile`. The re-point cannot otherwise be expressed in place: `check-step-completed` reads the stale IR as `integrity=ok`, the phase sits at `step_result_written` (not the `child_finished` the `write-step-result` path requires), and `retry_decisions` only models a within-step substep retry. `reopen-phase` invalidates the `from_phase` and all downstream phases — snapshotting the prior attempt's step/substep runs as **superseded** (`reopen/superseded_runs.json`, exempt from the `_validate_orchestration_completion_for_pass` vouch), archiving their `step_result.json` aside, dropping the affected `completed_steps` checkpoint entries, and resetting the affected `phase_state` to `not_started` — so `Compile` → `Generate` → `Build` → `Validate` re-run against the corrected IR. `--trigger-agent-run-id` must be a terminal non-pass step/substep strictly downstream of `--from-phase` (the anti-abuse gate). On `--resume`, `orchestration_meta.resume_directive` supplies the arguments. Canonical: `docs/CLI_REFERENCE_RARE.md#reopen-phase`, `docs/RUNBOOK.md` §3-1.

## Decision Criteria
- Per workflow, an `orchestration_id` is issued and `orchestration_meta.json` exists.
- Each `step` or each `substep` has an independent `agent_run_id`.
- The `context_id` of `step` and `substep` do not duplicate, and `context_isolated=true` is recorded for all.
- The `agent_runs.jsonl` of `step` and `substep` records `agent_session_id` and the various references, and the referenced entities exist.
- `launches/<agent_run_id>.response.json` and `agents/<agent_run_id>/dialogs/child.response.json` hold the same content of the leaf launch response.
- `agent_runs.jsonl.agent_session_id` matches the child-`agent` identifier of the corresponding `launch response`.
- `preflight.json` exists and satisfies `can_launch_step_agents=true` and `can_launch_substep_agents=true`.
- `sandbox_runtime=bwrap` and `sandbox_enforced=true` are recorded in the preflight.
- From the execution record of each phase, it can be traced that `Compile` / `Generate` / `Validate` used a `substep agent` and `Build` used a `step agent`.
- The parent-child relationship `orchestration -> step` or `orchestration -> substep` can be traced in `agent_graph.json`.
- The `executor_agent_run_id` of `step_result.json` matches the relevant directory name, and `substep_agent_run_ids` is consistent with the parent-child relationship. For a phase with no standard `substep` (`Build`), `substep_agent_run_ids=[]` is allowed.
- For a `step` that has `substep`, the `agent_run_id` of all `substep` of that `step` recorded in `agent_runs.jsonl` is included in the `substep_agent_run_ids` of some `step_result.json`. A `substep` / `step` recorded in `reopen/superseded_runs.json` is exempt from this vouch requirement. The invariant is that a terminal `agent_run_id` no `step_result.json` can vouch for is ALWAYS tombstoned, by whichever writer replaced or discarded it: a `reopen-phase` cross-phase retry (its `step_result.json` was archived to `step_result.superseded.<seq>.json`); a phase that terminalized without writing a `step_result.json` at all (`leaf_transport_error_orphan:` / `validate_gate_fail_orphan:` / the escalate-terminal orphans); an attempt replaced by a warm resume (`validate_post_judge_warm_resume_orphan` / `<phase>_verify_meta_schema_warm_resume_orphan`); and a dead attempt of a **leaf transient retry**, superseded by the surviving attempt of the same `substep` (`leaf_transient_retry_orphan:`). A reopened `(node, phase)` (every `(node_key, step)` carried by a superseded run) must nonetheless regain at least one fresh non-superseded terminal run before `pass`; otherwise `_validate_orchestration_completion_for_pass` rejects the `pass`, so a `reopen-phase` that archived the only evidence cannot be marked `pass` until the replacement attempt runs. A recovered transient retry satisfies this by construction (its surviving attempt is a fresh non-superseded terminal run of the same `(node_key, step)`).
- The `required_outputs` of `step_result.json` match the phase contract of `docs/workflow/WORKFLOW_CORE.md` and the corresponding `docs/workflow/phases/phase_*.md`.
- When `step_result.json` holds `retry_decisions`, the `effective pass substep` set can be uniquely restored.
- For `Compile` / `Generate` / `Build` / `Validate` where `step_result.json` has a terminal status, `validation_stage` matches the per-phase allowed values (item 45).

## Patch application contract (removed in P2-7)

`guarded-apply-patch` and its `apply_patch_writes` gate have been **removed** (the subcommand no longer exists). Under Phase-2 a child writes every artifact **directly with the `Edit` / `Write` tool**; the write is authorized at terminalization by `write_roots`-containment of the FS-diff under mandatory `bwrap` (`_validate_actual_write_paths`), with no gate evidence. ## Capability / Manifest contract

The required fields and invariants of the 3 manifests that `record-launch` issues (canonical source: this section).

### Required fields of `capabilities/<agent_run_id>.json`

| field | type | description |
|---|---|---|
| `agent_run_id` | string | the child agent's UUID |
| `capability_token` | string | a 32-byte hex token |
| `orchestration_id` | string | the orchestration ID |
| `agent_role` | `"orchestration"\|"step"\|"substep"` | the agent role |
| `node_key` | string | `<spec_kind>/<spec_id>@<spec_version>` |
| `step` | string | `"compile"\|"generate"\|"build"\|"validate"` |
| `substep` | string (optional) | the substep token, present for substep agents; scopes the per-substep `write_root` file pins (`compile.verify`→`ir_meta.json`, `generate.verify`→`source_meta.json`, `validate.judge`→`semantic_review.json`) |
| `write_roots` | array of strings | the list of write roots the capability permits (a directory root, trailing `/`, or a single-file pin) |
| `mcp_permissions` | array of strings | the gated build-runtime MCP tools this capability permits, granted per-substep least-privilege: `build`→`["compile_project"]`, `generate.lint`→`["run_linter"]`, `generate.syntax`→`["run_syntax_check"]`, `validate.execute`→`["run_program", "run_quality_checks"]` (the four conductor in-process bodies that call a gated tool). **Every other `(step, substep)` fails closed to `[]`** — all LLM leaves (`compile`/`generate`.`generate`/`verify`, `validate.judge`, forbidden by SKILL from calling any MCP tool) and every other deterministic in-process validator (`generate.static`, `compile.static`, `validate.pre_judge`/`post_judge`). |
| `expires_at` | ISO8601 | the capability expiry |

**Invariant:** a capability whose `agent_role` is `"step"` or `"substep"` must not have an empty array for `write_roots`. **The single exception is a `pure-function leaf`** (§pure-function leaf (`Z2` generate-executor)), whose capability declares `mode: "pure_readonly"` with `write_roots: []` and `mcp_permissions: []` by construction — it holds no write authority. The builder relaxes the empty-`write_roots` rejection only for that mode; every other `step` / `substep` capability still fails closed on an empty array. An empty `write_roots` is enforced as deny-all (never fail-open): any change inside a pure leaf's window is an unauthorized write.

### Required fields of `output_manifests/<agent_run_id>.json`

| field | type | description |
|---|---|---|
| `allowed_output_paths` | array of strings | the permitted path set for `.json`/`.txt` output |
| `allowed_file_tool_paths` | array of strings | the permitted path set for direct `Edit`/`Write` |
| `allowed_tmp_root` | string | the temporary-file permission root (`workspace/tmp/<agent_run_id>`) |

**Usage:** the agent writes by directly specifying the literal path of `allowed_tmp_root` (`workspace/tmp/<agent_run_id>/...`). Because `output_manifest_write_guard` judges only the write-target path and does not reference the `$TMPDIR` env, bootstrap Bash such as `export TMPDIR=...` / `jq -er ...` is unnecessary and forbidden (the Claude Code session sandbox's approval request would stop the workflow; for details see the tmp-area rules in `docs/AGENT_CONTRACT.md`).

### Required fields of `read_manifests/<agent_run_id>.json`

| field | type | description |
|---|---|---|
| `allowed_read_roots` | array of strings | the list of root paths that are permitted to read |
| `denied_read_roots` | array of strings | the list of root paths that are explicitly denied |

`output_manifests/<agent_run_id>.json` and `read_manifests/<agent_run_id>.json` may be read directly with the `Read` tool (`run-gate` not needed).
