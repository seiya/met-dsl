# Controlled Spec（正本）の要件と書式

## 目的
Controlled Spec は本プロジェクトの**唯一の物理仕様正本**であり、
- ドメイン研究者が読んで理解できること
- 実行モデルへ**決定的に**変換できること
を同時に満たす必要がある。

## 役割分離（最重要）
本プロジェクトでは、仕様を次の 2 種類に分ける。

1. `Controlled Spec`（本書）
- 物理・数値アルゴリズムの定義を書く。
- 生成器はこれを入力として、物理計算を担う model を作る。

2. `physical_tests`（`physical_tests.md`）
- 妥当性検証で使う入力条件（初期条件、実行条件、ケース展開）と判定閾値を書く。
- テストランナーはこれを入力として検証を実行する。
- 記述規律は `PHYSICAL_TESTS.md` を正本とし、自然言語中心で定義する。

注意:
- 実行モジュールはテスト専用ではない。科学計算の本番実行では、ユーザーが目的に応じて実行時入力を設計できる。
- `physical_tests` は、その実行時入力のうち「妥当性検証に使う既定プロファイル」を定義する文書である。
- 言語に依らず、生成物は model（物理計算）と runner（実行・判定連携）を分離し、runner が model を呼び出す構造にする。

境界ルール:
- `Controlled Spec` に**ケース固有の設定**（nx sweep、t_end、case_id 群など）を書かない。
- `physical_tests` に**離散化スキームの定義そのもの**を書かない。

## 基本方針（自然言語ファースト）
- 記述の主役は**自然言語**とする。
- 構造化ブロック（YAML/JSON/ 表）は、**自然言語だけでは曖昧になる箇所**に限定する。
- `Markdown` の数式記法は、インラインを `$...$`、ブロックを `$$...$$` とし、`\(...\)` と `\[...\]` を使用しない。
- LLM や変換器による欠落補完は禁止する。**不足はエラー**として扱う。
- 物理アルゴリズム（A）は `Controlled Spec` で固定し、実行アルゴリズム（B）は `impl.resolved.yaml` で扱う。

## 記述フォーマット（固定テンプレート）
冒頭に **0. メタ情報** を置き、その後に以下の節をこの順で記述する。節名は原則変更しない。

0. **メタ情報（先頭固定）**
- 必須文: `spec_id`、`spec_version`、`status` を文書先頭で明示する。

1. **問題定義**
- 必須文: 対象方程式、保存形/非保存形、対象変数、物理仮定を文章で明示する。

2. **変数と座標の定義**
- 必須文: 各変数について「名前・意味・配置・単位」を文章で明示する。
- 必須文: 座標系と次元を文章で明示する。

3. **領域と境界条件の型定義**
- 必須文: 領域の型（例: 1D 一様格子）と、境界条件アルゴリズム（例: 周期写像）を文章で明示する。
- 必須文: 実行時入力（runtime inputs）として可変な項目（例: `nx`,`L`）を明示する。
- 必須文: 検証時に `physical_tests` が与えるのは、実行時入力の一部プロファイルであることを明示する。

4. **物理アルゴリズム（A）**
- 必須文: 空間離散化、時間積分、安定化/フィルタの有無を文章で明示する。
- 必須文: 選択肢が複数あり得る箇所は、固定値または優先順位を文章で明示する。

5. **モデルパラメタと実行時入力契約**
- 必須文: 物理定数の固定/可変、単位、既定値を明示する。
- 必須文: 実行時入力契約（初期条件、終了時刻、刻み規則など）を列挙する。
- 必須文: 検証で用いる既定入力は `physical_tests` に定義することを明示する。

6. **禁止事項**
- 必須文: 未サポート機能、暗黙補完禁止、未定義パラメタの扱いを文章で明示する。

7. **トレーサビリティ**
- 必須文: `spec_version`、参照文献/根拠、`case.resolved.yaml` への対応規則を文章で明示する。

8. **AD 準備情報**
- 必須文: `ad_readiness.enabled` の値と、`true` の場合の必須情報を文章で明示する。

## 構造化ブロックの使用基準
### 使ってよい場合（真に必要な場合のみ）
- パラメタ一覧のように、**項目数が多く取り違えやすい**場合
- 入出力スキーマのように、**機械処理で厳密一致が必要**な場合
- 自然言語では冗長になり、可読性が下がる場合

### 使ってはいけない場合
- 方針説明、背景説明、選定理由などの説明文
- 1 つか 2 つの値を示すだけの箇所
- 文章で一意に定義できる箇所

## 曖昧性排除ルール
- 「適切」「十分小さい」などの主観表現は禁止。
- 取り得る選択肢が複数ある場合は、**選択規則**（優先順位または固定値）を明示する。
- 単位・閾値・数式・手順で記述する。
- 未定義項目は「暗黙の既定値」で埋めず、Spec 不備として差し戻す。

## レビュー時チェックリスト
- 文書先頭にメタ情報（`spec_id`,`spec_version`,`status`）がある。
- 物理・アルゴリズム定義とテスト入力条件が混在していない。
- ケース固有設定（sweep、`t_end`、`case_id`）が本書に入っていない。
- 実行時入力契約が明記され、`physical_tests` は検証用プロファイルとして位置づけられている。
- 未定義パラメタ、単位抜け、閾値抜けがない。
