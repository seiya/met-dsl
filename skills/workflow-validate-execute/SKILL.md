# Workflow Validate Execute

## Purpose
As the execute substep of the Validate phase, generate judgeable runtime artifacts. This substep operates non-LLM (MCP only), and the pass/fail judgment is borne by the same phase's `Validate.judge` substep in an independent LLM context.

## Scope
- the artifact generation of `workspace/pipelines/<pipeline_id>/runs/<run_id>/<node_key_safe>/`
- `runner` execution and `quality check` execution

## Requirements
- The validator gates this phase can launch use the "substep ↔ allowed validator gate correspondence table" of `skills/workflow-orchestration/references/launch_prompts.md` as the canonical source.
- `run` uses the `MCP` server's `run_program`.
- The execution command of `run_program` must always include `spec.ir.yaml.case` as input arguments.
- `quality check` uses the `MCP` server's `run_quality_checks`.
- To establish a `quality check`, `test` source, harness, auxiliary `script`, or a temporary `Makefile` must not be additionally generated under `runs/<run_id>/<node_key_safe>/`.
- With `spec.ir.yaml.impl_defaults.toolchain.build_system=make` and `toolchain.language=fortran` / `c` / `cpp` / `mixed` families, the `quality check` must run with `run_quality_checks preset=make_test` or `preset=make_check` against `source/<source_id>/src/`. When no conforming path exists, it is a `Validate.execute fail`.
- The above `run_quality_checks` must pass the out-of-source dirs via `env`: `env={OBJDIR:<abs>/workspace/tmp/<agent_run_id>/build, BINDIR:<abs>/<pipeline>/binary/<source_binary_id>/bin, RUNDIR:<abs>/workspace/tmp/<agent_run_id>/qc_run}`. Because `binary/` and `source/` are read-only-bound in Validate.execute, `make test` must not relink (reference the existing binary via the Makefile's `test -x $(BINDIR)/$(BIN) || $(MAKE) …` guard), and the `diagnostics.json` / `perf.json` / `raw/*` emitted by the `make test` binary re-run are confined under **tmp (`workspace/tmp/<agent_run_id>/qc_run`, a separate subdir from `run_program`'s `run/`)**. `RUNDIR` must not be pointed at the canonical run node dir: the canonical run node dir is this substep's write_root, but the canonical `.json` is not authorized merely by being under the write_root and requires the gate evidence of `guarded-apply-patch`, so the `make test` direct binary write overwrites the gate-authored copy and invites an `unauthorized_write_violation` → `fail_closed` at the terminal `record-agent-run`. A `src/` write during `Validate.execute` allows only the cross-phase audit log (`source/<source_id>/src/mcp_command_log.jsonl`), and anything else (the `src/` output of `.o`/`.mod`/exe/`raw/`/`diagnostics.json`/`perf.json`) invites an `unauthorized_write_violation`.
- The `runner` calls the `model` and outputs `diagnostics.json` and `perf.json`.
- The `runner` (the binary that `run_program` runs) **must not be made to write `diagnostics.json` / `perf.json` directly** to the canonical run dir (`runs/<run_id>/<node_key_safe>/`). Because these are canonical `.json` and require the gate evidence of `guarded-apply-patch`, a direct binary write invites an `unauthorized_write_violation` → `fail_closed` at the terminal `record-agent-run` (this is handled differently from an MCP-owned audit log such as `mcp_command_log.jsonl`). The launch points the working/output destination of `run_program` (the `project_dir` or the case output destination) at `allowed_tmp_root` (`workspace/tmp/<exec_agent_run_id>/run/`), and the binary drops `diagnostics.json` / `perf.json` / `raw/` / log to tmp (auto-authorize). Make `command_log_path` explicit as the canonical `runs/<run_id>/<node_key_safe>/mcp_command_log.jsonl` (it satisfies the MCP log placement gate even with `project_dir` as tmp).
- The execute agent reads the `diagnostics.json` / `perf.json` of the **`run/` tmp tree** (`workspace/tmp/<agent_run_id>/run/`, = the output of the `run_program` execution with `spec.ir.yaml.case` as arguments), and re-authors the canonical `runs/<run_id>/<node_key_safe>/diagnostics.json` / `perf.json` with `guarded-apply-patch` (create-form). **The runner's program output (`diagnostics.json` / `perf.json` / `raw/metrics_basis.json` / `raw/state_snapshots/*`) is always promoted from `run/`, and must not be promoted from `qc_run/` (the output of the `run_quality_checks` / make-test re-run)** — if `qc_run/` is promoted, the canonical evidence does not correspond to the required `run_program` invocation (the `spec.ir.yaml.case` and its `command_log`), and `Validate.judge` consumes evidence with a provenance mismatch. The output of `qc_run/` is referenced only for the quality-check comparison (the verdict computation of `quality_check.json`). `trial_meta.json` and `quality_check.json` are **not runner output but metadata the agent authors** (`trial_meta.json` is composed from the MCP command refs / `source_source_id` / `source_command_ref`, and `quality_check.json` from the comparison result. The runner must not output either file directly), and are generated with `guarded-apply-patch` rather than promoted from tmp. The `.json` under `raw/` (`metrics_basis.json` / `state_snapshots/*.json` etc.) **is also canonical `.json` and is re-authored with `guarded-apply-patch`** (writing it with the `Write` tool is rejected by `enforce_guarded_apply_patch`, and even if it could be written, the terminal `record-agent-run` detects an `unauthorized_write_violation` from the missing gate evidence). What is written with the `Write` tool is limited to **non-`.json`** files under `raw/`, `stdout.log`, and `stderr.log`, all written to a path within `allowed_file_tool_paths`.
- The binary output of both `run_program` and `run_quality_checks` is dropped to tmp (`run/` / `qc_run/`), and must not be written directly to the canonical run node dir. Run the `guarded-apply-patch` re-author of all canonical `.json` (`diagnostics.json` / `perf.json` / `trial_meta.json` / `quality_check.json` / `raw/metrics_basis.json` / `raw/state_snapshots/*`) as the **final step after both MCP calls (`run_program` and `run_quality_checks`) complete**, ensuring an order in which a subsequent binary re-run does not overwrite the gate-authored copy. The ordering defect in which re-authoring after `run_program` is overwritten by the `run_quality_checks` (`make test`) binary re-run is the typical cause of an `unauthorized_write_violation`.
- The `runner` must not directly output `verdict.json`, `aggregate_verdict.json`, `summary.json`, or `trial_meta.json` (these are the responsibility of `Validate.judge`).
- The agent-authored `quality_check.json` must simultaneously satisfy `checks.verdict_available=true`, `checks.diagnostics_match=true`, and `checks.verdict_match=true`, **and record the literal `"pass"` in the top-level `status` field**. When `status` is other than `"pass"` or missing (e.g. only `verdict:"pass"`), the `post_execute` gate rejects it with `quality_check.json:status must be pass`.
- The `Validate.execute` completion condition includes that `diagnostics.json` and `perf.json` are a UTF-8 `JSON object` restorable by a standard `JSON` parser. When an invalid `JSON` is detected, it is a `Validate.execute fail`, and `Validate.judge` must not start.
- Before `Validate.execute` completes, run `python3 tools/check_artifact_syntax.py --format json --expect-top object`, and must check the syntax validity of `diagnostics.json`, `perf.json`, `quality_check.json`, and `trial_meta.json`.
- Require saving the execution evidence for `Validate.judge` recomputation into `runs/<run_id>/<node_key_safe>/raw/`. The required composition uses `spec.ir.yaml.io_contract.raw_requirements.required_evidence` as the canonical source.
- Require saving a state snapshot only when `raw_requirements.required_evidence` declares `artifact=state_snapshots` as required.
- `raw` saves only the primary evidence, and a copy of `diagnostics.json` must not be saved as `metrics_basis`.
- Require saving `stdout.log`, `stderr.log`, and `trial_meta.json`.
- Record `runner_command`, `process_trace_ref`, and `raw_artifact_refs` in `trial_meta.json` as required.
- Record `source_source_id` in `trial_meta.json` as required. The value is the id of `<pipeline>/source/<source_id>/` that this `Validate.execute` references in `quality_check`, and that `source_meta.json`'s `verification_status=pass` must hold (referencing a failed / stale source as quality_check evidence is forbidden).
- The Validate launch request requires recording `source_binary_id`, specifying the id of `<pipeline>/binary/<source_binary_id>/` from which this `Validate.execute` obtains the binary. `record-launch` cross-reference-verifies that the `source_source_id` of `<source_binary_id>/binary_meta.json` and the request's `source_id` match, and rejects a mismatch (mixed-binary forge prevention).
- Each entry of `trial_meta.json`'s `source_command_ref` requires declaring the `tool_name` field, specifying either `run_program` or `run_quality_checks` (`compile_project` is a build-phase tool recorded in binary_meta.json and is not accepted in the execute trial_meta). The entry's `tool_name` must match the `tool_name` of the corresponding MCP `command_log` record. At least one entry must have `tool_name='run_program'` (the actual program-execution evidence).
- The MCP `command_log` output of `run_program` uses `<run node_dir>/mcp_command_log.jsonl` (= `<pipeline>/runs/<run_id>/<node_key_safe>/mcp_command_log.jsonl`) as the canonical placement, and `source_command_ref.<run_program-key>.command_log_ref` allows only that path. When calling `run_program`, configure with the `command_log_path` argument or the `project_dir` setting so the log lands at that path.
- The MCP `command_log` output of `run_quality_checks` allows only the cross-phase canonical placement (`<pipeline>/source/<source_source_id>/src/mcp_command_log.jsonl`). A non-canonical placement (e.g. an arbitrary `.jsonl` under `raw/`) is rejected by the `post_execute` validator.
- The quality comparison of `impl_defaults.target.class=cpu` saves the execution results of `threads_per_rank=1` and `threads_per_rank>1` as comparison targets.
- The comparison canonical source of `quality check` is `diagnostics.json` and `verdict.json`, and pass/fail must not be finalized by `stdout` diff alone.
- The storage root for workflow artifacts allows only `workspace/`, and the workflow-root judgment targets only `workspace/`.
- Before `Validate.execute` completes, run `python3 tools/validate_pipeline_semantics.py --stage post_execute --pipeline-root <pipeline_root> --run-id <run_id>`, and on `fail` it is a `Validate.execute fail`. `--pipeline-root` can be specified repeatedly, and in a trial holding `spec.ir.yaml.dependency.all_nodes`, all `pipeline_root` corresponding to `all_nodes` must be specified. For `--run-id`, specify this trial's `run_id` to scope the verification to that run (to avoid failing on a broken sibling run of a past retry that remains in the `append-only` pipeline).

## Operations Rules
1. Issue a `run_id`, and save artifacts separately per `run_id`. `run_id` is the `run_<YYYYMMDD>_<seq3>` form with a fixed literal `run_` prefix (e.g. `run_20260605_001`), and the `<slug>_<YYYYMMDD>_<seq3>` form of `ir_id` / `pipeline_id` must not be reused (a hyphen-slug form such as `run-rsn-p0_20260605_001` is rejected by the phase contract of `record-launch`, and even if it passes, the run discovery of `post_execute` silently fails).
2. To avoid mixing judgment input, forbid artifact references across `run_id`.
3. Separate the artifact directory per `node_key`.
4. On an execution failure, record the environment information and the failure cause in `trial_meta.json`.
5. When the `raw` execution evidence is missing, it is a `Validate.execute fail`, and `Validate.judge` must not start.
6. When the output destination is not `workspace/`, it is a `Validate.execute fail`, and the relevant `run` is invalidated.
7. When `workspace/` does not exist before workflow execution starts, create `workspace/` directly under the repository root.
8. Before start and before completion, run `python3 tools/validate_workspace_root.py`, and on `fail` it is a `Validate.execute fail`.
9. Run `python3 tools/validate_pipeline_semantics.py --stage post_execute --pipeline-root <pipeline_root> --run-id <run_id>`, and on `fail`, `Validate.judge` must not start.
10. Before completion, run `python3 tools/check_artifact_syntax.py --format json --expect-top object` on `diagnostics.json`, `perf.json`, `quality_check.json`, and `trial_meta.json`, and on `fail` it is a `Validate.execute fail`.

## Decision Criteria
- `diagnostics.json`, `perf.json`, and the log group are present.
- The `raw` execution evidence and the reference information of `trial_meta.json` are consistent.
- The artifacts are separated per `node_key`.
- The execution method is limited to `MCP run_program` and `MCP run_quality_checks`.
- `python3 tools/validate_pipeline_semantics.py --stage post_execute` returns `exit code 0`.
