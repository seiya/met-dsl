# Workflow Compile Generate

## Purpose
Fix the generation responsibility of the Compile stage, and create a deterministic structured IR (`spec.ir.yaml`) from the input spec.

## Scope
- the work of generating the `case` section of `spec.ir.yaml` from `controlled_spec.md` and `tests.md`
- the work of generating the `algorithm` section of `spec.ir.yaml` from `controlled_spec.md` and the dependency-resolution result
- the work of generating the `impl_defaults` section of `spec.ir.yaml` by applying the defaults of `target` and `toolchain`
- the work of generating the `dependency` section (`node_key` + `direct_deps` only) of `spec.ir.yaml` from `deps.yaml` and `spec_catalog.yaml`
- the work of deriving the `io_contract` section (the IO contract and verification contract) of `spec.ir.yaml`

## Requirements
- Use `spec/<...>/controlled_spec.md`, `spec/<...>/tests.md`, `spec/<...>/deps.yaml`, and `spec/registry/spec_catalog.yaml` as the canonical source for the input.
- For the requirement definition of the output format, input/output contract, and judgment conditions, reference `controlled_spec.md`, `tests.md`, `deps.yaml`, and the `docs/` canonical source, and do not reference the verification `python` scripts and `quality check` implementation under `tools/`.
- Do not require additional required items in the `Controlled Spec`. Derive the verification contract from the existing input.
- `spec.ir.yaml` integrates and holds the 5 sections `case` / `algorithm` / `impl_defaults` / `io_contract` / `dependency` in a **single file**. The canonical source for the schema is `docs/workflow/phases/phase_01_compile.md`.
- Expand the `sweep` and `refinement` of the `case` section in a deterministic order. `case` holds only the determined values of runtime input, and must not hold the verification output contract.
- The `algorithm` section requires holding `execution_mode`, `steps[]`, `ordering`, `control_condition`, `iteration_contract`, `update_semantics`, `temporaries`, `derived_field_rules`, `invariants`, and `splitting_policy`, and expresses the `problem`'s integration order, the dependency `operation` call order, conditional branches, iteration conditions, column processing, derived-quantity definitions, and update targets.
- The notation rules for `temporaries[].shape_expr` and `state_variables[].shape_expr` use `spec/schema/ir/shape_expr.schema.json` as the canonical source. The allowed forms are limited to the 3 forms `scalar` (case-insensitive) / `[d1, d2, ...]` / `(d1, d2, ...)`, and function-call notation such as `vector(N)` / `matrix(M,N)` / `tensor` is a `Compile fail`.
- `algorithm.execution_mode` allows only `sequence` / `conditional` / `iterative` / `columnwise`.
- Each element of `algorithm.steps[]` requires holding `step_id`, `step_kind`, `operation_ref`, `inputs`, and `outputs`. `inputs` and `outputs` are a **list of non-empty strings** (e.g. `["U_L", "U_R", "h_B", "h_T"]`), and the object form (`[{name: ..., source: ...}]`) is forbidden. Reference form: `docs/examples/spec_ir_algorithm_section.example.yaml`.
- Each string token appearing in `algorithm.steps[].inputs` / `algorithm.steps[].outputs` must be **traceable** to one of a direct input/output variable of `controlled_spec.md`, an intermediate variable of `temporaries`, or a derived quantity of `derived_field_rules`. A name derived by a slice, alias, or component decomposition (e.g. `h_L` ← `U_L[0]`) must be declared in `temporaries` or `derived_field_rules` to make its provenance explicit. A token that cannot be mapped to either a direct name or a derived name is a `Compile fail` because the dataflow cannot be traced from `spec.ir.yaml` alone.
- `algorithm.steps[].step_kind` allows only `boundary_apply` / `reconstruct` / `flux_compute` / `source_term` / `time_integrate` / `column_process` / `pointwise_process` / `iterative_solve` / `filter` / `reduction` / `diagnostic`.
- **Self-sufficiency: lower each NON-dependency `operation_ref` with its defining math, not just a name.** A step resolved by a dependency `call` carries its definition in the certified dependency. But a LOCAL operation the node implements itself (a reconstruction, source term, limiter — not in `dependency.direct_deps`) must be lowered into `spec.ir.yaml` WITH the math a `spec/`-blind `Generate` leaf needs: the defining/update expression + the constraints pinning its form (and any form `controlled_spec.md` FORBIDS, carried into the IR). E.g. an Audusse reconstruction lowers `h* = max(0, eta − max(z_b,i, z_b,i+1))` and the ban on an analytic bed-slope source, not merely the operation names + I/O shapes + invariants. Name + I/O shape + invariants ONLY is a `Compile.verify` **major** (a tool-less `pure` leaf cannot read `spec/`). Do NOT invent a new IR field — use the stable existing ones (`derived_field_rules`, step/operation descriptions, `invariants`). Removal trigger for the interim `controlled_spec.md` carve-out on the `pure` leaf (phase_02 §Generate-executor).
- `spec.ir.yaml` is the canonical source input of `Generate`, and must hold information that lets `Generate` implement without reading `controlled_spec.md` directly.
- Auto-generate `algorithm.summary.md` and save it as a view-only artifact under `workspace/ir/<node_key_safe>/<ir_id>/`.
- When the user does not specify the language, adopt `fortran` for `impl_defaults.target.class=cpu` and `cuda_fortran` for `target.class=gpu`.
- When `impl_defaults.toolchain.language` is a `fortran` / `c` / `cpp` / `mixed` family, choose `toolchain.build_system` from `make` / `cmake` / `meson` / `ninja`, and adopt `make` when unspecified.
- With `impl_defaults.target.class=cpu` and no specification of the loop parallelization method, record `OpenMP` as a default application to parallelizable loops within `impl_defaults.abstract`.
- The fixed / knob layer boundary of `impl_defaults` (`target.*` / `toolchain.*` / `selected.*` are fixed, `abstract.*` / `backend_overrides.*` are knob) uses the "fixed / knob boundary of impl_defaults" section of `docs/workflow/phases/phase_01_compile.md` as the canonical source. `Compile.generate` decides all fixed sub-keys without omission, and finalizes the leaf values of the knob layer as defaults too (no plug-hole such as `null` / `<TBD>`).
- The `dependency` section records ONLY `node_key` and `direct_deps[]` (each with `kind` + `operations`) — the directly-read deps from `deps.yaml`. Do NOT author `transitive_deps` / `all_nodes`: the conductor derives the closure/topo graph into `<ir_ref>/dependency_graph.json`, and `--stage compile` cross-checks your `direct_deps` against it, so `direct_deps` must exactly match the directly-required set of `deps.yaml`. **R1/M3c-β:** an `infrastructure/...` (runner-harness) direct_dep carries `operations: []` — a physics node never *calls* the harness API (its runner is host-rendered glue; the harness surface is never surfaced to the leaf). `--stage compile` also pins that a declared infrastructure dep equals `harness_<language>_<target.class>` (e.g. `harness_fortran_cpu`), so it must match this node's own target.
- The `io_contract` section (IO + verification contract) is **authored by this `Compile.generate substep`** (all 5 sections are produced here) so the deterministic `Compile.static` gate — whose `--stage compile` requires a structurally-complete `io_contract` — runs on a complete IR before `Compile.verify`. Author it per the `io_contract` schema + V3 invariants of `docs/workflow/phases/phase_01_compile.md` (force-read): `inputs`/`outputs`, `raw_requirements.required_evidence`, `test_evidence_requirements` (the `tests.md` `test_id` set, neither more nor less — gated; each **sufficient for independent recomputation**: recompute *inputs* e.g. `U_L`/`U_R` for `F*=F(U_L)`, declared in `schema.variables`), `diagnostics_contract` (covering `tests.md §3`), `semantic_dependency.required_sources`. `Compile.verify` only CHECKS this section (V3); it does not author it.
- Two `io_contract` rules the `Compile.static` gate (`--stage compile`) enforces and a generated IR commonly violates (canonical: V3 of `docs/workflow/phases/phase_01_compile.md`): (1) **every `inputs[]` entry needs a non-empty `evidence_ref`** (not just `outputs[]`; same form, e.g. `raw/state_snapshots`) — an `inputs[]` entry with only `name` + `shape_expr` fails; (2) **`raw_requirements.required_evidence[].artifact` must normalize to one of `state_snapshots` / `metrics_basis.json` / `execution_trace.json`** (author the canonical bare form) — `metrics_basis` / `execution_trace` without the `.json` suffix fails. **R1/M3c-β:** on a harness-backed node the host-rendered runner writes the snapshot time under the harness's fixed key `t` — so `state_snapshots` `schema.time_variable` MUST be `t`; `--stage compile` fails other names. `time_variable` is one of several **render preconditions** `--stage compile` pins on a harness-backed node — author against the full list under "Verification tools" in `docs/workflow/phases/phase_01_compile.md`.
- **`io_contract.test_predicates` (R2) — author the deterministic per-test verdict DSL.** One entry per `tests.md` `test_id` (set equality gated by `--stage compile`): `{test_id, expected_outcome: pass|xfail, target_cases: [<case_id>...], pass_when: {all: [{ref, op, value, per_case?, case?, na_allowed?}]}}`. Formalize each test's `pass_when`/`judgment` prose as a conjunction of comparisons over `diagnostics.json` — `op ∈ eq/ne/le/ge/lt/gt/includes`; `ref` is a dotted path resolving to a declared vocabulary: `verdict.<field>` (overall/failed_checks), `checks.<id>...` (an `id` in `diagnostics_contract.checks`), or a per-case metric address that **must be pinned in `diagnostics_contract.metrics`** (add that list for problem specs whose tests compare `metrics.*`/`errors.*`/`cfl.*`/`convergence.*` — the per-case addressing pin). Choose each condition's **scope** deliberately: `per_case: true` resolves `ref` inside EVERY `target_cases` slice and must hold in all of them (a per-case value map `value: {per_case: {<case_id>: v}}` expresses nx-dependent thresholds and must cover EVERY target case); `case: <case_id>` resolves `ref` inside exactly ONE target case's slice (it must be a member of this predicate's `target_cases`, and it may not be combined with `per_case`); omitting both resolves against the suite-level diagnostics object. `case:` is how a **cross-case reduction** is compared — the checks module accumulates across cases and emits the derived metric (`convergence.order_n032_to_n064`, a symmetry residual) as a per-case metric of the case where it first becomes computable (`docs/workflow/CHECKS_MODULE_CONTRACT.md` §2/§3), and the predicate reads it there. `na_allowed: true` treats an absent metric as satisfied (a "not applied" check). **Not degenerate:** a pass set whose EVERY `expected_outcome: pass` predicate asserts only `verdict.*` is deterministically REJECTED at `Compile.static` (it collapses the per-test judgment to the runner's `verdict.overall`). Give each pass test the concrete conditions its `tests.md` prose states (a `checks.<id>.status`, or a pinned `diagnostics_contract.metrics` address; nx-dependent thresholds as a `per_case` value map, cross-case reductions as a `case:`-scoped condition). An `xfail` predicate is exempt.

  **A multi-target test is normal, not exceptional.** A convergence sweep lists all its resolutions in `target_cases`, an equivariance test both members of the pair. `target_cases` is also the evidence contract: the runner records one `raw/metrics_basis.json` entry per (`test_id`, target `case_id`) pair and `post_execute` pins that matrix, so list every case the judgment reads and no case it does not. An ordered op (`le/ge/lt/gt`) needs a numeric threshold written as a YAML float **with a decimal point** — `1.0e-10`, never `1e-10` (which YAML parses as a string and the gate rejects). A `verdict.*` ref requires `diagnostics_contract.verdict.required=true` with that field listed. The runner emits every numeric judgment already reduced to a field, so predicates never do arithmetic. `xfail_condition` is a case-construction fact — encode it in `case.test_case_set`, never as a runtime predicate. `Validate.execute` evaluates these against `diagnostics.json` to author `verdict.json` deterministically; the judge no longer authors it. Canonical schema + invariants: V3 of `docs/workflow/phases/phase_01_compile.md`. **R1/M3c-β harness fold alignment:** on a harness-backed node the `diagnostics.json` fold is done by the harness — a per-case `verdict.overall == fail` iff any of that case's checks is `fail`; the top-level `verdict.overall`/`failed_checks` fold excludes an `expected_xfail` case's failure. Author `verdict.*` / `checks.*` refs consistent with that (an `xfail` test's `pass_when` targets the per-case guard failing — `verdict.overall == fail` + `verdict.failed_checks includes <guard>` with `per_case: true`); a metric-addressed predicate uses `per_case: true` (metric leaves are per-case). See `docs/workflow/CHECKS_MODULE_CONTRACT.md` §2 for the fold semantics.
- **`public_api` (R1) — `infrastructure` nodes ONLY** (`spec_kind: infrastructure`; OMIT for component/profile/problem). Author `public_api.published_operations[]` (`{operation_id, exercised_by?}`) + `public_api.published_types[]` enumerating the COMPLETE surface `controlled_spec §5` declares ("the published `operation_id`s are exactly: ..."). Include EVERY operation — helper emitters/writers that no test exercises as its primary op too (a test-driven subset is the R1 failure mode: Generate then never publishes them and the runner reimplements them). The `--stage compile` gate pins these sets == §5, so a dropped/extra entry is a Compile fail. Also author `public_api.signatures` — one `{symbol, signature}` per published op AND type, transcribing the controlled_spec §5's `### 5.1` fenced **structured** (language-neutral YAML) interface block: `signature` is that symbol's structured signature MAPPING (a procedure `{kind, name, args, result?}` or a type `{name, components}`) copied UNCHANGED from §5.1 — not re-encoded as Fortran text. `Generate.generate` cannot read controlled_spec (phase_02 §2-1), so `public_api.signatures` is the leaf's only carrier of the signatures it must publish; the gate pins `signatures` == §5.1 (rendered to the target language and normalized) so a transcription drift is a Compile fail. Canonical: V8 of `docs/workflow/phases/phase_01_compile.md`.
- The validator gates this substep can launch use the "substep ↔ allowed validator gate correspondence table" of `docs/workflow/LAUNCH_PROMPT_REFERENCE.md` as the canonical source (mixing an out-of-responsibility gate into the launch prompt fires `noncanonical_phase_write_attempt`).
- A `node` that is `problem` and whose `spec_id` includes `2d` or `3d` requires holding the 4 contract keys `state_variables` (each entry with `name` + `shape_expr`), `required_update_paths`, `diagnostics_from_state=true`, and `fallback_policy=fail_closed` as **direct children of `algorithm`**. `state_variables[].shape_expr` is limited to the 3 forms of `spec/schema/ir/shape_expr.schema.json`. `required_update_paths` is a **list of non-empty strings**, each one a `state_variables[].name` (`["h", "hu", "hv"]`); the object form (`[{target: h, path: [...]}]`) is forbidden — `ordering` / `steps[].step_id` already carry the update order.
- **The 3 placements of the multi-dimensional contract, and why only one of them is safe.** `--stage compile` resolves the contract from the FIRST of these that exists, and validates only that one: (1) `algorithm.state_contract` — **any** mapping wins here, even an empty one; (2) `algorithm.update_semantics` — wins if it holds **any** of the 4 contract keys; (3) the direct children of `algorithm`. So **author the direct children and nothing else**: never write an `algorithm.state_contract` key at all, and never let one of the 4 contract keys stray into `algorithm.update_semantics` (keep that to its own vocabulary — `target_variables` / `update_order` / …). Either mistake makes the gate resolve the contract somewhere else and **never read** your correct direct children, which then fail as if absent. The gate names every finding `state_contract.<field>` after the RESOLVED contract, not after where you authored it: a `state_contract.required_update_paths` violation refers to `algorithm.required_update_paths` in the form above. Reference form: `docs/examples/spec_ir_algorithm_2d_problem_contract.example.yaml`.
- An `io_contract.outputs[]` entry whose `evidence_ref` references `state_snapshots` must resolve to a **single** snapshot shape: every `raw_variables` token shares one `shape_expr` among the `state_snapshots` `schema.variables` + `time_variable`, and the entry's own `shape_expr` equals it. Mixing the grid variables with the scalar `time_variable` in one output is a `Compile fail` — author one output per resolved shape.
- The `Compile` of an upper `node` confirms the `ir_ref` and `ir_meta.json.verification_status` of the immediate dependency `node`, and must not start when `direct dependency compile readiness` is not satisfied.
- When the above generation contract cannot be derived, it is a `Compile fail`, and it must not proceed to `Generate` with an incomplete IR.
- Unregistered dependencies, unimplemented dependencies, and compatibility-violating dependencies are made resolution errors, and the relevant `node` is `blocked`.
- The validator invocation before `Compile` completes defaults to `run-gate`. `check_artifact_syntax.py` and `validate_workspace_root.py` are permitted to run directly only for a read-only check and a gate-independent check.
- Before the `Compile.generate substep` completes, run `python3 tools/check_artifact_syntax.py --expect-top object`, and it must be confirmed that `spec.ir.yaml` is a mapping / object restorable by a standard parser.
- The authoritative deterministic IR gates (`validate_workspace_root` + `check_artifact_syntax` + `--stage compile`) run AFTER this substep in the conductor's `Compile.static` substep (no leaf); a violation routes back here as a warm-resume reuse repair. Your optional self-checks above are a convenience, not the gate.

## Operations Rules
0. Every artifact — `.json` (`ir_meta.json` etc.), `.yaml`, `.md` — is written directly with the `Edit` / `Write` tool, to a path enumerated in the `allowed_file_tool_paths` of `output_manifests/<agent_run_id>.json` (managed JSON is now direct-write eligible; see `docs/AGENT_CONTRACT.md`). An inline write via Bash variable assignment, heredoc redirection, or `python3 -c` is blocked by `output_manifest_write_guard` / `forbid_python_inline_write`. Before writing, confirm `allowed_output_paths`, and use only a project-root-relative path starting with `workspace/`. Always specify every output path including `ir_meta.json` in the `workspace/ir/<node_key_safe>/<ir_id>/...` form. A path missing the `workspace/` prefix such as `ir/` is blocked by `output_manifest_write_guard` and recorded as an `unauthorized_write_violation`.
1. Issue `ir_id` in the `<slug>_<date>_<seq3>` form. `slug` is a short readable token derived from `spec_id`, `date` is `YYYYMMDD`, and `seq3` is a 3-digit serial number within the same day.
2. Fix the output destination to `workspace/ir/<node_key_safe>/<ir_id>/`.
3. The storage root for workflow artifacts allows only `workspace/`, and the workflow-root judgment targets only `workspace/`.
4. When `workspace/` does not exist before workflow execution starts, create `workspace/` directly under the repository root.
5. Before start and before completion, `python3 tools/validate_workspace_root.py` may be run. In an execution environment where `run-gate --gate validate_workspace_root` is available, prefer `run-gate`, and on `fail` it is a `Compile fail`.
6. Record `attempt_count`, `verification_status`, `last_fail_reason`, `debug_mode`, and `context_isolated` in `ir_meta.json`. `last_fail_reason` is a **single plain JSON string** or `null` — never an object or an array.
7. When `context_isolated=false`, record `ir_meta.json.constraint_reason` as required.
8. With `debug_mode=false`, do not save failed-attempt artifacts.
9. Before completion, `python3 tools/check_artifact_syntax.py --expect-top object` may be run on `spec.ir.yaml`. In an execution environment where the equivalent check via `run-gate` is available, prefer `run-gate`, and on `fail` it is a `Compile fail`.
10. When re-submitted in response to a retry from `Validate` (when `launches/<agent_run_id>.request.json#repair_reason` includes `validate_feedback`), follow the "Acceptance of retry from Validate" section of `docs/workflow/phases/phase_01_compile.md`, and record the fixed `spec.ir.yaml` sections in `ir_meta.json.repair_target_sections[]`.

## Decision Criteria
- When regenerated from the same input, all sections of `spec.ir.yaml` match.
- `dependency.topo_level` does not include a cyclic dependency.
- The output is consistent with the contracts of `docs/workflow/WORKFLOW_CORE.md`, `docs/workflow/phases/phase_01_compile.md`, and `docs/RUNBOOK.md`.
