# Workflow Generate Generate

## Purpose
Fix the generation responsibility of the Generate stage, and create a `Build`-able implementation artifact.

## Scope
- the work of generating implementation code into `workspace/pipelines/<pipeline_id>/source/<source_id>/src/`
- the work of generating `source_meta.json`

## Requirements
- The validator gates this substep can launch use the "substep ↔ allowed validator gate correspondence table" of `docs/workflow/LAUNCH_PROMPT_REFERENCE.md` as the canonical source.
- Immediately after starting work, read the `spec.ir.yaml` of `ir_ref`, enumerate for all elements of `io_contract.raw_requirements.required_evidence` the `artifact`, `required`, `min_samples`, and (when `artifact=state_snapshots` and `required=true`) `schema.variables[].name` and `time_variable`, reconcile them with the `runner`'s `raw/` output design and the index design of `raw/metrics_basis.json`, then start the implementation. When `state_snapshots` is required, design the `runner` to write **one snapshot per case** at `raw/state_snapshots/<case_id>.json`, keyed on the `case_id` it receives on argv (`--cases <spec.ir.yaml> <case_id>...`) — see the runner-output bullet below.
- The input is `spec.ir.yaml` (the 5 sections `case` / `algorithm` / `impl_defaults` / `io_contract` / `dependency`).
- Do not take `controlled_spec.md` as direct input. The requirement definition of the operation composition must be interpreted from `spec.ir.yaml.algorithm`.
- The implementation code separates `model` and `runner`, and the `runner` uses the `model` via `call` / `use` / `import`.
- Do not duplicate the physics-update logic in the `runner`. (Scope: for an `infrastructure` harness node — `spec_kind: infrastructure` — there is NO physics; the `model` publishes the runner-plumbing operations and the self-test `runner` legitimately CALLS them, so "do not duplicate" here means the runner must not reimplement an operation the `model` publishes — it must `call` it.)
- When `spec.ir.yaml.impl_defaults.toolchain.language` is a `fortran` / `c` / `cpp` / `mixed` family, an external interpreter such as `python` / `bash` / `sh` / `node` must not be launched from the `runner`.
- The `model` implements the operation contract of the target `node`, and a fixed-value-return-only, fixed-`JSON`-output-only, or `no-op`-only implementation is forbidden.
- A `node` that is `problem` and whose `spec_id` includes `2d` or `3d` reads the state-update contract of `spec.ir.yaml.algorithm` as input, and uses `state_variables` for the update computation as required.
- A `node` that is `problem` and whose `spec_id` includes `2d` or `3d` must not generate multiple `diagnostics` metrics with only `case_id` branching and scalar constant assignment.
- When the state-update contract of `spec.ir.yaml.algorithm` is missing, or `state_variables` and `required_update_paths` are missing, it is a `Generate fail`, and generation must not be continued by guessed completion.
- An implementation composition satisfying the `steps[]`, `ordering`, `control_condition`, and `iteration_contract` of `spec.ir.yaml.algorithm` must be generated.
- Confirm the `ir_ref`, `pipeline_ref`, and `aggregate_verdict` of the immediate dependency `node`, and when `direct dependency execution readiness` is not satisfied, generation must not start.
- A `node` that has dependencies requires an implementation that calls the published `operation` of the dependency `node` resolved by `spec.ir.yaml.dependency.direct_deps`.
- A function equivalent to a dependency `operation` must not be re-implemented in the depending `node`'s `model` / `runner`.
- A `model` / `runner` / `module` / `subroutine` / `Makefile` fragment of a dependency `node` must not be copied, relocated, or redefined into the depending `src/`.
- The `model` of a `node` that has a dependency `component` with `impl_defaults.toolchain.language=fortran` requires `use <spec_id>_model` and `call <spec_id>__*` per dependency `spec_id`, and forbids redefining `subroutine <spec_id>__*`.
- Each `call <dep>__<op>(...)` must use the **exact positional argument order** of the dependency's published interface — do not guess or reorder. The conductor injects that order in the launch prompt under **"Published dependency operations"** in `<dependency_facts>`, resolved from the dependency's certified source (the same source Build compiles + links). Match your local actual-argument expressions to that order/role position-by-position. Fortran arguments are positional: a swapped order (e.g. `(n,x,y)` for a `(x,n,y)` interface) compiles your node against a type/rank mismatch and **fails Build**. If `<dependency_facts>` carries no published-operation line for a called op (a dependency not yet certified), do not invent an order — stop with `blocked` (the dependency closure must be built first, e.g. `--with-deps`).
- Under each published op's header, `<dependency_facts>` also lists **every dummy argument's declared type, intent, and rank/shape** (e.g. `U: real(dp), intent(inout), rank-2 (:,:)`). The actual argument you pass must **match that rank and shape**, not only the order. A common trap: a dependency operates on a **single field** (a rank-2 `(:,:)` dummy) while your node holds a **multi-component state** (rank-3 `(ncomp,:,:)`). Do **not** pass the whole rank-3 array — that is a rank mismatch and **fails Build**. Instead **loop over the component index and pass a rank-2 slice** each call, e.g. `do c = 1, ncomp; call <dep>__<op>(u(c,:,:), nx, ny, ...); end do`. If a dummy's rank line reads "rank/shape not resolved", its rank could not be parsed host-side; pass that argument consistent with the operation's role (the header shows the positional order) and rely on Build to verify — do **not** try to open the dependency's source, it is outside your read scope.
- When the launch prompt carries a **`Certified exemplar`** block (R5 — conductor-injected prior art from a previously-certified SIBLING node in the same family/kind), use it as a **structural reference** for the cross-node-identical runner plumbing (argv/`--cases` parsing, the case loop, the JSON/snapshot/perf/`metrics_basis` emission conventions) so you reproduce those conventions correctly on the first attempt instead of re-deriving them. It is PRIOR ART, **not this node's spec**: author THIS node's physics kernel and per-test check logic from ITS `controlled_spec.md` / `tests.md` / `io_contract` — do **not** copy the exemplar's physics, checks, or case set. Using the injected block is permitted (it is host-selected, already in your prompt); do **not** go and `Read` other nodes' sources yourself (outside your read scope). When no exemplar block is present, proceed normally.
- In a configuration where the dependency is a `profile` with no published `operation`, the depending `problem` must be implemented to reference the `profile`'s selection result and constraints.
- The `runner` outputs only `diagnostics.json`, `perf.json`, and the `raw/` primary evidence, and must not write `verdict.json`, `aggregate_verdict.json`, `summary.json`, or `trial_meta.json`.
- The `perf.json` the `runner` outputs must have, without omission, the required fields of [docs/workflow/RUNNER_OUTPUT_CONTRACT.md](../../docs/workflow/RUNNER_OUTPUT_CONTRACT.md) §2 (your must-read): `case_id`, `target`, `walltime_sec`, `steps`, `cells_updated`, `throughput_cells_per_sec` (`= cells_updated / walltime_sec`), `parallelism` (object: `mpi_ranks` / `threads_per_rank` / `gpu_devices` / `parallel_degree_total`). A custom minimal schema like `{case_count, wall_seconds}` is rejected by the `post_execute` gate of `Validate.execute` (which mandatorily verifies `walltime_sec` / `throughput_cells_per_sec` / `parallelism`).
- The `raw/metrics_basis.json` the `runner` outputs must have a **`per_test` list (or `tests` object)** targeting all `test_id` of `io_contract.test_evidence_requirements`. Each entry has the `test_id` and the primary evidence holding the `required_raw_variables` of that `test_id` without omission. A custom structure without a per-test index like `evidence[]` is rejected by the `post_execute` gate (`must contain per_test list or tests object`). `metrics_basis.json` holds only the primary evidence and forbids copying `diagnostics.json`.
- When `state_snapshots` is required, the `runner` writes **exactly one** snapshot per case to `raw/state_snapshots/<case_id>.json`, with `<case_id>` taken from the argv it receives (`--cases <spec.ir.yaml> <case_id>...`, one positional per `case.test_case_set[]`). Build the path by concatenating the received `case_id` (`trim(case_id)//'.json'`), never a hardcoded literal — a sequential/combined name like `snapshot_0001.json` is flagged by `post_generate` static analysis (for a string-literal name) or, for a runtime-built name, fails the `Validate.execute` deliverable gate (which requires every `raw/state_snapshots/<case_id>.json`). Canonical: [docs/workflow/RUNNER_OUTPUT_CONTRACT.md](../../docs/workflow/RUNNER_OUTPUT_CONTRACT.md) §3 (runner output; your must-read) / `phase_04_validate.md` §43 (deliverable gate).
- The `diagnostics.json` the `runner` outputs must satisfy `spec.ir.yaml.io_contract.diagnostics_contract`: a `checks.<id>` entry for every `diagnostics_contract.checks[].id`, and — when `diagnostics_contract.verdict.required=true` — a top-level `verdict` object holding the `diagnostics_contract.verdict.fields` keys (e.g. `verdict.overall` / `verdict.failed_checks`). This encodes the `tests.md §3` diagnostics contract for the runner to emit (the IR is the only input). A custom per-case array that omits the contracted `checks.*`/`verdict` is a `Generate fail`. Note: emitting a `verdict` **field name inside `diagnostics.json`** is permitted; the rule below that forbids the strings `verdict.json` / `aggregate_verdict.json` / `summary.json` / `trial_meta.json` in the runner source targets those judge-artifact *filenames*, not a diagnostics field name.
- The `runner` source (conventionally `*_runner.f90` in `Fortran`) must not contain the above 4 file names as a substring of the **whole source including comments**. The substring check does not exclude comments.
- The `diagnostics.json` and `perf.json` the `runner` outputs must be a UTF-8 `JSON object` restorable by a standard `JSON` parser. With `impl_defaults.toolchain.language=fortran`, the `F0` / `F0.d` numeric descriptor must not be used for a `JSON` numeric token. The `post_generate` check is **descriptor-syntactic**: it flags the mere presence of `F0`/`F0.d` (and the `L`-family logical descriptor) in a runner `JSON` write spec, so a runtime leading-zero fixup does **not** pass — the descriptor must not appear. Use instead a scientific `ES`/`EN` descriptor (e.g. `ES24.16E3` — width 24 fits a leading sign, so negatives do not overflow to `***`) or an explicit-width `Fw.d` with `trim(adjustl(...))` for reals, `I0` for integers, and the literals `true` / `false` (branch on the logical; never an `L` descriptor) for booleans (the same applies to `raw/metrics_basis.json` and `raw/state_snapshots/*.json`). See [docs/workflow/RUNNER_OUTPUT_CONTRACT.md](../../docs/workflow/RUNNER_OUTPUT_CONTRACT.md) §4 (your must-read) for the canonical `jnum()` / `jbool()` reference snippet.
- With `impl_defaults.target.class=cpu` and the parallelization method unspecified, apply `OpenMP` to parallelizable loops by default.
- The generated artifact has a composition consistent with the target `node_key`.
- The generated artifact holds per-`node_key` differences, and the whole `src` must not be copied without an explicit common library.
- With `impl_defaults.toolchain.language=fortran`, match the `module` name and the source file name, and output in the `<module_name>.f90` form.
- With `impl_defaults.toolchain.language=fortran`, name the node's own `module` as the **literal** `<spec_id>_model` (prefix the `spec_id` verbatim; do **not** abbreviate or derive a short form) and the published `subroutine`s as `<spec_id>__<op>`. The deterministic `post_generate` validator accepts only `<spec_id>_model.f90` and rejects abbreviations, and a depending node resolves it via `use <spec_id>_model`, so the literal prefix is required (not just for collision avoidance). **Every Fortran identifier must be ≤ 63 characters (the f2008 name limit)** — an over-limit name is a `compile_error` that surfaces only at Build, forcing a regenerate→rebuild retry (`post_generate` now also flags it). If `<spec_id>_model` / `<spec_id>__<op>` would exceed 63 chars, do **not** abbreviate at your discretion — stop and treat it as a `spec`-level problem (the `spec_id` is too long). Canonical: `docs/workflow/phases/phase_02_generate.md`.
- **`src/Makefile` authorship:** for any node with `build_system=make` ∧ `language=fortran` — **leaf or with dependencies** — the conductor authors `src/Makefile` host-side (for a dependency node this includes the closure's object rules, with each `<dep>_model.f90` staged into `$(OBJDIR)` at Build; the file is NOT in your `allowed_output_paths`); do **not** create or edit it. The Makefile bullets below apply to **non-fortran make families** that still author their own Makefile, and document what the conductor's authored Makefile satisfies.
- With `impl_defaults.toolchain.build_system=make` and `impl_defaults.toolchain.language=fortran`, the `src/Makefile` makes explicit the object dependencies corresponding to the `use` dependencies, and requires writing the dependency `module`'s `.mod` or dependency `.o` into the prerequisites of each object target. Write both the targets and prerequisites of the object rules in a form resolvable to a literal `.o`/`.mod` basename. **For the validity of an out-of-source build (`$(OBJDIR)` override), when the build-rule target of a dependency object is prefixed with `$(OBJDIR)/` like `$(OBJDIR)/foo.o:`, write the consuming prerequisite also with the same `$(OBJDIR)/` prefix (`$(OBJDIR)/foo.o` / `$(OBJDIR)/foo.mod`, or a variable that expands to `$(OBJDIR)/`). A bare basename (`foo.o` / `foo.mod` without a prefix) works in-source (`OBJDIR=.`), but when `OBJDIR` is overridden, it becomes a separate target with no build rule and `make -j` stops with `No rule to make target`** (the `$(OBJDIR)/` prefix is not cosmetic. The post_generate static analysis detects this prefix mismatch and fails it). When writing all rules bare for in-source only, align both targets and prerequisites as bare (e.g. `foo.o:`). **When writing a prerequisite with a variable** (e.g. defining `MODEL_OBJ = $(OBJDIR)/foo_model.o` and referencing it like `$(RUNNER_OBJ): foo_runner.f90 $(MODEL_OBJ)`), it is also allowed because the static analysis expands a simple variable definition (`=` / `:=` / `?=` / `+=`) within the same Makefile and resolves it including the `$(OBJDIR)/` prefix. However, define the variable **before the relevant rule** (because make immediately expands the rule's prerequisites at read time, a forward reference defined later expands empty and is treated as a missing dependency). Also, using an undefined variable with no definition in the same Makefile (which cannot be expanded and leaves no `.o`/`.mod` basename) in a prerequisite or target is treated as a missing dependency.
- With `impl_defaults.toolchain.build_system=make` and `impl_defaults.toolchain.language=fortran`, the `src/Makefile` must not cause an indeterminate failure due to a missing dependency in `make -j`.
- **Out-of-source output contract (`build_system=make`):** the `src/Makefile` parameterizes `OBJDIR ?= .` / `BINDIR ?= .` / `RUNDIR ?= .` **and `BIN ?= <spec_id>_runner`** with `?=` defaults, and must not fix the build artifacts into `src/`. Object/module go under `$(OBJDIR)` (`gfortran` outputs `.mod` with `-J$(OBJDIR)` and references with `-I$(OBJDIR)`), and the linked execution binary is output to `$(BINDIR)/$(BIN)`. **`BIN` must use the overridable `?=` form (not `=`/`:=`/`+=`)**: Build imposes the canonical `<spec_id>_runner` binary name on the make command line and `Validate.execute` imposes the same name through the `make test` environment, which overrides a `?=` assignment only (a hard `BIN =` desyncs `make test` from the binary Build produced and is a `post_generate` `fail`). The `?=` default value is free (the conductor overrides it), but `<spec_id>_runner` is recommended so a local plain `make` matches. With the `?=` defaults, a local plain `make` maintains the conventional in-source behavior. The premise is that the generated `src/` is kept pristine (`Build` passes overrides from `compile_project` to put the object in a per-run tmp and the exe in `binary/<binary_id>/bin/`. For details, [phase_03_build.md](../../docs/workflow/phases/phase_03_build.md)).
- The `compile rule` guarantees the existence of `$(OBJDIR)` / `$(BINDIR)` with an order-only prerequisite (e.g. `| $(OBJDIR)`) + a `mkdir -p` rule, maintaining `make -j` safety.
- The preset-compatible quality path needed for `quality check` execution must be established by the `Generate` output alone. The premise of generating additional `test` source, harness, auxiliary `script`, or a temporary `Makefile` in `Validate.execute` is forbidden.
- With `impl_defaults.toolchain.build_system=make` and `impl_defaults.toolchain.language=fortran` / `c` / `cpp` / `mixed` families, the `src/Makefile` requires defining a `test` or `check` target usable with `run_quality_checks preset=make_test` or `preset=make_check`. The `test`/`check` target references the existing binary in `$(BINDIR)` (in `Validate.execute`, `binary/` is read-only-bound, so **do not relink**: place a **non-relinking** fail-closed guard `test -x $(BINDIR)/$(BIN) || { echo "error: $(BINDIR)/$(BIN) not built; run 'make all' first" >&2; exit 1; }` and **never** invoke `$(MAKE)` in the guard — a relink writes into the read-only `binary/` outside the execute `runs/` write_root and turns a binary-name/availability mismatch into an `unauthorized_write_violation` → `fail_closed`, whereas the fail-closed guard yields a clean retryable failure), guarantees the output destination with `mkdir -p $(RUNDIR)` (and the `raw/` required subdirectories the runner outputs, e.g. `mkdir -p $(RUNDIR)/raw/state_snapshots`) **before** `cd $(RUNDIR)`, then `cd $(RUNDIR)` and runs the runner as `$(BINDIR)/$(BIN) --cases $(SPEC) $(CASES)`, confining its relative output (`raw/` / `diagnostics.json` / `perf.json`) under `$(RUNDIR)`. The `--cases $(SPEC) $(CASES)` argv is **required** — the runner consumes its case set from `--cases` (and aborts without it), and `Validate.execute` imposes the same `SPEC`/`CASES` `run_program` uses so the make-test re-run's diagnostics match; declare `SPEC ?=` / `CASES ?=` (`?=` overridable, e.g. `SPEC ?= spec.ir.yaml` / `CASES ?= <all case_ids>` so a local `make all test` runs standalone). A bare `$(BINDIR)/$(BIN)` (no `--cases`) makes the make-test candidate emit no `diagnostics.json` and fails `quality_check`; `post_generate` flags a `test`/`check` recipe that runs the binary without `--cases`. Because `$(RUNDIR)` points to a run node dir not yet created at `Validate.execute` time, omitting `mkdir -p` makes `cd $(RUNDIR)` fail (`$(RUNDIR)` is RW-bound under the `runs/` write_root, so `mkdir` is authorized). With the `?=` default `.`, a local `make all test` maintains the in-source behavior. The runner writes the output path **relatively**, making it redirectable by `cd $(RUNDIR)`.
- Record `attempt_count`, `verification_status`, `last_fail_reason`, and `debug_mode` in `source_meta.json`.
- **`static lint` is NOT yours to run.** Lint is the deterministic `generate.lint` substep the conductor runs in-process (`tools/workflow_conductor.py:_lint_inproc`) AFTER `Generate.generate` produces `source/<source_id>/src/`, using the MCP `run_linter` `preset` for `impl_defaults.toolchain.language`. Do **not** call `run_linter` yourself, and do not record a `lint_command_ref`. The conductor's deterministic `Generate.static` substep (after `lint`, before `verify`) runs `post_generate`, which certifies the conductor-run lint against host-authored evidence (`<pipeline_root>/lint_evidence/<source_id>.json`), which you cannot write. On a lint finding the conductor re-runs `Generate.generate` (warm resume) so you fix the flagged source.
- **Write source that passes `static lint` on the first attempt (avoid a lint-finding regenerate).** When `impl_defaults.toolchain.language=fortran`, generate source that already satisfies fortitude's default ruleset (line length, `use ... only:` / intrinsic-module modifier, named integer kinds, `case default`, the `! allow(C003)` workaround, …). The canonical idiom checklist is `docs/workflow/phases/phase_02_generate.md` §2-1 ("fortitude default-rule idioms") — follow it; getting these right up front avoids a regenerate cycle. For C003, the **only** correct form is `! allow(C003)` on its own line immediately **before** a plain `implicit none`; never put the directive on the same line as `implicit none`, and never write `implicit none (external)` (a `compile_error` under `-std=f2008` that the `Generate.static` gate rejects).
- The `verification_status` of `source_meta.json` presumes `fail_closed`, and unperformed verification or unjudgeable must not be made `pass`.
- The storage root for workflow artifacts allows only `workspace/`, and the workflow-root judgment targets only `workspace/`.
- **`lineage.json` is NOT yours to write.** It sits at the pipeline root (`workspace/pipelines/<node_key_safe>/<pipeline_id>/lineage.json`), which must stay non-writable to the sandboxed leaf, so the conductor authors it host-side (see `docs/WORKSPACE_LAYOUT.md`; `tools/workflow_conductor.py:_write_lineage`) before the conductor's `Generate.static` gate runs. Do **not** list it in `allowed_output_paths` or attempt to write it — it is not in your write_roots and a write would be blocked. The conductor populates the top-level stage-id keys (`pipeline_id` / `source_id` / `binary_id` / `run_id`) per `docs/workflow/WORKFLOW_CORE.md`; `Generate.static`'s `post_generate` still validates the file.

## Operations Rules
1. Issue a `source_id`, and fix the output destination to `workspace/pipelines/<pipeline_id>/source/<source_id>/`. The form of `source_id` is `src_<YYYYMMDD>_<seq3>` (e.g. `src_20260511_001`). canonical: `docs/workflow/phases/phase_02_generate.md`. A format violation is pre-rejected by `record-launch` with a `ValueError` (the generate step launch contract).
2. With `debug_mode=false`, do not create `attempts/`.
3. Only with `debug_mode=true`, save a failed attempt to `attempts/<attempt_id>/`.
4. Hand only the `verification_status=pass` artifact to `Build`.
5. When the output destination is not `workspace/`, it is a `Generate fail`.
6. When `workspace/` does not exist before workflow execution starts, create `workspace/` directly under the repository root.
7. Before start and before completion, run `python3 tools/validate_workspace_root.py`, and on `fail` it is a `Generate fail`.
8. After generating the source, the conductor runs the deterministic `generate.lint` then `generate.static` substeps before `Generate.verify`. Do not run `static lint` — `generate.lint` (between `Generate.generate` and `Generate.static`) appends the lint record to `<src>/command_log.jsonl` and writes the host-side lint evidence; `generate.static` then runs `post_generate` + `validate_workspace_root`. On a lint or static finding the conductor re-runs `Generate.generate` (warm resume) with the flagged source to fix.
9. When receiving a retry from a `Build` failure (when `launches/<agent_run_id>.request.json#repair_reason` quotes the `failure_category` / `failure_source_refs[]` / `failure_excerpt` of `binary_meta.json`), follow the retry-trigger section of `docs/workflow/phases/phase_03_build.md`, and fix only the source pointed to by `failure_source_refs[]` (except for `repair_strategy=restart`).

## Decision Criteria
- The responsibility separation of `model` and `runner` is maintained.
- The output file set matches the contracts of `docs/workflow/WORKFLOW_CORE.md`, `docs/workflow/phases/phase_02_generate.md`, and `docs/RUNBOOK.md`.
- The required items of `source_meta.json` are not missing.
