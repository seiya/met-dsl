# Workflow Generate Verify

## Purpose
Fix the verification responsibility of the Generate stage output, and reduce `Build` failures in advance.

## Scope
- the check of `workspace/pipelines/<pipeline_id>/source/<source_id>/src/`
- the update of `workspace/pipelines/<pipeline_id>/source/<source_id>/source_meta.json`

## Requirements
- **M3c branch (harness-backed physics node).** When `spec.ir.yaml.dependency.direct_deps` includes an `infrastructure/...` (runner-harness) node, the leaf authored **only** `<spec_id>_model.f90` + `<spec_id>_checks.f90`; the `<spec_id>_runner.f90` is **host-rendered** by the conductor (deterministic glue over the certified harness plumbing) and `src/Makefile` is host-authored. Verify the **model + checks** only: the checks module implements the fixed ABI ([docs/workflow/CHECKS_MODULE_CONTRACT.md](../../docs/workflow/CHECKS_MODULE_CONTRACT.md)), its getters return shape-valid values for every required snapshot variable even for an xfail/guard case, `checks_compute` reports honest per-case statuses (the xfail exclusion is the harness fold, not the checks'), and neither `model` nor `checks` `use`s the harness. **Do NOT flag the host-rendered `<spec_id>_runner.f90` on style / shape / reachability** — it is deterministic conductor output, not leaf-authored, so a "runner" finding against it is a permanent false regenerate loop. Where a bullet below reads "`runner`", read it as "the model + checks the leaf authored" on an M3c node.
- This substep launches **no** validator gate (per the canonical "substep ↔ allowed validator gate correspondence table" of `docs/workflow/LAUNCH_PROMPT_REFERENCE.md`, `generate.verify` maps to the empty set): the `validate_workspace_root` + `validate_pipeline_semantics --stage post_generate` gates run in the conductor's deterministic `Generate.static` substep before you. You run only the LLM semantic checks below.
- The canonical source for the judgment rules is limited to `docs/AGENT_CONTRACT.md`, `docs/workflow/RUNNER_OUTPUT_CONTRACT.md` (the runner-output contract you verify — a must-read on a non-M3c runner-authoring node; on an M3c node the runner is host-rendered so this doc is not injected and you verify model + checks only), `docs/workflow/CHECKS_MODULE_CONTRACT.md` (§1-4 are the checks-module ABI of an M3c node; §5 binds every leaf-authored Fortran source of any node), `docs/workflow/phases/phase_02_generate.md`, `docs/RUNBOOK.md`, `spec.ir.yaml` (the 5 sections `case` / `algorithm` / `impl_defaults` / `io_contract` / `dependency`), `controlled_spec.md` (verify-only — the original requirement, used as a fidelity cross-check; `Generate.generate` is forbidden from reading it but `Generate.verify` reads it to catch a requirement the `spec.ir.yaml` translation dropped or distorted, with `spec.ir.yaml` remaining the primary basis), and the checked output. (`docs/workflow/WORKFLOW_CORE.md` and `phase_02_generate.md` are no longer force-read but remain readable under `docs/` as canonical reference.) The implementation under `tools/`, verification `script`, test code, and validator code must not be read to extract requirements or judgment rules.
- Check that all `case_id` and all expanded `case` of the `test case set` described in `spec.ir.yaml.case` are reachable from the implementation path of the `runner` or `model`. An unimplemented `case`, an unreachable branch, and a fixed-`case_id`-limited implementation are a `fail`.
- Check that the runtime input of `spec.ir.yaml.case` is propagated to the `runner` and `model`. When the acceptance or recording path of at least `case_id`, the grid conditions, the time conditions, the initial-condition identifier, the boundary-condition identifier, the `profile`- or `component`-selection result, `test_profile_id`, and `test_profile_version` cannot be confirmed, it is a `fail`.
- Check that the per-allowed-selection-value difference implementation in `spec.ir.yaml.case` is not collapsed into a fixed default. An implementation that ignores case-dependent inputs such as `boundary`, `initial_profile`, `topography_profile`, `dt_rule`, `refinement`, or the `sweep` expansion result is a `fail`.
- Check that the `runner` is aggregated into `model` calls and does not duplicate the physics-update logic. (Scope: for an `infrastructure` harness node — `spec_kind: infrastructure` — there is no physics; verify that the self-test `runner` CALLS the plumbing operations the `model` publishes rather than reimplementing them, and do NOT fail the node for "lacking physics". The published *signatures* (arg name/type/rank/`intent`/`result`) are pinned == controlled_spec §5.1 by the deterministic `Generate.static` gate, so do not re-audit them here — focus on the self-test's semantic use of the published surface.)
- When `spec.ir.yaml.impl_defaults.toolchain.language` is a `fortran` / `c` / `cpp` / `mixed` family, check that there is no launch of an external interpreter such as `python` / `bash` / `sh` / `node` from the `runner`.
- Check that the `model` implements the operation contract of the target `node`. A fixed-value-return-only, fixed-`JSON`-output-only, or `no-op`-only implementation is a `fail`.
- Check that the `model` satisfies the `steps[]`, `ordering`, `control_condition`, and `iteration_contract` required by `spec.ir.yaml.algorithm`, and does not drop the necessary branch or common computation path for the per-`test case set` difference input of `spec.ir.yaml.case`.
- Check that the `model` and `runner` reflect, without omission, the `update_semantics`, `temporaries`, `derived_field_rules`, `invariants`, and `splitting_policy` of `spec.ir.yaml.algorithm`. A missing state-update target, an unimplemented derived-quantity computation, an update order that breaks an invariant that should be conserved, and a `splitting_policy` mismatch are a `fail`.
- Check that the `model` and `runner` do not introduce an additional operation, additional iteration, additional conditional branch, or additional dependency `operation` call not described in `spec.ir.yaml.algorithm`. An execution path not present in `spec.ir.yaml` is a `fail` (`controlled_spec.md` is a fidelity cross-check only and never authorizes an extra path). A load-bearing requirement `controlled_spec.md` carries that `spec.ir.yaml` dropped or distorted — so the generated source, even if faithful to the IR, cannot satisfy it — is a deterministic `fail` attributed to the `spec.ir.yaml` translation (a `Compile`-stage defect), never a silent pass; only a dropped/distorted **requirement** (not abstracted natural-language flavor) qualifies. A benign `associate (unused_<name> => <name>)` / `end associate` block placed immediately after the declarations is the sanctioned binding for an intentionally-unused dummy argument (canonical: `docs/workflow/CHECKS_MODULE_CONTRACT.md` §5) — it computes nothing and is **not** an additional operation; its absence on an inert dummy, not its presence, is the defect. The same carve-out applies to an **inert dependency call** (`docs/GLOSSARY.md`): when `spec.ir.yaml` declares no sink for a dependency `operation`'s results (typical for a `profile` node), the `use` + `call <dep>__<op>(...)` the dependency mandate requires is licensed by `spec.ir.yaml.dependency.direct_deps` without an `algorithm` step, so the call itself is **not** an additional dependency `operation` call — flagging it oscillates against the dependency-call check below. What is a `fail` is an invented purpose built around it (the inert-call bullet below).
- Check that the `model` satisfies the dependency `operation` and the data dependency of the output metrics (`semantic_dependency.required_sources` and `io_contract.outputs`) required by `spec.ir.yaml.io_contract`. A specific control structure such as a space-time loop must not be uniformly required.
- Check that the `runner`'s raw-evidence output design satisfies `spec.ir.yaml.io_contract.raw_requirements.required_evidence` and `test_evidence_requirements`. At minimum, `raw/metrics_basis.json` must be a `per_test` list indexed by (`test_id`, `case_id`) — one entry per `test_id` × each case named in that test's `io_contract.test_predicates[].target_cases`, each entry recording `test_id`, `case_id`, and that test's `required_raw_variables` for that case without omission. A design that emits one entry per `test_id` (dropping the extra cases of a multi-target test) is a `fail`.
- When `state_snapshots` is required, check that the `runner` writes its snapshot files as `raw/state_snapshots/<case_id>.json` — exactly one per `case.test_case_set[].case_id`, the filename built from the `case_id` received on argv (`--cases <spec.ir.yaml> <case_id>...`), not a hardcoded/sequential literal (`snapshot_0001.json`) or a single combined file. A wrong name fails `Validate.execute`'s per-`<case_id>.json` deliverable gate, so it is a `fail` here; the deterministic `post_generate` static analysis also flags a literal snapshot filename not built from the `case_id`. Canonical: [docs/workflow/RUNNER_OUTPUT_CONTRACT.md](../../docs/workflow/RUNNER_OUTPUT_CONTRACT.md) §3 (runner output) / `phase_04_validate.md` §43 (deliverable gate).
- When a design that collapses the primary evidence of multiple `test` into one summary, a design that overwrites the raw evidence with the value of the last-run `case`, or a design that reuses only the suite-level boolean of `diagnostics.json` as `Validate.judge` input is detected, it is a `fail`.
- When an output metric that is a constant output not depending on the `model` execution result, a fixed-`JSON` output, or a direct assignment of an analytic expression is detected, it is a `fail`.
- A `node` that is `problem` and whose `spec_id` includes `2d` or `3d` mandatorily checks the state-update contract of `spec.ir.yaml.algorithm`, and when a missing one or `fallback_policy!=fail_closed` is detected, it is a `fail`.
- For a `node` that is `problem` and whose `spec_id` includes `2d` or `3d`, the case where a `subroutine` that outputs multiple `intent(out)` metrics has no `state_variables` array reference is a `metric-only scalar kernel` and a `fail`.
- Check that the generated code matches the input/output contract of the target `node_key`.
- The case where the `ir_ref`, `pipeline_ref`, and `aggregate_verdict` of an immediate dependency `node` cannot be confirmed is a `dependency workflow missing` and a `fail`.
- Check that a `node` that has dependencies implements the published `operation` call of the dependency `node` resolved by `spec.ir.yaml.dependency.direct_deps`. On omission it is a `fail`.
- When a re-implementation of a function equivalent to a dependency `operation` is detected, it is a `fail`.
- The case where a `model` / `runner` / `module` / `subroutine` / `Makefile` fragment of a dependency `node` is copied, relocated, or redefined in the depending `src/` is a `dependency implementation encapsulation` violation and a `fail`.
- A `node` that has a dependency `component` with `impl_defaults.toolchain.language=fortran` mandatorily checks `use <spec_id>_model` and `call <spec_id>__*` in the `model` per dependency `spec_id`, and when a redefinition of `subroutine <spec_id>__*` is detected, it is a `fail`.
- Check that each emitted `call <dep>__<op>(...)` passes its positional arguments in the order and role of the conductor-resolved published-operation interface in `<dependency_facts>` ("Published dependency operations"). Match position-by-position: e.g. for `subroutine d__scale(x, n, y)` the call's 1st actual must be the input array, 2nd its length, 3rd the output. A swapped order (a different-rank/type actual at a position) is a `fail`. Also check **rank/shape**: each dummy's declared rank is listed under its header (e.g. `U: real(dp), intent(inout), rank-2 (:,:)`); the actual argument's rank must match. In particular, when a lower-rank dummy (rank-2 `(:,:)`) is called on a multi-component state, confirm the call passes a **rank-2 slice inside a component loop** (`u(c,:,:)`), not the whole rank-3 array — the latter is a rank mismatch. This is an LLM semantic check — no deterministic gate enforces argument order/rank (Fortran is positional; the consumer uses its own local names), so Build's compiler is the final backstop and a missed mismatch surfaces there as a Build fail routed back to `Generate`.
- **Inert dependency call check.** Classify each dependency `operation` by whether `spec.ir.yaml` declares a sink for its results (`docs/GLOSSARY.md`): a result corresponding to a variable in any `algorithm.steps[]` entry (`inputs` OR `outputs` — a result can flow straight from a step's `outputs` to an `io_contract` output with no intervening `steps[].inputs` reference), a `derived_field_rules` / `invariants` quantity, an `io_contract` output, or a `semantic_dependency.required_sources` variable makes the call **load-bearing** — check its results are bound to exactly those declared variables and flow as declared; an implementation that discards a dependency result the `algorithm` declares a sink for (an inert-authored load-bearing call) is a `fail`. When no such sink exists (typical for a `profile` node whose `algorithm` resolves request facts only), check that the mandated dependency calls stay **inert**: every actual argument is a local variable assigned before the call, and no call result reaches an `io_contract` output, a `diagnostics_contract` check input, or an `algorithm.invariants` quantity. A probe configuration, an invented tolerance or acceptance predicate, a validity/binding flag folded into an output, or a conditional branch on the call's results is an execution path absent from `spec.ir.yaml.algorithm` and a `fail` (`major`) — every declared output must remain a function of exactly its declared `algorithm` inputs. Conversely, do **not** flag the inert calls themselves (locally initialized work variables whose post-call values are unread) as dead code, an unused-result defect, or a style finding — they satisfy the dependency-call mandate above, and their **absence**, not their presence, is the defect.
- In a configuration where the dependency is a `profile` with no published `operation`, check that the depending `problem`'s implementation references the `profile`'s selection result and constraints. On omission it is a `fail`.
- When an implementation in which the `runner` writes `verdict.json`, `aggregate_verdict.json`, `summary.json`, or `trial_meta.json` is detected, it is a `fail`. As in `docs/workflow/phases/phase_02_generate.md` and `docs/RUNBOOK.md`, those forbidden names are a substring-match target against the **whole runner source including comments**, and even an identical string in an example inside a comment is a `fail`.
- Check that the `runner`'s `diagnostics.json` and `perf.json` output satisfy a UTF-8 `JSON object` restorable by a standard `JSON` parser. When a `impl_defaults.toolchain.language=fortran` `runner` uses a format that can omit the leading zero such as `F0.d` directly for a numeric token of `perf.json` or `diagnostics.json`, it is a `fail`.
- When a `impl_defaults.toolchain.language=fortran` `runner`, when outputting a logical value as a JSON boolean, uses not the literal `true` / `false` but an `L`-family edit descriptor (`L1` etc., which generates a `T`/`F` token) directly for the boolean token of `diagnostics.json` / `perf.json` / `raw/metrics_basis.json` / `raw/state_snapshots/*.json`, it is a `fail`. This check is enforced deterministically by `validate_pipeline_semantics --stage post_generate` with a static analysis of the runner source (reporting the descriptor, file, and line number).
- Check that the composition is consistent with the language and `build_system` of `spec.ir.yaml.impl_defaults`.
- Check that the `target.class`, `target.backend`, `target.architecture`, `toolchain.language`, `toolchain.standard`, `toolchain.build_system`, and `selected.backend_key` of `impl_defaults` are reflected in the generated source composition and the `build` artifacts. A language mismatch, a `build_system` mismatch, a code-path output for an unselected backend, and an unreflected `selected.backend_key` are a `fail`.
- Check that the execution-algorithm choices specified in `impl_defaults.abstract` and `impl_defaults.backend_overrides` — parallelization, layout, fusion, tiling, vectorization, asynchrony, etc. — are reflected in the generated code or `build` settings to the extent expressible in the target language and target. Ignoring of a specified knob, the mixing-in of an optimization for a forbidden target, and a violation of the default `OpenMP` rule for `target.class=cpu` are a `fail`.
- Check for an exact match with the `source/src` of a different `node_key`, and a copy without an explicit common library is a `copy_based_artifact_reuse` and a `fail`.
- With `impl_defaults.toolchain.language=fortran`, check that the `module` name and the source file name match in `<module_name>.f90`.
- With `impl_defaults.toolchain.language=fortran`, check that the node's own `module` name is the **literal** `<spec_id>_model` (file `<spec_id>_model.f90`) and the published `subroutine` names use the literal prefix `<spec_id>__<op>`. An abbreviated or derived short prefix is a `fail` (the deterministic `post_generate` validator accepts only the literal name and a depending node resolves it via `use <spec_id>_model`). Canonical: `docs/workflow/phases/phase_02_generate.md`.
- For any node with `build_system=make` ∧ `language=fortran` — **leaf or with dependencies** — the `src/Makefile` is conductor-authored (for a dependency node it includes the closure object rules); verify it as-is (the `post_generate` gate still checks it) — do **not** require the generate child to have produced it (it is not in the generate.generate `output_refs`). The Makefile checks below apply unchanged regardless of author.
- With `impl_defaults.toolchain.build_system=make` and `impl_defaults.toolchain.language=fortran`, check that each object target of the `src/Makefile` makes explicit the `.mod` or dependency `.o` prerequisites corresponding to the `use` dependencies. Confirm that the targets and prerequisites of the rule lines are resolvable to a literal `.o`/`.mod` basename (one of a literal name, `$(OBJDIR)/<basename>`, or a reference `$(VAR)` via a simple variable defined **before the relevant rule** in the same Makefile). **A forward reference that defines a variable after the relevant rule** is treated as a missing dependency because make expands the prerequisite empty. A target or prerequisite of only an undefined variable with no definition in the same Makefile is also treated as a missing dependency. For supplements, refer to 1-2-1 of `docs/RUNBOOK.md`.
- With `impl_defaults.toolchain.build_system=make` and `impl_defaults.toolchain.language=fortran`, check that the `src/Makefile` does not lack the `make -j`-compatible dependency description.
- With `impl_defaults.toolchain.build_system=make` and `impl_defaults.toolchain.language=fortran` / `c` / `cpp` / `mixed` families, check that the `src/Makefile` satisfies the out-of-source output contract (the `?=` parameterization of `OBJDIR ?= .` / `BINDIR ?= .` / `RUNDIR ?= .` **and `BIN ?= <spec_id>_runner`**, a composition that puts object/module under `$(OBJDIR)`, the linked binary in `$(BINDIR)/$(BIN)`, and the run output of `test`/`check` under `$(RUNDIR)`. For `fortran`, `.mod` output with `gfortran -J$(OBJDIR)`). A Makefile that fixes the build artifacts into `src/` (does not reference `OBJDIR`/`BINDIR`, or writes `.o`/`.mod`/exe directly under `src/`) is a `fail`. **`BIN` must use the overridable `?=` form** (a hard `BIN =`/`:=`/`+=` is a `fail`): Build and `Validate.execute` impose the canonical `<spec_id>_runner` name, and `Validate.execute`'s `make test` override travels through the environment, which overrides `?=` assignments only. Because `Build` passes those overrides from `compile_project`, a non-parameterized Makefile invites an `unauthorized_write_violation` → `fail_closed` in `Build` (the canonical source is [docs/workflow/phases/phase_02_generate.md](../../docs/workflow/phases/phase_02_generate.md) 2-1). Note that the object prerequisite of a `use` dependency may be any of a literal `.o`/`.mod` basename, `$(OBJDIR)/<basename>`, or a reference via a simple variable defined before the relevant rule (`$(MODEL_OBJ)` to `MODEL_OBJ = $(OBJDIR)/foo_model.o`, etc.) (the static analysis expands the variable and resolves it to a basename). A prerequisite written with only a forward reference (a later-defined variable) and an undefined variable is treated as a missing dependency.
- Check that the preset-compatible quality path needed for `quality check` execution is established by the `Generate` output alone. A composition that is established only by generating additional `test` source, harness, auxiliary `script`, or a temporary `Makefile` in `Validate.execute` is a `fail`.
- With `impl_defaults.toolchain.build_system=make` and `impl_defaults.toolchain.language=fortran` / `c` / `cpp` / `mixed` families, check that a `test` or `check` target exists in the `src/Makefile` and conforms to `run_quality_checks preset=make_test` or `preset=make_check`. Check that the target references the existing binary in `$(BINDIR)` with a **non-relinking** fail-closed guard `test -x $(BINDIR)/$(BIN) || { echo "error: $(BINDIR)/$(BIN) not built; run 'make all' first" >&2; exit 1; }` that **never** invokes `$(MAKE)` (a relink in the read-only `binary/` of `Validate.execute` is an `unauthorized_write_violation` → `fail_closed`), generates the output destination with `mkdir -p $(RUNDIR)` (and the `raw/` required subdirectories the runner writes) before `cd $(RUNDIR)`, and invokes the runner **with `--cases $(SPEC) $(CASES)`** (declaring `SPEC ?=` / `CASES ?=`; the runner aborts without `--cases` and `Validate.execute` injects the authoritative `SPEC`/`CASES`), confining the run output under `$(RUNDIR)`. A target that relinks the binary (recursive `$(MAKE)`/`make` or a compiler/linker invocation, in its recipe or via a build prerequisite), relinks/run-outputs directly under `src/`, runs the binary without `--cases`, or does `cd $(RUNDIR)` without generating `$(RUNDIR)`, is a `fail` (`post_generate` enforces the no-relink and `--cases` contracts via `validate_pipeline_semantics`).
- Check that the generated code does not introduce a reference to a dependency `node` not present in `spec.ir.yaml.dependency` or to an undeclared `operation`. A call outside `direct_deps`, an unresolved `component` reference, and an implementation choice contradicting a `profile` constraint are a `fail`.
- Check the required items of `source_meta.json`. Do **not** look for a `lint_command_ref` — it is no longer recorded by the leaf. `static lint` is the deterministic `Generate.lint` substep and the compiler syntax gate is the deterministic `Generate.syntax` substep, both conductor-run; the conductor's `Generate.static` substep runs `post_generate` (before you), which certifies them against the host-authored `<pipeline_root>/lint_evidence/<source_id>.json` / `<pipeline_root>/syntax_evidence/<source_id>.json` (which the leaf cannot write). You do not run those gates or inspect the evidence by hand. `static lint` is distinct from `quality check` (`run_quality_checks`).
- `Generate.verify` must not run the *write/refresh*-family tools of the `build-runtime` MCP including `run_linter` (`run_linter` / `run_syntax_check` / `compile_project` / `run_program` / `run_quality_checks`); see [docs/workflow/phases/phase_02_generate.md](../../docs/workflow/phases/phase_02_generate.md) 2-1. It launches **no** external validator gate (`validate_workspace_root` and `post_generate` run in the conductor's `Generate.static` substep). Verify's sole authorized output is `source_meta.json` (`allowed_output_paths` lists only it and the `bwrap` `write_root` is pinned to it), so a tool call that writes elsewhere — such as `run_linter`'s `command_log.jsonl` under `src/` — is refused by the sandbox (`EROFS`) and fails the leaf from inside; an `Edit`/`Write` to an unlisted path is likewise blocked by the `output_manifest_write_guard` hook.
- The workflow mode uses `METDSL_WORKFLOW_EXEC_MODE` as the canonical source, and applies `dev` when unset.
- A finding always sets `verification_status=fail` (record `issue_severity`); `minor` is **never tolerated**. The conductor warm-repairs `minor` (re-runs `Generate.generate`) and stops(`dev`)/escalates(`prod`) on `major|critical` — see `docs/AGENT_CONTRACT.md`.
- Check that `attempts/` does not exist when `debug_mode=false`.
- Check that the storage root of the checked artifact is `workspace/`, and the workflow-root judgment targets only `workspace/`.

## Operations Rules
1. Based on the check result, update the `verification_status` of `source_meta.json` by writing it directly with the `Edit` / `Write` tool (managed JSON is direct-write eligible; no `guarded-apply-patch` and no `apply_patch_writes` gate evidence is required — under `bwrap` confinement the write is authorized by `write_roots` containment). Even when the inspection finds nothing to change, re-author `source_meta.json` (e.g. refresh an idempotent field such as `verify_attempts`) so the substep produces its declared output. An inspect-only verify that writes nothing cannot terminate `pass`.
2. On `fail`, record the convention-violation content and the fix target in `last_fail_reason`. `last_fail_reason` is a **single plain JSON string** (or `null`); an object or an array is a contract violation. Fold multiple findings into one string. When originating from a `Validate` retry, record `code` in the `attribution_hint` of `source_meta.json` as required (routing consistency).
3. On `verification_status=fail`, request regenerate, and issue a new `source_id` with the same `ir_id`. The form of `source_id` is `src_<YYYYMMDD>_<seq3>` (e.g. `src_20260511_001`). canonical: `docs/workflow/phases/phase_02_generate.md`. The `<slug>_<YYYYMMDD>_<seq3>` form of `ir_id` must not be reused.
4. Start `Build` only on `verification_status=pass`.
5. When `workspace/` does not exist before workflow execution starts, create `workspace/` directly under the repository root.
6. Do **not** run `validate_workspace_root.py` or `validate_pipeline_semantics --stage post_generate` yourself: both run in the conductor's deterministic `Generate.static` substep **before** verify. A `Generate.static` failure warm-resumes `Generate.generate` and verify never starts, so verify is reached only on a source that already passed them.
7. When it `fail` in `dev` mode, record in `last_fail_reason` the basis needed to create `failure_analysis.json` (the violated convention, the target artifact, the failure reason), as a single string — not a JSON object or dict.

## Decision Criteria
- Make it `verification_status=pass` only when all check items are `pass`.
- The check result has a reproducible file reference.
- The judgment rules are consistent with `docs/AGENT_CONTRACT.md`, `docs/workflow/RUNNER_OUTPUT_CONTRACT.md`, `docs/workflow/phases/phase_02_generate.md`, and `docs/RUNBOOK.md`.
