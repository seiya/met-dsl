# Workflow Orchestration

## Purpose
On the target execution platform, run the whole workflow not as the single-thread processing of a parent `agent` but as the hierarchical launch of independent child `agent`. The core workflow is a 5-phase structure `Spec â†’ Compile â†’ Generate â†’ Build â†’ Validate` (`Tune` / `Promote` are optional flows separated from core).

## Scope
- issuing the `orchestration_id` at `workflow` start
- generating `preflight.json`
- generating the launch evidence of `step agent` / `substep agent`
- recording `agent_runs.jsonl` / `agent_graph.json` / `step_result.json`

## Requirements
- The `orchestration agent` must not directly generate phase artifacts.
- Each `step` that has no standard `substep` must be delegated to an independent `step agent` launched with `spawn_agent`.
- For a phase that has `substep` like `Compile` / `Generate` / `Validate`, the `orchestration agent` must directly launch the 2 substeps (`generate` and `verify` for `Compile`/`Generate`, `execute` and `judge` for `Validate`) as separate `substep agent` with `spawn_agent`.
- The `step` of `Build` must be completed with a single `step agent`.
- The optional flows `Tune` / `Promote` are not included in the core workflow. `Tune` is an optional flow that has substeps, and `Promote` is an optional flow of a step, launched from a separate entrypoint.
- The launchability confirmation of the execution platform and the evidence write-out must use `tools/orchestration_runtime.py` as the canonical-source implementation.
- The canonical entrypoint for starting the workflow is `python3 tools/run_workflow.py <spec_ref> <until_phase> [--llm <codex|cursor|claude>]`, and a manual `init` / manual `preflight` must not be used as the normal path. `<until_phase>` is one of `compile` / `generate` / `build` / `validate`.
- Resuming a failed workflow also uses `python3 tools/run_workflow.py --resume [--orchestration-id <id>]` as the canonical entrypoint, and a manual `init --resume-from-checkpoint` must not be used as the normal path. `--resume` restores `spec_ref` / `until_phase` / `--llm` / `--mode` from the existing artifacts of the target orchestration (the most recent in chronological order when `--orchestration-id` is omitted), and internally sets `resume_enabled=true`. When terminated with a terminal status, it resets the live status to `running` (to avoid the reject of a terminal â†’ other status transition). The skip judgment of a completed `step` follows this SKILL's Operations Rule 19 (`check-step-completed`). For details, [docs/RUNBOOK.md](../../docs/RUNBOOK.md) Â§3-1.
- The workflow mode uses `tools/run_workflow.py --mode <dev|prod>` as the canonical source, with a default of `dev`.
- In `dev` mode, operate the verify judgment strictly, and when `issue_severity=major|critical` is detected, it must stop with `fail`.
- When a fail occurs in `dev` mode, `workspace/orchestrations/<orchestration_id>/failure_analysis.json` must be saved as a required output recording the cause-investigation result.
- Making `preflight.json` `pass` by manual editing or post-editing is forbidden. `preflight` uses the execution result of `tools/orchestration_runtime.py preflight` as the canonical source.
- The preflight of `backend=codex` must simultaneously satisfy `checks.hooks_enabled.pass=true` and `checks.codex_home_writable.pass=true`. When not met, the workflow must not start.
- Just before launching a child `agent`, satisfying the live preflight gate is required, and when the live check is `fail`, `record-launch` must not be run.
- The initial loading before launch uses `references/startup_contract.md` as the primary reference, and only when the detailed contract is needed must it additionally reference `docs/workflow/WORKFLOW_CORE.md` and `docs/ORCHESTRATION.md`.
- Before starting each phase, whether the target phase requires a `substep agent` or a `step agent` must be judged by a fixed table. `Compile` / `Generate` / `Validate` are `substep agent`, and `Build` is `step agent`. In the optional flows, `Tune` is `substep agent` and `Promote` is `step agent`.
- Before starting the first phase, `python3 tools/orchestration_runtime.py workflow-launch-check --repo-root <repo_root> --orchestration-id <orchestration_id> --node-key <node_key> --step <step> --backend <backend> --require-child-agent <step|substep>` must be run.
- When `workflow-launch-check` returns `status=fail_closed`, run `python3 tools/orchestration_runtime.py set-status --repo-root <repo_root> --orchestration-id <orchestration_id> --status fail_closed --reason-code <reason_code>` to stop, and a phase artifact must not be generated. A re-call of `set-status` with the same terminal status is **idempotent**, and is accepted as a cleanup retry if the cleanup_committed marker is not written, or as a no-op replay (assuming a defensive retry after response loss) if the marker is written. The narrative fields (`reason_code` / `reason_detail` / `blocking_policy_scope`) are fixed on the first call and are not overwritten on a re-call. Append the failure content by directly editing `workspace/orchestrations/<orchestration_id>/failure_analysis.json`. The only permitted terminal-to-terminal transition is `fail` â†’ `fail_closed`; others are rejected by the runtime guard.
- In the first `commentary`, the target phase, the `SKILL` to use, the kind of `agent` to launch, and the place that uses `MCP` must be made explicit as an execution declaration. When the execution declaration and the actual work do not match, stop and redo from the declaration.
- The launch-request body of a `step agent` / `substep agent` must always be generated based on the corresponding template in `references/launch_prompts.md`. Any free-form prompt that does not use the template, the mixing of a different template, and the omission or renaming of required items are forbidden.
- In the child-`agent` launch-request body, it must be made explicit that the canonical source for the requirement definition and judgment rules is `docs/`, `spec/`, and the relevant trial's artifacts, and that the implementation under `tools/`, verification `script`, test code, and validator code must not be read to extract rules.
- `ir_ref`, `pipeline_ref`, and `dependency_ref` must be finalized to canonical paths at the time the launch request is generated. A placeholder such as `<agent-determined-...>` is forbidden.
- When launching the `generate` substep and `verify` substep of the `Compile` phase, `dependency_ref` must always be passed an existing path of the form `spec/<component_path>/deps.yaml`. The `verify` substep must also inherit the same `dependency_ref` as the `generate` substep. A value pointing to `workspace/ir/` is always wrong in the `Compile` phase. When `dependency_ref` is empty or a placeholder, it must not be launched. Use `run-gate --gate orchestration_read` for the existence confirmation of `deps.yaml`.
- The changes to phase artifacts permitted to a child `agent` must be limited to under the `write_root` permitted by the capability token. Changes under `ir_ref` / `pipeline_ref` are limited to a canonical path that passed `guarded-apply-patch` or the corresponding gate, and changes outside the permitted root are forbidden.
- When changing a phase artifact, a `step agent` / `substep agent` must branch the write path by the output path's extension. For `.json` and `.txt` output, the canonical invocation is `guarded-apply-patch`, and for other extensions (`.yaml` / `.yml` / `.md` / source code etc.), write directly with the `Edit` / `Write` tool only to a path enumerated in the `allowed_file_tool_paths` of `output_manifests/<agent_run_id>.json`. `apply_patch_writes` is an internal gate, and `run-gate --gate apply_patch_writes` and `apply-patch-gate` must not be used as a public path. The direct execution of normal `apply_patch`, shell redirection, `tee`, `sed -i`, `perl -0pi`, and file writes via `python` / `sh` / `bash` are forbidden.
- The connectivity check of `guarded-apply-patch` must be performed with a dry-run or no-op patch. A dummy file must not be newly created for the connectivity check.
- `record-launch` must generate `output_manifests/<agent_run_id>.json` and `read_manifests/<agent_run_id>.json`. The output manifest holds both `allowed_output_paths` (all permitted paths) and `allowed_file_tool_paths` (paths directly writable with `Edit` / `Write`), and `allowed_file_tool_paths` by default auto-includes the set of `allowed_output_paths` minus `.json` / `.txt`. `guarded-apply-patch` and `record-agent-run` reference the output manifest, and `orchestration-read` references the read manifest and must reject a path outside the manifest.
- The `orchestration agent` must not `Read` the following 4 kinds of internal artifact owned by a child agent (= an `agent_run_id` other than itself). These are blocked fail-closed by `read_manifest_read_guard`:
  - `launches/<child_arid>.prompt.txt` â€” the child prompt is passed as the input of the `Agent` tool at launch, so re-reading is unnecessary.
  - `capabilities/<child_arid>.json` â€” only the child reads its own capability immediately after launch.
  - `output_manifests/<child_arid>.json` / `read_manifests/<child_arid>.json` â€” same as above, only the child's self-reference is permitted.

  For the child's response content, read `agents/<child_arid>/dialogs/agent.result.json` / `agent.summary.txt` via the `agent_result_ref` / `agent_summary_ref` of `agent_runs.jsonl`. Any other workspace path goes via `python3 tools/orchestration_runtime.py run-gate --gate orchestration_read --agent-run-id <self_arid> --capability-token <capability_token> --args-json '{"read_path":"â€¦"}'`.
- The `orchestration agent` must not generate a UUID on its own. Its own `agent_run_id` is already recorded in `orchestration_meta.json` by `init_orchestration()` of `tools/run_workflow.py`, and when issuing a child `agent_run_id`, use `python3 tools/new_agent_run_id.py` as the canonical path. `python3 -c 'import uuid; â€¦'` is blocked fail-closed by `forbid_python_inline_write`, and `cat /proc/sys/kernel/random/uuid` and `uuidgen` are not used because they stop on a session-sandbox approval request every time (for details, see the `orchestration_agent_run_id` acquisition procedure of `references/startup_contract.md`).
- To **inspect** the status fields of `preflight.json` / `orchestration_meta.json`, use the `Read` tool (or `python3 tools/orchestration_runtime.py preflight-status`). `python3 -c 'import json; â€¦'` is blocked by `forbid_python_inline_write` even when it only reads, and `jq` / bootstrap Bash stop on session-sandbox approval. The files are small, so a `Read` is the canonical path (see `references/startup_contract.md` preflight section).
- A file write via shell is a forbidden target regardless of whether the target path is a phase artifact, and when a pre-gate or `record-agent-run` detects it, that gate or `record-agent-run` must reject that `agent_run`. After the reject, the `orchestration agent` must record `orchestration_meta.status=fail_closed` and stop.
- The launch-request body of a `step agent` / `substep agent` must make explicit the input contract, expected output, storage location, on-failure stop condition, and `spawn_agent` obligation.
- The launch-request body of a `step agent` / `substep agent` must record `skill_name`, `skill_ref`, and `skill_must_read_refs` as required, so that the child `agent` can read the target `SKILL` immediately after launch.
- In the launch request of `Compile verify`, the `spec.ir.yaml` under `ir_ref` must be recorded as required in `skill_must_read_refs`.
- In the launch request of `Generate verify`, the `spec.ir.yaml` under `ir_ref` must be recorded as required in `skill_must_read_refs`.
- In the launch request of `Validate execute` / `Validate judge`, the `spec.ir.yaml` under `ir_ref`, the `source/<source_id>/source_meta.json` under the target `pipeline_ref`, and `binary/<binary_id>/binary_meta.json` must be recorded as required in `skill_must_read_refs`. The resolution rule of `<binary_id>` and `<source_id>` differs per substep:
  - `Validate execute`: the launch request records `source_binary_id` (= `<binary_id>`) and `source_id` as required (runtime-enforced). The runtime also verifies the match `binary_meta.json.source_source_id == launch_request.source_id`.
  - `Validate judge`: the launch request requires only `run_id` (`source_id` / `source_binary_id` are not required). The judge always reads `trial_meta.json` under the same `run_id`, and resolves `trial_meta.json.source_source_id` as `<source_id>` and `trial_meta.json.source_binary_id` as `<binary_id>` (because trial_meta is written by execute and the runtime has verified its match with binary_meta, even if multiple sources / binaries coexist under `pipeline_id` due to retries, there is no path for the judge to read the wrong id).
- The prompt saved at `launch` recording must be the complete template that matches the required-field values of the request payload. A summary prompt or a simplified prompt that holds only markers is forbidden.
- The `launch response` saved by `record-launch` must be the complete actual response immediately after `spawn_agent` succeeds. Post-generation, fixed wording, or a substitute holding only a summary text is forbidden.
- The `launch response` records the child-`agent` identifier as required, and the `agent_session_id` of `record-agent-run` must match that identifier.
- Before launching the `Compile` of an upper `node`, confirm the `ir_ref` and `ir_meta.json.verification_status` of the immediate dependency `node`, and satisfying `direct dependency compile readiness` is required.
- Before launching the `Generate` / `Build` / `Validate` of an upper `node`, confirm the `ir_ref`, `pipeline_ref`, and latest `aggregate_verdict` of the immediate dependency `node`, and satisfying `direct dependency execution readiness` is required.
- To pass the dependency-readiness judgment of `workflow-launch-check`, record `direct_dependency_compile_readiness`, `direct_dependency_execution_readiness`, `detail.ir_ref_verified`, `detail.pipeline_ref_verified`, and `detail.aggregate_verdict_verified` in `orchestration_meta.json.dependency_readiness` as required. When not recorded or not met, `workflow-launch-check` returns `fail_closed`. A leaf node (the deps.yaml dependency is empty) is auto-set to trivial-true by the preflight. For a node with a non-trivial dependency, after generating and verifying the dependency artifacts (`workspace/ir/<dep>/.../ir_meta.json`, `workspace/pipelines/<dep>/.../binary_meta.json`, `workspace/pipelines/<dep>/.../aggregate_verdict.json`), run `python3 tools/orchestration_runtime.py mark-dependency-readiness --repo-root <repo_root> --orchestration-id <orchestration_id>`. The CLI is not an assertion but a runtime-verification request, and every time it inspects the **actual artifact selected by canonical id order (`(date, seq)` of `<slug>_<YYYYMMDD>_<seq3>`) on the workspace** (`verification_status=pass`, or `aggregate_verdict âˆˆ {pass, xfail}`) and the version_constraint resolution result, and recomputes/overwrites all flags (it does not trust mtime: Codex round 23 F1; a slug-difference collision of the same `(date, seq)` is fail-closed: Codex round 35 F1). The `dep_set_fingerprint` (the SHA-256 of spec_ref + deps.yaml) is also updated every time, and when a fingerprint mismatch is detected on a preflight re-run, the persisted flags are all reset. The flag stays false in any of artifact not generated / verification_statusâ‰ pass / version_constraint unresolvable / spec_ref replacement (gate-bypass prevention).
- When the immediate dependency `node` is incomplete, a substitute implementation that embeds the dependency's code into the upper `node`'s `src/` must not be instructed.
- Before directly editing a phase artifact or running `MCP`, the 3 conditions of preflight-done, launch-prompt-prepared, and child-`agent`-launched must be satisfied. When any is unmet, the editing and execution of the phase body must not start.
- When a path reservation is needed before launching a child `agent`, use `python3 tools/orchestration_runtime.py reserve-phase-root --repo-root <repo_root> --orchestration-id <orchestration_id> --node-key <node_key> --step <step> --reserved-id <id> --reserved-by-agent-run-id <agent_run_id>`, and the materialization of `workspace/ir/` and `workspace/pipelines/` is forbidden.
- Even for a provisional implementation aimed at workflow validity confirmation, verification, or a connectivity check, the parent `agent` must not proxy the body processing of a child-`agent`-required phase.
- For each child-`agent` launch, save the launch-request body to `launches/<agent_run_id>.prompt.txt` and the launch-reply body to `launches/<agent_run_id>.reply.txt`, and record the references in the `launch_prompt_ref` and `launch_reply_ref` of `agent_runs.jsonl`.
- On completion of each `step agent` / `substep agent`, save `agents/<agent_run_id>/dialogs/agent.result.json` and `agents/<agent_run_id>/dialogs/agent.summary.txt`, and record the references in the `agent_result_ref` and `agent_summary_ref` of `agent_runs.jsonl`.
- `agent.summary.txt` is an investigation log including the final `status` and the main `output_refs` or the failure cause, and must not end with only a single-line `pass` / `fail`.
- The child `agent` must run the stage-specific validator invocation that the responsible `SKILL.md` defines via `python3 tools/orchestration_runtime.py run-gate --gate validate_pipeline_semantics --agent-run-id <agent_run_id> --capability-token <capability_token> --args-json '<json>'` before the relevant phase completes, and record a summary of the success or failure in `agent.summary.txt`.
- The display name of `openai.yaml` alone must not be treated as satisfying the orchestration contract.
- After evaluating the child `agent`'s returned result, judge `issue_severity` (`minor` / `major` / `critical`), and when re-submission is needed, choose `repair_strategy` (`reuse` / `restart`).
- The `orchestration agent` must not directly perform the repair of a phase artifact itself. When a repair is needed, it must re-delegate to the child `agent` of the target `step` or `substep`.
- `repair_strategy=reuse` is limited to a contract-invariant local fix, and `repair_strategy=restart` must be chosen when contract reinterpretation or wide-scope regeneration is needed.
- The launch request on re-submission must record `issue_severity`, `repair_strategy`, `repair_target_agent_run_id`, and `repair_reason` as required.
- The retry target on a `Validate` failure (`Generate` / `Compile` / `Spec` / `Validate.execute`) is decided deterministically from the combination of `semantic_review.json#findings[*].attribution` and `verdict.json#failure_class` (canonical mapping: the "Decision criteria for retry on failure" section of `docs/workflow/phases/phase_04_validate.md`). On a `Build` failure, the `repair_strategy` of `Generate` is decided deterministically from `binary_meta.json#failure_category` (canonical mapping: the retry-trigger section of `docs/workflow/phases/phase_03_build.md`).
- On re-submission, regardless of `repair_strategy`, issue a new `agent_run_id`, and permit `agent_session_id` reuse only when `repair_strategy=reuse`.
- In the `write-step-result` of a phase that has `substep`, `substep_agent_run_ids` must enumerate, without omission, all `substep` of that `step` on `agent_runs.jsonl`. Even for a `substep` whose terminal `status` is other than `pass`, the `agent_run_id` must not be omitted. For supplements, refer to Operations Rule 20 of `docs/ORCHESTRATION.md` and `docs/RUNBOOK.md` 1-3.
- In a `step_result.json` that includes `retry_decisions`, the `effective pass substep` set must be uniquely restorable from the substitution relationship `repair_target_agent_run_id -> new_agent_run_id`. The old failed run remains in `substep_agent_run_ids` for history retention and is excluded from the `status=pass` judgment target. In a `status=pass` `step_result.json`, each `new_agent_run_id` is limited to the finally-adopted `pass` run remaining in the `effective pass substep` set, and the intermediate run of a chained retry re-substituted in a subsequent retry must not be retained.
- `orchestration_checkpoint.json` is auto-updated by `tools/orchestration_runtime.py` after `write-step-result` completes with `status=pass`. `orchestration_checkpoint.json` must not be manually edited.
- In an orchestration set to `resume_enabled=true`, use only the result of `check-step-completed` as the canonical source for the skip judgment. A skip must not be decided by a direct reference to `step_result.json`.
- A `step` detected as `stale` by `verify-checkpoint-integrity` must not be skipped.

## Operations Rules

**tmp-area usage contract (premise)**: the `allowed_tmp_root` of the orchestration agent / child `agent` is fixed at `workspace/tmp/<agent_run_id>/` (recorded in `output_manifests/<agent_run_id>.json` at `record-launch` time). For a temporary-file write, directly specify under that literal path. `output_manifest_write_guard` only judges whether the write-target path is under `allowed_tmp_root`, and does not reference the `$TMPDIR` env. Bootstrap Bash such as `export TMPDIR=...`, `jq -er ...`, `printenv`, or `bash -c` must not be called (it is a root cause of the workflow stopping on a Claude Code session-sandbox approval request). The env variables (`METDSL_ORCHESTRATION_ID` / `ORCHESTRATION_AGENT_RUN_ID` / `TMPDIR`) are inherited into the subprocess by `tools/run_workflow.py`, so a confirmation Bash is also unnecessary. Embed `allowed_tmp_root` literally into the child `agent`'s launch prompt too (see `references/launch_prompts.md`).

1. Run `python3 tools/run_workflow.py <spec_ref> <until_phase> --llm <backend>` to perform the initialization of `workspace/orchestrations/<orchestration_id>/`, the generation of `preflight.json`, and the generation of the launch prompt all at once.
2. The workflow must not be started by a path other than `tools/run_workflow.py`. When running `tools/orchestration_runtime.py` directly for exceptional operation, record the reason and the diff, and return to the normal operation.
3. The `probed_at` field of `preflight.json` is used for the TTL-cache decision at `record-launch` time, and is auto-updated by `tools/orchestration_runtime.py` on a successful live probe after TTL expiry. Because this auto-update does not change the judgment-result fields such as `status` / `can_launch_*`, it is outside the application of the manual-editing-prohibition rule.
4. When `can_launch_step_agents=true` and `can_launch_substep_agents=true` of `preflight.json` are not satisfied, do not start the workflow.
5. The `orchestration agent` reads `references/startup_contract.md` to fix the launch conditions, and in the first `commentary` declares the execution of the target phase, the `SKILL` to use, the kind of `agent` to launch, and the place that uses `MCP`.
6. After the execution declaration, re-confirm the phase type by a fixed table, and select a `substep agent` for `Compile` / `Generate` / `Validate` and a `step agent` for `Build`.
7. Assemble the launch-request body and `skill_must_read_refs` by the canonical generation rules equivalent to `prepare_launch_request_payload` and `render_launch_prompt_text` of `tools/orchestration_runtime.py`. A field omission by manual concatenation, a missing verify-required ref, and a value mismatch between the prompt and the request are forbidden.
8. Before running `record-launch`, check that no placeholder remains in `ir_ref`, `pipeline_ref`, and `dependency_ref`; that the `dependency_ref` of the `step=compile` substep launch requests (`generate` / `verify`) is a string starting with `spec/` and ending with `/deps.yaml` (confirm non-empty / non-placeholder); that the `skill_must_read_refs` of the `verify` launch request covers the required IR sections; and that the launch-request body includes the prohibition of non-canonical `tools/` / validator references.
9. Before launching the child `agent` of `Compile`, reconcile the `ir_ref` and `ir_meta.json.verification_status` per immediate dependency `node` of the target `node`, and when `direct dependency compile readiness` does not hold, do not launch the child `agent` and record `blocked` or `fail`.
10. Before launching the child `agent` of `Generate` onward, reconcile the `ir_ref`, `pipeline_ref`, and `aggregate_verdict` per immediate dependency `node` of the target `node`, and when `direct dependency execution readiness` does not hold, do not launch the child `agent` and record `blocked` or `fail`.
11. Before proceeding to the phase body, confirm the 3 conditions of preflight-done, launch-prompt-prepared, and child-`agent`-launched. When unmet, editing, `MCP` execution, and phase-artifact generation must not start.
12. Launch the child `agent` with the generated launch-request body. Save only the actual `spawn_agent` response immediately after a successful launch with `record-launch`, and record `launch_prompt_ref` and `launch_reply_ref` at the same time. A provisional record before the actual launch, a completion record after a launch failure, and the post-injection of an arbitrary `response_payload` are forbidden.
13. After the child `agent` completes, run the recording processing in the following order. The order must not be reversed. The `record-child-return` â†’ `deactivate-child` handshake is required for the Adv-14/16/20 guards, and a missing one stops the runtime with a `ValueError`.
    1. Run `python3 tools/orchestration_runtime.py record-child-return --repo-root <repo_root> --orchestration-id <orchestration_id> --agent-run-id <agent_run_id> --return-token "$(cat workspace/orchestrations/<orchestration_id>/launches/<agent_run_id>.parent_return_token)"`, and record the evidence `child_returns/<agent_run_id>.txt` that the Agent tool return was observed (Adv-30: a parent-bound token that prevents forgery by an arbitrary caller; auto-generated by `record-launch`. Use the command substitution `$(cat ...)` inline, and do not use the 2-step form `VAR=$(cat ...)` â€” the leading `VAR=` breaks the `Bash(python3 ...)` allowlist match and requires session approval).
    2. Run `deactivate-child` to switch the active context back to the orchestration agent (rejected on ack absence or token mismatch).
    3. Overwrite `launches/<agent_run_id>.reply.txt` with the final response text of the Agent tool using `record-reply`.
    4. Run `python3 tools/orchestration_runtime.py record-agent-run --repo-root <repo_root> --orchestration-id <orchestration_id> --agent-run-json '<json>'` to append 1 line to `agent_runs.jsonl`.
    `record-agent-run` must also save `agent.result.json` and `agent.summary.txt` at the same time. Because `record-agent-run`, in addition to the declared `output_refs`, the `apply_patch_writes` internal gate record, and the output manifest record, inspects the actually-changed paths by the diff against the baseline, when an `unauthorized write` not included in the capability token's `write_root`, a gate-permitted path, or a manifest-permitted path exists, it rejects that `agent_run`. After a reject occurs, the `orchestration agent` must record `orchestration_meta.status=fail_closed` and stop.
14. For a phase that has `substep`, evaluate the returned result and decide `issue_severity` and `repair_strategy`. When re-submission is needed, attach `repair_target_agent_run_id` and `repair_reason` to the launch request, relaunch, and add `record-launch`.
15. In a `repair_strategy=reuse` re-submission, request only the diff fix without changing the target `substep`'s contract. In a `repair_strategy=restart` re-submission, regenerate from the target `substep`'s contract input.
16. When tempted to take a contract-violating shortcut, make explicit in `commentary` that a child-`agent` launch is required, and return to the launch procedure. A local implementation must not be continued.
17. For a phase with no standard `substep`, after the `step agent` completes; for a phase that has `substep`, after the `orchestration agent` aggregation completes; run `python3 tools/orchestration_runtime.py write-step-result --repo-root <repo_root> --orchestration-id <orchestration_id> --node-key <node_key> --step <step> --agent-run-id <agent_run_id> --result-json '<json>'`. When re-submission was performed, include `retry_decisions` in `step_result.json`. Include in `substep_agent_run_ids` the `agent_run_id` of **all** `substep` of that `step` recorded in `agent_runs.jsonl` without omission, and do not omit an ID that terminated with `fail` / `cancel` etc. In a `status=pass` `step_result`, each `new_agent_run_id` recorded in `retry_decisions` is limited to the finally-adopted `pass` run remaining in the `effective pass substep` set, and an intermediate run re-substituted in a subsequent retry must not be included. It is required that each run of the `effective pass substep` set is `pass` and that `required_outputs` is covered by only the `output_refs` of that set. It must not depend on the `output_refs` of a pre-retry failed run or a superseded run.
18. At workflow end, run `python3 tools/orchestration_runtime.py set-status --repo-root <repo_root> --orchestration-id <orchestration_id> --status <status>` to update `orchestration_meta.json` to a terminal status. The narrative fields (`reason_code` / `reason_detail` / `blocking_policy_scope`) are fixed on the first call. A re-call with the same terminal status is accepted as **idempotent** (a cleanup retry if the cleanup_committed marker is not written, a no-op replay if the marker is written), and the narrative is not overwritten. Add/fix the narrative by editing `failure_analysis.json`. The only permitted terminal-to-terminal transition is `fail` â†’ `fail_closed`; others are rejected by the runtime guard.
19. In an orchestration where `resume_enabled=true` is set in `orchestration_meta.json`, the following confirmation may be done before launching each `step` / `node`. Run `python3 tools/orchestration_runtime.py check-step-completed --repo-root <repo_root> --orchestration-id <orchestration_id> --node-key <node_key> --step <step>`. When `completed=true` and `integrity=ok` are returned, skip the launch of that `step` and pass the returned `ir_ref` / `pipeline_ref` / `output_refs` to the subsequent `step`. When `completed=false` or the integrity has failed, run that `step` as usual. For a skipped `step`, issue a new `agent_run_id` with `python3 tools/orchestration_runtime.py record-agent-run` as `agent_role=skipped_by_checkpoint`, and append a minimal entry including `skipped_step` and `reason=checkpoint_integrity_ok` to `agent_runs.jsonl`. In an orchestration with `resume_enabled=false` (default), do not apply this rule, and run all `step` newly even if a checkpoint exists.
20. `status` and `can_launch_*` must not be changed by manually editing or post-editing `preflight.json`. A change in the check conditions is reflected only by a `preflight` re-run.
21. When the live preflight gate is `fail` at `record-launch` time, stop that launch and permit only `set-status --status fail`.
22. In the completion record of each child `agent` of `Compile` / `Generate` / `Build` / `Validate`, the `--stage` value run with `validate_pipeline_semantics.py` must be recorded in the `validation_stage` field of `step_result.json`. In a `status=pass` `step_result`, when the allowed `validation_stage` value corresponding to that `step` (`compile`: `compile`/`full`, `generate`: `post_generate`/`post_build`/`full`, `build`: `post_build`/`full`, `validate`: `post_execute`/`pre_judge`/`full`) is not recorded, `write-step-result` fails. The `step_result` of the optional flows `Tune` / `Promote` does not require `validation_stage`.
23. When a child `Agent` tool is cut off midway by an API stream idle timeout etc., do not hand-write `record-agent-run` with `status=timeout`, but finalize in the following order. A missing step stops with a `ValueError` by the Adv-14/16/20 guards.
    1. `record-child-return --agent-run-id <arid>`: record the evidence that the Agent tool return (including the timeout-termination response) was observed.
    2. `deactivate-child --child-run-id <arid>`: release the active marker (consume the ack).
    3. `record-timeout --agent-run-id <arid> --reason "<text>"`: record the terminal entry, run the partial-write consistency check and the `workspace/tmp/<arid>/` deletion.

    Only in the exceptional case where the Agent tool wedges and no return can be observed at all (process kill etc.), the marker check can be bypassed with `record-timeout --force-reason "<operator override content>"`. `forced=True` and `forced_reason` remain in the audit record. Prioritize the normal flow, and use `--force-reason` as a last-resort escape hatch. For details, refer to `docs/RUNBOOK.md#substep-timeout-recovery`.

## Decision Criteria
- The `orchestration agent` does not directly generate phase artifacts.
- `workspace/orchestrations/<orchestration_id>/preflight.json` exists and satisfies the `pass` conditions.
- The execution declaration of the target phase, the `SKILL` used, the kind of `agent` launched, and the place that uses `MCP` exists in the first `commentary`.
- `orchestration` and, as needed, each role of `step` / `substep` are recorded in `agent_runs.jsonl`.
- `Compile` / `Generate` / `Validate` are launched with a `substep agent`, and `Build` with a `step agent`.
- The `agent.result.json` and `agent.summary.txt` corresponding to each `agent_run` of `step` / `substep` exist and match the reference values of `agent_runs.jsonl`.
- The request and response of `launches/` match the `launch_request_ref` / `launch_response_ref` of `agent_runs.jsonl`.
- The response of `launches/` and `agents/<agent_run_id>/dialogs/child.response.json` holds the same actual `spawn_agent` response.
- `agent_runs.jsonl.agent_session_id` matches the child-`agent` identifier of the corresponding `launch response`.
- The prompt and reply of `launches/` match the `launch_prompt_ref` / `launch_reply_ref` of `agent_runs.jsonl`.
- The prompt of `launches/` is based on the corresponding template of `references/launch_prompts.md`, with no omission or semantic change of the template's required items.
- No placeholder ref exists in the request of `launches/`.
- The `launches/` request of `verify` records the required IR sections in `skill_must_read_refs`.
- The child `agent`'s `launches/` prompt forbids reading the implementation under `tools/`, verification `script`, test code, and validator code as a rule source.
- `step_result.json` holds `executor_agent_run_id` and `substep_agent_run_ids`. For a `step` that has `substep`, `substep_agent_run_ids` covers the `agent_run_id` of all `substep` of that `step`.
- When `step_result.json` holds `retry_decisions`, the `effective pass substep` set can be uniquely restored, and the `status=pass` judgment and the `required_outputs` coverage judgment are based on only that set. When `status=pass`, each `new_agent_run_id` is the finally-adopted `pass` run and does not include an intermediate retry run.
- When re-submission was performed, the relevant `launch` request includes `issue_severity`, `repair_strategy`, `repair_target_agent_run_id`, and `repair_reason`.
- All `launch` requests of the child `agent` include `skill_name`, `skill_ref`, and `skill_must_read_refs`.
- The choice of `repair_strategy=reuse` and `repair_strategy=restart` matches the decision conditions of `ORCHESTRATION.md`.
- When `check-step-completed` was used in an orchestration with `resume_enabled=true`, the skipped `step` is recorded in `agent_runs.jsonl` with `agent_role=skipped_by_checkpoint`.
- The `output_refs` of the skipped `step` are consistent with the `ir_ref` / `pipeline_ref` of the subsequent `step`.
- When `valid=false` is returned by `verify-checkpoint-integrity`, the relevant `step` is re-run.
- In a child-`agent`-required phase, there is no direct editing of a phase artifact, `MCP` execution, or verification-purpose provisional implementation before launching the child `agent`.
- The child `agent`'s phase-artifact change uses only `guarded-apply-patch` for `.json` / `.txt`, and for other extensions a direct `Edit` / `Write` to a path within `output_manifests/<agent_run_id>.json.allowed_file_tool_paths`, and an `unauthorized write` including a shell file write or a write to a path outside the manifest is rejected by `record-agent-run` or a pre-gate.
- `agent.summary.txt` includes the final state and the main `output_refs` or the failure cause, not only a single-line boilerplate `pass` / `fail`.
